Arrays

Overview

An array stores a sequence of elements in consecutive memory locations. The position of an element is called its index, and indexing gives arrays their most important feature: direct access to any element in constant time.

Arrays are often the simplest data structure in a course, but they are also the foundation for many later structures. Stacks, queues, heaps, hash tables, adjacency matrices, and dynamic-programming tables all commonly use arrays internally.

The word array is used in two related ways. Sometimes it means a fixed-size block of memory, such as a Java primitive array or a C++ built-in array. Other times it means an array-like sequence that can grow, such as a Java ArrayList, C++ vector, or Python list. The fixed-size version is the lower-level storage idea; the resizable version is usually built on top of it.

Abstract Data Type and Operations

As an abstract sequence, an array-like structure stores values in positions from 0 to n - 1. The user mainly thinks in terms of positions, not memory addresses.

  • get(i): Return the value at index i.
  • set(i, value): Replace the value at index i.
  • search(value): Find whether a value occurs, and possibly where.
  • append(value): Add a value after the current last element.
  • insert(i, value): Add a value at index i, shifting later values right.
  • delete(i): Remove the value at index i, shifting later values left.

Conceptual Model

The key idea is that the address of an element can be computed from its index. If each element takes the same amount of space, then the element at index i is located a fixed distance from the beginning of the array. This is why A[i] does not require scanning through earlier elements.

More formally, if the array begins at memory address base and each element uses w bytes, then index i is found at base + i * w. This formula is the reason arrays have constant-time random access. It is also why ordinary arrays require all elements to have the same size.

Index 0 1 2 3 4
Value 14 8 31 5 20

Inserting at index 2 does not simply place a value into an empty gap. The values currently at indices 2, 3, and 4 must shift one position to the right first.

Worked Example

Suppose a dynamic array has logical values [14, 8, 31, 5, 20], size 5, and capacity 8. The backing array has room for three more values, but those extra slots are not part of the logical sequence yet.

To perform insert(2, 99), the implementation must make a hole at index 2.

  1. Move 20 from index 4 to index 5.
  2. Move 5 from index 3 to index 4.
  3. Move 31 from index 2 to index 3.
  4. Store 99 at index 2.
  5. Increase the size from 5 to 6.

The final logical array is [14, 8, 99, 31, 5, 20]. Three values moved, so this particular insertion did work proportional to the number of elements after the insertion point. If the insertion had been at the end, no shifting would have been needed.

Core Operations

Access and Update

Access and update are constant-time operations because the index maps directly to a memory position. The usual corner case is an invalid index: values less than 0 or at least n are outside the logical array.

The value stored at an index can be read or overwritten without affecting the positions of the other elements. This makes arrays a good choice when an algorithm repeatedly revisits known positions, such as a dynamic programming table where dp[i] has a specific meaning.

Sequential Search

Without extra information, searching an array for a value requires checking elements one at a time. This is the same idea as Sequential Search.

If the array is sorted, searching can be improved with binary search, but that changes the tradeoff. Keeping an array sorted makes searches faster, but insertions may become more expensive because the new value must be placed in its sorted position instead of simply appended.

Insertion and Deletion

Insertion and deletion are cheap at the end, but expensive near the front. The cost comes from shifting elements to preserve the contiguous order.

  1. To insert at index i, shift elements i through n - 1 one step right.
  2. Store the new value at index i.
  3. Increase the logical size.

Deletion works in the opposite direction. To delete index i, shift elements i + 1 through n - 1 one step left and then decrease the size. The old final slot may still contain a leftover value in memory, but it is outside the logical array.

Resizing

A fixed array has a fixed capacity. A resizable array, often called a dynamic array, allocates a larger backing array when it runs out of room and copies the old values over. If capacity is usually doubled, appending has amortized O(1) time even though an individual resize costs O(n).

Doubling is important. If the array grew by only one slot each time, then appending n items could require copying about 1 + 2 + 3 + ... + n old values, which is \(O(n^2)\) total work. Doubling spreads the copying cost across many cheap appends.

Implementation Options

Implementation 1: Fixed-Size Array

A fixed-size array stores exactly its allocated capacity. The main invariant is that every valid index maps to one stored element, and the capacity does not change.

Fixed arrays are common when the number of elements is part of the problem statement. For example, if an algorithm stores one distance value for each vertex in a graph with V vertices, an array of length V is natural. There is no need to pay for resizing logic when the size is already known.

Notes

  • Good when the final size is known in advance.
  • Cannot append past capacity without allocating a different array.
  • Provides excellent cache locality because elements are adjacent in memory.

Implementation 2: Resizable Array

A resizable array stores a logical size and a physical capacity. The backing array may have unused slots after the last logical element.

The main invariant is that the first size positions hold the logical sequence in order, and positions from size through capacity - 1 are available storage. Client code should never treat those unused positions as meaningful elements.

Operation Sketch

append(value):
    if size == capacity:
        allocate a new array with about twice the capacity
        copy the old elements into the new array
        replace the old backing array
    A[size] = value
    size = size + 1

insert(i, value):
    check that 0 <= i <= size
    resize if size == capacity
    for j from size down to i + 1:
        A[j] = A[j - 1]
    A[i] = value
    size = size + 1

delete(i):
    check that 0 <= i < size
    removed = A[i]
    for j from i to size - 2:
        A[j] = A[j + 1]
    size = size - 1
    return removed

Library Implementations

  • Java: primitive arrays such as int[]; ArrayList<E> for a resizable array.
  • C++: std::array for fixed size; std::vector for dynamic arrays.
  • Python: list is a dynamic array-like sequence.

These library structures usually handle resizing, bounds checks, and memory management for you. In normal software development, use the library version unless the point of the assignment is to understand how it works.

Operation Costs

Operation Fixed Array Resizable Array Notes
Access or update by index \(O(1)\) \(O(1)\) Index arithmetic gives direct access.
Search by value \(O(n)\) \(O(n)\) Without ordering or hashing, scan the elements.
Append at end \(O(1)\) if space exists \(O(1)\) amortized A resize occasionally costs \(O(n)\).
Insert/delete near front \(O(n)\) \(O(n)\) Elements must shift to keep order contiguous.
Extra space \(O(n)\) \(O(n)\), possibly with unused capacity A dynamic array intentionally keeps extra room to make future appends cheap.

The table gives asymptotic costs, but constants matter here. Arrays often outperform pointer-based structures even when both have the same Big-O cost because nearby array elements tend to be loaded into cache together.

Applications

  • Dynamic programming tables: Many dynamic programming algorithms define subproblems by one or more integer indices. Arrays are a natural fit because a state such as dp[i] or dp[i][j] can be accessed directly when the recurrence needs it.
  • Backing storage for other structures: Stacks, circular queues, binary heaps, hash tables, and disjoint-set forests are commonly stored in arrays. The higher-level structure adds rules about how the array positions are interpreted.
  • Graph representations: An adjacency matrix is a two-dimensional array where entry A[u][v] records whether an edge exists. Adjacency-list representations also often use an array indexed by vertex, where each entry stores that vertex's neighbor list.
  • Numerical and systems code: Arrays are preferred when code repeatedly scans or indexes large blocks of data. Their contiguous layout helps hardware caches and makes them a good match for numerical kernels, image buffers, and low-level memory operations.
  • Counting and frequency tables: If the keys are small integers or characters from a limited alphabet, an array can store counts directly. Counting sort, radix sort subroutines, histograms, and simple frequency analyses all use this idea.
  • Precomputed lookup tables: Algorithms sometimes compute useful values once and store them by index for fast later access. Examples include parent pointers in graph search, powers in modular arithmetic, and cost tables in optimization problems.
  • Sliding-window storage: A fixed array or circular buffer can store the most recent \(k\) observations in a stream. This is useful for moving averages, bounded logs, and producer-consumer buffers where old values are discarded as new ones arrive.

Comparisons and Tradeoffs

  • Arrays vs. linked lists: Arrays give constant-time indexed access and usually better cache performance. Linked lists avoid shifting for local insertions and deletions, but they must follow links to find positions.
  • End updates vs. middle updates: Appending to a dynamic array is usually cheap, especially when extra capacity is available. Inserting near the front or middle may move many elements, so the cost depends on the insertion position.
  • Dynamic arrays vs. fixed arrays: A fixed array is simple and predictable when the final size is known. A dynamic array is more flexible, but its implementation must manage capacity, resizing, and occasionally copying all elements.
  • Arrays vs. hash tables: Arrays require integer-like indices or a way to map values to dense positions. Hash tables support more general keys, but arrays preserve a simple positional order and have less structural overhead.
  • Arrays vs. trees: Trees maintain order or hierarchy explicitly and can support efficient ordered operations. Arrays by themselves do not maintain search order, but they are compact and support fast position-based access.
  • Dense vs. sparse data: Arrays work well when most positions are meaningful. If almost all entries are empty or default values, a sparse representation may use far less memory.

Variations and Extensions

  • Multidimensional arrays: These arrays are indexed by two or more coordinates, such as A[row][col]. They are often stored internally in one contiguous block using row-major or column-major order.
  • Circular buffers: A circular buffer treats the physical end of the backing array as adjacent to the beginning. This is the usual array-based implementation idea for queues and fixed-size rolling windows.
  • Sparse arrays: A sparse array stores only the positions whose values differ from a default. This saves memory when the logical index range is large but most entries are unused.
  • Sorted arrays: Keeping an array sorted allows binary search in \(O(\log n)\) time. The tradeoff is that insertion often requires shifting elements to preserve sorted order.
  • Bit arrays: A bit array packs Boolean values into individual bits instead of full bytes or words. This can greatly reduce memory use for visited sets, masks, and simple flags.
  • Array views and slices: Some languages provide lightweight views into part of an array. These can avoid copying, but they also require care because changes through the view may affect the original data.

Common Pitfalls

  • Confusing size and capacity: The size is the number of meaningful elements; the capacity is the number of allocated slots. Treating unused capacity as real data leads to incorrect searches, prints, and updates.
  • Shifting in the wrong direction: Insertion should shift from right to left so values are not overwritten before they are copied. Deletion should shift from left to right to fill the removed position.
  • Ignoring bounds checks: Accessing an index less than 0 or at least size is outside the logical array. Depending on the language, this may raise an error, return garbage, or corrupt memory.
  • Misusing amortized analysis: Dynamic-array append is amortized \(O(1)\), not worst-case \(O(1)\). A single append can still be expensive if it triggers a resize.
  • Forgetting resize side effects: In some languages, resizing a dynamic array can invalidate references, pointers, or iterators into the old backing storage. Code that keeps such references must understand the library's guarantees.
  • Leaving stale values after deletion: Decreasing the logical size does not necessarily erase the old final slot. If the stored values are references, some implementations deliberately clear that slot to allow garbage collection.
  • Choosing arrays for sparse data: An array indexed over a huge range can waste memory if only a few positions are used. A map, dictionary, or sparse representation may be more appropriate.

Summary

Arrays store elements contiguously and support constant-time indexed access. Their main weakness is that ordered insertions and deletions require shifting elements. Dynamic arrays reduce the pain of fixed capacity by resizing as needed, giving amortized constant-time append.
