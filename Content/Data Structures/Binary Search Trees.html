Binary Search Trees

Overview

A binary search tree, or BST, is a binary tree with an ordering invariant: every value in a node's left subtree is smaller than the node's value, and every value in the right subtree is larger. This invariant lets search make one comparison per level.

The BST idea is independent of the exact implementation. The tree might be represented by linked nodes, objects, or records in memory. What matters conceptually is the ordering rule and the path it creates for search, insert, and delete.

BST Invariant

For each node \(x\):

  • all values in \(x\)'s left subtree are less than \(x.value\), and
  • all values in \(x\)'s right subtree are greater than \(x.value\).

This rule applies recursively to every subtree. It is not enough for a node's immediate children to be on the correct side; every descendant must also respect the range implied by its ancestors.

Different implementations make different choices about duplicate keys. Some disallow duplicates, some store a count in each node, and some consistently put equal keys on one side. The demos on this page disallow duplicate values so the ordering rule stays uncluttered.

Search and Insert

Searching starts at the root. If the target equals the current value, the search succeeds. If the target is smaller, move left. If it is larger, move right. The search fails only when the next subtree is empty.

Insertion follows the same comparison path. When the search reaches an empty child position, the new value is placed there. This preserves the BST invariant because every comparison on the path has narrowed the valid range for the new value.

Traversal

In-order traversal of a BST visits the values in sorted order. This follows directly from the invariant: the left subtree contains smaller values, the node itself comes next, and the right subtree contains larger values. 

inorder(node):
    if node is empty:
        return
    inorder(node.left)
    visit(node.value)
    inorder(node.right)

The traversal still takes \(O(n)\) time because every node is visited once.

Delete

Deletion has three conceptual cases.

  • Leaf: remove the node.
  • One child: replace the node by its child.
  • Two children: replace the value by its in-order predecessor or successor, then delete that replacement value from its old location.

The two-child case is the only one that feels different. The in-order successor is the smallest value in the node's right subtree; the in-order predecessor is the largest value in the left subtree. Either replacement keeps the values in sorted order.

Rotations

A rotation is a local restructuring operation. It changes which node is the parent inside a small subtree while preserving the BST in-order sequence. Rotations are the basic tool used by AVL trees and red-black trees to restore balance after updates.

The simple rotations are left rotation and right rotation. The double rotations are built from those: a left-right rotation is a left rotation followed by a right rotation, and a right-left rotation is a right rotation followed by a left rotation.

Complexity

Search, insert, and delete follow a single root-to-leaf path, so their running time is \(O(h)\), where \(h\) is the height of the tree.

Tree Shape Height Search/Insert/Delete
Balanced \(\Theta(\log n)\) \(\Theta(\log n)\)
Degenerate \(\Theta(n)\) \(\Theta(n)\)

A plain BST does not automatically stay balanced. If values are inserted in sorted order, the tree can become a chain. Balanced BST variants add rules and rotations to keep \(h = O(\log n)\).

Implementation Sketch

search(root, target):
    current = root
    while current is not empty:
        if target == current.value:
            return current
        if target < current.value:
            current = current.left
        else:
            current = current.right
    return not found

insert(root, value):
    if root is empty:
        return new node containing value
    if value < root.value:
        root.left = insert(root.left, value)
    else if value > root.value:
        root.right = insert(root.right, value)
    return root

This sketch omits duplicate handling and parent references. Production implementations also need careful update logic for deletion and, in balanced trees, rebalancing metadata such as heights or colors.
