Linked Lists

Overview

A linked list stores a sequence of values in separate node objects. Each node stores a value and one or more links to other nodes. Instead of using an index to jump directly to a position, a linked list is usually traversed by following links from the head.

Linked lists are useful when you often insert or remove nodes near known positions and do not need constant-time random access. They also provide a clean way to teach pointer/reference manipulation.

The phrase near known positions is important. A linked list can insert after a node in constant time once that node is already known, but finding the node may still require a traversal. Many incorrect analyses accidentally count only the pointer update and ignore the search needed to reach the update location.

Abstract Data Type and Operations

Abstractly, a linked list represents an ordered sequence. The implementation decides how nodes are linked.

  • front: Return the first value.
  • pushFront(value): Insert a value at the beginning.
  • pushBack(value): Insert a value at the end.
  • insertAfter(node, value): Insert a value after a known node.
  • deleteAfter(node): Remove the node after a known node.
  • search(value): Traverse until a matching value is found or the list ends.

Conceptual Model

In a singly linked list, each node has a next reference. The list object usually stores a reference to the head node, and sometimes also to the tail node. 

head
 |
 v
[14 | next] -> [8 | next] -> [31 | next] -> null

In a doubly linked list, each node also stores a prev reference. That extra link makes it easier to move backward and to delete a known node, but it uses more memory and requires more pointer updates.

Unlike an array, the nodes do not need to be adjacent in memory. The logical order comes entirely from the links. This flexibility avoids shifting elements, but it also means there is no formula like base + i * w for jumping directly to index i.

Worked Example

Suppose a singly linked list stores 14 -> 8 -> 31 -> 5, and we want to insert 99 after the node containing 8. The node containing 8 is already known.

  1. Create a new node x containing 99.
  2. Set x.next to the old successor of the 8 node, which is the 31 node.
  3. Set the 8 node's next reference to x.

The order becomes 14 -> 8 -> 99 -> 31 -> 5. The sequence of assignments matters: if we overwrite the 8 node's next link before saving the old successor, the rest of the list can become unreachable.

Deleting after the 8 node is the reverse idea. If x is the node after 8, then setting 8.next = x.next bypasses x. In languages with garbage collection, the removed node is eventually reclaimed if nothing else references it. In languages with manual memory management, the removed node must also be freed appropriately.

Core Operations

Traversal and Search

Traversal starts at head and repeatedly follows next. Search is traversal with a comparison at each node. If the desired value is near the end or absent, every node is inspected.

Traversal code usually uses a reference such as cur. At each step, cur points to the current node, and cur = cur.next advances one node. The traversal stops when cur is null. Forgetting that final null check is one of the most common sources of list bugs.

Insert at Head

  1. Create a new node whose next points to the old head.
  2. Update head to point to the new node.
  3. If the list was empty, also update tail.

Inserting at the head is the simplest constant-time list update. It does not require traversal because the list object already stores the head reference.

Insert After a Known Node

  1. Create the new node.
  2. Set its next reference to the known node's old next.
  3. Set the known node's next reference to the new node.

If the known node was the tail, the new node becomes the tail. Implementations that store a tail reference must remember to update it. Without a tail reference, appending at the end requires walking from the head to the last node.

Delete After a Known Node

Deleting after a known node bypasses the removed node. The known node's next becomes the removed node's next. The removed node is then unreachable from the list.

Deleting the head is a separate case because there is no predecessor node. The list's head reference itself must change. If the list becomes empty, both head and tail should represent an empty list consistently.

Implementation Options

Implementation 1: Singly Linked List

Each node stores a value and a next reference. The invariant is that following next from head visits every node exactly once and eventually reaches null.

Operation Sketch

pushFront(value):
    newNode = Node(value)
    newNode.next = head
    head = newNode
    if tail == null:
        tail = newNode

pushBack(value):
    newNode = Node(value)
    if head == null:
        head = newNode
        tail = newNode
    else:
        tail.next = newNode
        tail = newNode

deleteFirst(value):
    if head == null:
        return false
    if head.value == value:
        head = head.next
        if head == null:
            tail = null
        return true
    prev = head
    cur = head.next
    while cur != null:
        if cur.value == value:
            prev.next = cur.next
            if cur == tail:
                tail = prev
            return true
        prev = cur
        cur = cur.next
    return false

Implementation 2: Doubly Linked List

A doubly linked list stores both next and prev references. The added invariant is that if x.next is y, then y.prev is x. This symmetry makes deletion of a known node constant time.

Doubly linked lists are especially useful when an iterator points directly at a node and the program wants to remove that node without searching for its predecessor. This is one reason doubly linked lists appear in some library list implementations.

Notes

  • More convenient for deques and bidirectional iteration.
  • Requires more memory per node.
  • Every insertion or deletion must update both directions correctly.

Library Implementations

  • Java: LinkedList<E> implements list and deque operations.
  • C++: std::forward_list is singly linked; std::list is doubly linked.
  • Python: no built-in linked list; collections.deque supports efficient end operations.

In practice, linked lists are less common than students sometimes expect. Dynamic arrays are often faster for many workloads because of cache locality and lower memory overhead. Linked lists are still conceptually important and useful when stable node references or frequent local splicing are central to the problem.

The most important distinction is between finding a position and updating a known position. Linked lists are good at the second task. They are not good at the first task unless the program already has a reference to the relevant node.

Applications

  • Stacks and queues: A singly linked list can implement a stack by using the head as the top. A queue can also be implemented efficiently if the structure stores both head and tail references.
  • Adjacency lists: Graphs are often represented by an array of neighbor lists. A linked list is one possible representation for each neighbor list, especially when edges are inserted and removed dynamically.
  • Frequent local splicing: If a program already has references to the relevant nodes, linked lists can insert or remove nodes without shifting a large block of elements. This makes them useful for workloads centered on local structural changes.
  • Pointer and reference education: Linked lists are one of the clearest ways to practice reference updates, aliasing, null checks, and object identity. Even when arrays are faster in practice, linked lists remain valuable for learning how dynamic structures work.
  • Sparse symbolic data: Polynomials, sparse vectors, and similar structures can be represented as linked nodes storing only nonzero or meaningful terms. This avoids allocating a slot for every possible exponent or index.
  • Hash-table chaining: One collision-resolution strategy stores all keys that hash to the same bucket in a linked chain. Modern implementations may use other bucket structures too, but chaining is still an important baseline idea.
  • Low-level memory management: Free lists use links to connect available blocks of memory or reusable objects. This is one reason stacks and linked lists appear in systems-level discussions of allocation.

Comparisons and Tradeoffs

  • Linked lists vs. arrays: Arrays provide \(O(1)\) indexed access, while linked lists must follow links from a known starting point. Linked lists avoid shifting for local updates, but finding the update position may still cost \(O(n)\).
  • Known node vs. searched node: Inserting after a known node is constant time. Inserting after the first node containing a value includes the search, so the total operation is usually linear time.
  • Singly vs. doubly linked: Singly linked lists use less memory per node and are simpler to maintain. Doubly linked lists support reverse traversal and easier deletion of a known node, but every update must preserve both directions.
  • Sentinel vs. no sentinel: Sentinel nodes can eliminate many boundary cases by ensuring there is always a node before the first real element. The cost is extra conceptual overhead and dummy nodes that must not be confused with real data.
  • Memory overhead vs. flexibility: Each linked-list node stores at least one reference in addition to the data. This overhead is worthwhile only when the update pattern benefits from node-level flexibility.
  • Practical performance: Even when Big-O notation looks favorable, linked lists often lose to dynamic arrays because nodes are scattered in memory. Pointer chasing can be slower than scanning adjacent array cells.

Variations and Extensions

  • Singly linked list: Each node points only forward. This is enough for stack-like behavior, simple queues with a tail reference, and one-way traversal.
  • Doubly linked list: Each node points both forward and backward. This supports bidirectional traversal and constant-time deletion of a known node without separately tracking its predecessor.
  • Circular linked list: The final node links back to the first node instead of ending at null. Circular lists are useful for round-robin scheduling and repeated cyclic scans.
  • Sentinel-node list: A dummy head or tail node can simplify insertions and deletions at the boundaries. The algorithm can often treat the beginning and middle of the list more uniformly.
  • Unrolled linked list: Each node stores a small array of values rather than one value. This reduces per-element pointer overhead and improves locality while still allowing block-level splicing.
  • Skip list: A skip list adds extra forward links at multiple levels. It behaves like a probabilistic alternative to balanced search trees, supporting expected logarithmic search, insert, and delete.
  • Intrusive linked list: In systems code, the link fields may be stored inside the objects being linked rather than in separate wrapper nodes. This can reduce allocation overhead but tightly couples the object type to the list structure.

Common Pitfalls

  • Overwriting links too early: If an insertion changes prev.next before saving the old successor, the rest of the list may become unreachable. The order of reference assignments is part of the algorithm.
  • Forgetting boundary updates: Operations that change the first or last real node may also need to update head or tail. Empty-list and one-node-list cases are especially easy to mishandle.
  • Ignoring the search cost: Deleting a known node can be constant time in some list variants. Deleting by value still requires finding the node first, which is linear in the worst case.
  • Accidental cycles: A wrong link assignment can create a cycle, causing traversal to loop forever. Cycle detection with slow and fast references is a useful diagnostic technique.
  • Inconsistent prev links: In a doubly linked list, updating only next leaves the backward links wrong. Bugs may not appear until a later reverse traversal or deletion.
  • Memory-management mistakes: In languages without automatic garbage collection, removing a node from the list does not automatically free it. In garbage-collected languages, forgotten external references can still keep removed nodes alive.
  • Using a list for random access: Repeatedly asking for the item at index i turns into repeated traversal. If indexed access is central, an array or dynamic array is usually the better representation.

Summary

Linked lists store values in nodes connected by references. They are flexible for local insertion and deletion, but they do not support fast indexed access and they use extra memory for links.

  4. Known Position: What is the difference between inserting after a known node and inserting after the first node containing a given value?
  5. Link Order: Why should a new node's next reference usually be set before changing the previous node's next reference?
  6. Doubly Linked Lists: What extra invariant must be maintained when each node has both next and prev references?
  7. Empty List: When deleting the only node in a linked list with head and tail references, what must happen to both references?
  8. Array Comparison: Name one advantage and one disadvantage of linked lists compared with arrays.
  1. Answer: The list must follow links from the head until it reaches index i.
  2. Answer: Inserting at the tail.
  3. Answer: The predecessor's next reference must be changed to bypass the deleted node.
  4. Answer: Inserting after a known node only changes references; finding the first node with a value may require an \(O(n)\) search first.

In-Class Activities

  1. Pointer Trace: Draw the reference updates for inserting between two existing nodes.
  2. Boundary Cases: List all special cases for deleting the first copy of a value.
  3. Compare Workloads: Decide whether an array or linked list is better for several update patterns.
  4. Bug Hunt: Given an incorrect insertion snippet that overwrites next too early, identify exactly where the rest of the list is lost.
  5. Tail Debate: Decide which list operations justify storing a tail reference and which do not benefit from it.
  6. Doubly Linked Trace: Draw all next and prev changes needed to remove a middle node from a doubly linked list.
  7. Sentinel Design: Redesign a deletion algorithm using a dummy head node and identify which special cases disappear.
  8. Cycle Diagnosis: Draw a list with an accidental cycle and trace slow/fast references until they meet.
