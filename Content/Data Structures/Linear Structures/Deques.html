Deques

Overview

A deque is a double-ended queue: it supports insertion and removal at both the front and the back. The word is usually pronounced like "deck." It is related to, but not the same word as, dequeue, the operation that removes an item from a regular queue.

A deque generalizes both stacks and queues. If a program only uses one end, the deque behaves like a stack. If a program inserts at the back and removes from the front, the deque behaves like a FIFO queue. The full deque ADT allows both ends to be active.

Deques are useful when an algorithm needs to keep a sequence of candidates and discard from either end. They appear in sliding-window algorithms, task scheduling, undo-like histories, palindrome checks, and library implementations that want efficient front and back operations.

Abstract Data Type and Operations

A deque exposes two symmetric ends. Different languages use different method names, but the abstract operations are the same.

  • addFirst(value): Insert value at the front.
  • addLast(value): Insert value at the back.
  • removeFirst(): Remove and return the front value.
  • removeLast(): Remove and return the back value.
  • peekFirst(): Return the front value without removing it.
  • peekLast(): Return the back value without removing it.
  • size() and isEmpty(): Report the number of stored values or whether none are stored.

Some libraries use names such as push_front, push_back, pop_front, and pop_back. A stack-oriented interface may call the same actions push and pop. The important rule is that both ends can be updated efficiently.

Conceptual Model

Think of a deque as a line where people may enter or leave from either doorway: 

front -> [14] [8] [31] [5] <- back

Calling addFirst(20) produces: 

front -> [20] [14] [8] [31] [5] <- back

Calling removeLast() from that state returns 5 and leaves: 

front -> [20] [14] [8] [31] <- back

A deque does not promise fast access to the middle. Its purpose is efficient work at the two ends. If an algorithm repeatedly needs arbitrary index access, a dynamic array list is usually a better fit.

Worked Example

Start with an empty deque and perform: addLast(4), addLast(7), addFirst(2), removeLast(), addFirst(9), removeFirst().

Operation Deque Afterward Return Value
addLast(4) [4] none
addLast(7) [4, 7] none
addFirst(2) [2, 4, 7] none
removeLast() [2, 4] 7
addFirst(9) [9, 2, 4] none
removeFirst() [2, 4] 9

The final deque stores [2, 4]. The example shows that neither pure FIFO nor pure LIFO order fully describes a deque. The caller chooses which end to use for each operation.

Core Operations

Adding at Either End

addFirst creates a new front value; addLast creates a new back value. An efficient implementation must avoid shifting the entire sequence when adding at the front. This is the key difference between a true deque and a naive dynamic-array list used as a deque.

Removing at Either End

removeFirst removes the front value, and removeLast removes the back value. Both operations must define what happens when the deque is empty. Many libraries raise an exception or report an underflow error.

Viewing the Ends

peekFirst and peekLast read values without removing them. These operations are useful when an algorithm needs to compare a new value with a candidate at one end before deciding whether to remove it.

Using a Deque as a Stack or Queue

A deque can simulate a stack by adding and removing at the same end. It can simulate a queue by adding at one end and removing at the other. Even so, using the smaller stack or queue ADT can make code clearer when the extra deque operations are not needed.

Implementation Options

Implementation 1: Circular Array Deque

A circular array deque stores a backing array, a frontIndex, a size, and a capacity. Logical positions wrap around the physical end of the array. If the capacity is m, moving one step forward uses (index + 1) mod m, and moving one step backward uses (index - 1 + m) mod m.

The main invariant is that the logical element at position k is stored at physical index (frontIndex + k) mod capacity for 0 <= k < size.

Operation Sketch

addFirst(value):
    if size == capacity:
        resize
    frontIndex = (frontIndex - 1 + capacity) mod capacity
    A[frontIndex] = value
    size = size + 1

addLast(value):
    if size == capacity:
        resize
    backIndex = (frontIndex + size) mod capacity
    A[backIndex] = value
    size = size + 1

removeFirst():
    if size == 0:
        report underflow
    value = A[frontIndex]
    A[frontIndex] = empty
    frontIndex = (frontIndex + 1) mod capacity
    size = size - 1
    return value

removeLast():
    if size == 0:
        report underflow
    backIndex = (frontIndex + size - 1) mod capacity
    value = A[backIndex]
    A[backIndex] = empty
    size = size - 1
    return value
Implementation Exercise

Implement a circular array deque with the four update operations and two peek operations.

  • Use modulo arithmetic for both forward and backward movement.
  • Keep size so empty and full states are not confused.
  • When resizing, copy logical elements into positions 0 through size - 1 in order.
  • Test wrap-around by forcing both frontIndex and the back position to cross the physical boundary.

Implementation 2: Doubly Linked Deque

A doubly linked deque stores references to the first and last nodes. Each node stores a value, a next reference, and a prev reference. Adding or removing at either end changes only a constant number of references.

Many linked deque implementations use sentinel nodes: dummy front and back nodes that do not store real data. Sentinels reduce boundary cases because there is always a node before the first real item and a node after the last real item.

Operation Sketch with Sentinels

addFirst(value):
    insert a new node between frontSentinel and frontSentinel.next

addLast(value):
    insert a new node between backSentinel.prev and backSentinel

removeFirst():
    if size == 0:
        report underflow
    remove and return frontSentinel.next

removeLast():
    if size == 0:
        report underflow
    remove and return backSentinel.prev

Notes

  • Every insertion must set both next and prev links consistently.
  • Every removal must unlink the node in both directions.
  • There is no resizing cost, but each element requires node and reference overhead.

Library Implementations

  • Java: ArrayDeque<E> is usually the preferred deque implementation; LinkedList<E> also implements Deque<E>.
  • C++: std::deque supports efficient operations at both ends and indexed access, though it is not usually implemented as one simple circular array.
  • Python: collections.deque supports efficient append and pop operations at both ends.

Library deques may use implementation strategies more sophisticated than the teaching versions on this page. The ADT expectation remains the same: updates at both ends should be efficient.

Applications

  • Stacks and queues: A deque can serve as either structure by restricting which operations are used.
  • Sliding-window maximum or minimum: A monotonic deque stores candidate values and removes obsolete or dominated candidates from the ends.
  • Palindrome checking: Compare and remove the front and back characters until the sequence is empty or a mismatch appears.
  • Undo and redo histories: Some history structures keep recent actions near one end and discard old actions from the other when capacity is bounded.
  • Round-based simulations: A deque can move an item from the front to the back, add urgent work to the front, or retire completed work from either end.
  • Work scheduling: Some task schedulers use deque-like structures when local and stolen work should be taken from different ends.

Comparisons and Tradeoffs

  • Deque vs. queue: A queue allows insertion at the back and removal from the front. A deque allows insertion and removal at both ends.
  • Deque vs. stack: A stack exposes one active end. A deque exposes both active ends.
  • Deque vs. dynamic array list: A normal dynamic array list can append and remove at the end efficiently, but front operations require shifting unless it is implemented as a circular buffer.
  • Circular array vs. linked deque: Circular arrays are compact and cache-friendly but must handle capacity and wrap-around. Linked deques avoid resizing but use more memory per item.
  • Expressiveness vs. intent: A deque is more general than a stack or queue. If only FIFO behavior is allowed by the algorithm, a queue interface communicates that rule more clearly.

Common Pitfalls

  • Confusing deque and dequeue: A deque is a data type. Dequeue is a queue removal operation.
  • Shifting on front operations: A deque implemented by shifting a whole array on every addFirst or removeFirst does not achieve the expected constant-time behavior.
  • Negative modulo mistakes: In many languages, (frontIndex - 1) % capacity may be negative. Use (frontIndex - 1 + capacity) % capacity.
  • Forgetting empty cases: Remove and peek operations need defined behavior when the deque has no elements.
  • Not resetting after resize: When a circular array deque resizes, copying logical elements to the front of the new array and setting frontIndex = 0 simplifies later operations.
  • Broken backward links: In a doubly linked deque, every update must preserve both next and prev relationships.

Summary

A deque is a double-ended queue. It supports adding, removing, and peeking at both the front and the back. Efficient implementations use a circular array or a doubly linked list so end operations do not require shifting the whole sequence.
