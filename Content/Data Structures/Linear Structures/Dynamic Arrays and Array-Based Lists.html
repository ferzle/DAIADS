Dynamic Arrays and Array-Based Lists

Overview

A list is an ordered collection whose elements occupy positions 0 through n - 1. A dynamic array is one of the most common ways to implement that list idea: it stores the elements in a contiguous backing array, keeps track of the current logical size, and allocates a larger backing array when it runs out of room.

Java ArrayList, C++ vector, and Python list all behave like dynamic arrays. They feel flexible because the list can grow, but internally they still rely on array storage. That means they inherit the strengths of arrays, especially constant-time indexed access and good cache locality.

Dynamic arrays are also useful for understanding the difference between an abstract data type and an implementation. The same list ADT can be implemented with a linked-based list, but the operation costs change: array-based lists are fast at indexing and appending, while linked-based lists are better at local insertion or deletion once the relevant node is already known.

List ADT and Operations

The list ADT describes what the user can ask for, not how the data is stored. A typical positional list supports these operations:

  • get(i): Return the element at index i.
  • set(i, value): Replace the element at index i.
  • append(value): Add value after the current last element.
  • insert(i, value): Add value so it becomes the element at index i.
  • remove(i): Remove and return the element at index i.
  • size(): Return the number of logical elements in the list.
  • isEmpty(): Report whether the list has no elements.

In an array-based implementation, the index is central. The implementation can jump straight to index i, but it must shift elements when an insertion or deletion changes the middle of the sequence. In a linked-based implementation, the nodes are connected by references, so reaching index i usually requires traversal.

Conceptual Model

A dynamic array stores two different quantities:

  • Size: the number of meaningful elements currently in the list.
  • Capacity: the number of slots currently allocated in the backing array.

For example, the logical list [12, 4, 18, 7] might be stored in a backing array with capacity 8:

Index 0 1 2 3 4 5 6 7
Value 12 4 18 7 unused unused unused unused

The unused slots are real memory, but they are not part of the list. Client code should see a list of size 4, not an array of length 8. The extra capacity is reserved so future appends can be fast.

A linked-based list represents the same logical sequence differently: 

head -> [12 | next] -> [4 | next] -> [18 | next] -> [7 | next] -> null

Both structures can represent [12, 4, 18, 7]. The difference is not the abstract sequence; the difference is how the sequence is stored and which operations become cheap.

Worked Example

Suppose a dynamic array has size 5, capacity 8, and logical contents [3, 9, 2, 6, 5]. We call insert(2, 14), so 14 should become the new value at index 2.

  1. Check that index 2 is a valid insertion position.
  2. Check whether size equals capacity. Here it does not, so no resize is needed.
  3. Shift 5 from index 4 to index 5.
  4. Shift 6 from index 3 to index 4.
  5. Shift 2 from index 2 to index 3.
  6. Write 14 at index 2.
  7. Increase size from 5 to 6.

The final logical list is [3, 9, 14, 2, 6, 5]. The operation is linear in the worst case because many later values may have to move. If the same value were appended at the end, no shifting would be needed.

Now compare a linked-based list. If we already have a reference to the node containing 9, inserting 14 after it only changes a few links. But if the operation is still expressed as insert(2, 14), the linked list must first walk from the head to index 1. That search cost is part of the operation.

Core Operations

Indexed Access and Update

For get(i) and set(i, value), an array-based list uses the same direct indexing idea as a normal array. After a bounds check, the implementation reads or writes the backing array at index i. No other elements move.

Append

Appending writes the new value at index size and then increments size. If there is spare capacity, this is constant time. If the backing array is full, the dynamic array must resize first. 

append(value):
    if size == capacity:
        resize to a larger capacity
    A[size] = value
    size = size + 1

Resize

Resizing allocates a new backing array, usually about twice the old capacity, copies the current logical elements into it, and then makes the new array the backing storage. The resize itself costs \(O(n)\), but it happens rarely enough that a long sequence of appends still has \(O(1)\) amortized cost per append.

Doubling capacity is the standard teaching policy because it spreads copying work over many future appends. Growing by only one slot at a time would make repeated append unnecessarily expensive.

Insert and Remove

Inserting in the middle creates a hole by shifting later elements one position right. Removing from the middle fills the gap by shifting later elements one position left. The closer the operation is to the front, the more elements may move. 

insert(i, value):
    check that 0 <= i <= size
    if size == capacity:
        resize to a larger capacity
    for j from size down to i + 1:
        A[j] = A[j - 1]
    A[i] = value
    size = size + 1

remove(i):
    check that 0 <= i < size
    removed = A[i]
    for j from i to size - 2:
        A[j] = A[j + 1]
    size = size - 1
    return removed

Implementation Options

Implementation 1: Array-Based List

An array-based list stores a backing array, a logical size, and a physical capacity. Its main invariant is that positions 0 through size - 1 hold the list in order. Positions from size through capacity - 1 are unused.

Operation Sketch

  • Access: compute the address from the index and read the value.
  • Append: write at A[size], resizing first if needed.
  • Insert: shift later values right, then write the new value.
  • Remove: save the removed value, shift later values left, and decrement size.
Implementation Exercise

Implement an integer array-based list with get, set, append, insert, remove, size, and isEmpty.

  • Keep logical size separate from physical capacity.
  • Double capacity when appending or inserting into a full backing array.
  • Reject invalid indices before touching the backing array.
  • Test empty lists, one-element lists, insertion at the front, insertion at the end, and removal at the end.

Implementation 2: Linked-Based List

A linked-based list stores values in nodes connected by references. It can support the same list ADT, but positions are not mapped directly to memory addresses. To access index i, the implementation usually starts at the head and follows next links until it reaches that position.

Linked-based lists are useful when updates happen near already-known nodes. If the program has a reference to the node before the insertion point, inserting a new node only requires a few reference changes. If the program has only an index or a value to search for, the traversal cost may dominate.

Notes

  • Singly linked lists use one forward link per node.
  • Doubly linked lists use forward and backward links, making some removals and deque operations easier.
  • Linked nodes avoid array resizing, but each element has node and reference overhead.

See the Linked Lists page for a fuller treatment of node-based lists.

Library Implementations

  • Java: ArrayList<E> is the standard dynamic-array list; LinkedList<E> is linked-based.
  • C++: std::vector is a dynamic array; std::list is a doubly linked list.
  • Python: list is dynamic-array-like; collections.deque is better for efficient end operations at both ends.

In ordinary programming, dynamic arrays are often the default sequence type because indexing, scanning, and appending are efficient in practice. Linked lists are still important when the assignment or workload is about reference manipulation, stable node positions, or frequent local splicing.

Operation Costs

Operation Array-Based List Linked-Based List Notes
get(i), set(i) \(O(1)\) \(O(n)\) Linked lists must traverse to the position.
append \(O(1)\) amortized \(O(1)\) with tail, \(O(n)\) without tail Dynamic arrays occasionally resize.
insert(i, value) \(O(n)\) \(O(n)\) by index Array shifts; linked list traverses.
insert after known node \(O(n)\) if position must be opened \(O(1)\) This is where linked lists can be strongest.
remove(i) \(O(n)\) \(O(n)\) by index Array shifts; linked list must find the predecessor or node.
extra space unused capacity node and link fields Dynamic arrays are usually more compact per element.

The phrase amortized matters. A single append can still cost \(O(n)\) if it triggers a resize, but over a long sequence of appends, the average cost per append is constant when capacity grows geometrically.

Applications

  • General-purpose sequences: Dynamic arrays are the usual default when programs need an ordered collection with fast indexing.
  • Stacks: A dynamic array can implement a stack by treating the end of the list as the top.
  • Heaps: Binary heaps use array indexing to represent complete binary trees compactly.
  • Dynamic programming: Tables and memo arrays often rely on direct index access.
  • Accumulating results: Algorithms commonly append outputs to a dynamic array when the final number of results is not known in advance.
  • Small lists in performance-sensitive code: Contiguous storage often beats pointer-heavy structures because scans make good use of caches.

Comparisons and Tradeoffs

  • Array-based vs. linked-based lists: Array-based lists are better for indexed access and compact storage. Linked-based lists are better for local updates after known nodes.
  • Append vs. front insertion: Appending to a dynamic array is usually cheap. Inserting at the front usually shifts every existing element.
  • Worst case vs. amortized cost: Resizing creates expensive individual appends, but geometric growth keeps the long-run append cost small.
  • Memory locality vs. pointer flexibility: Dynamic arrays store elements next to each other. Linked lists scatter nodes but can relink without moving the stored values.
  • Library use vs. implementation practice: Use library dynamic arrays in production. Implementing one by hand is valuable because it exposes size, capacity, resizing, and shifting behavior.

Common Pitfalls

  • Confusing size and capacity: Unused capacity is not part of the list and should not be searched, printed, or returned.
  • Shifting in the wrong direction: Insertions shift right from the end toward the insertion point. Deletions shift left from the removed position toward the end.
  • Forgetting insertion at size: Inserting at index size is valid and is the same as append.
  • Calling append worst-case constant time: Append is amortized \(O(1)\), but a resize still costs \(O(n)\).
  • Leaving stale references: After removal, some implementations should clear the old final slot so removed objects can be garbage collected.
  • Ignoring traversal in linked lists: A linked-list insertion is constant time only after the relevant node has already been found.

Summary

Dynamic arrays implement the list ADT with contiguous backing storage, a logical size, and a physical capacity. They provide constant-time indexed access and amortized constant-time append, but middle insertions and removals require shifting. Linked-based lists represent the same abstract sequence with nodes and links, trading away direct indexing for flexible local updates.

Reading Comprehension Questions

  1. Size vs. Capacity: In a dynamic array, what is the difference between size and capacity?
  2. Append: Why is append amortized \(O(1)\) instead of worst-case \(O(1)\)?
  3. Insert: Why does inserting at index 0 usually take linear time?
  4. Linked Comparison: When can a linked-based list insert in constant time?
  5. Indexing: Why is get(i) constant time in an array-based list but linear in a linked-based list?
  6. Stale Slot: After removing the final logical element, why might an implementation clear the old slot?
  1. Answer: Size is the number of meaningful elements; capacity is the number of allocated backing-array slots.
  2. Answer: Most appends are constant time, but an append that triggers resizing must allocate a new array and copy existing elements.
  3. Answer: Every existing element must shift one position right to preserve order.
  4. Answer: When the program already has a reference to the node before the insertion point.
  5. Answer: Array indices map directly to positions in memory, while a linked list must follow links from an end to reach index i.
  6. Answer: If the slot holds a reference, clearing it can allow the removed object to be garbage collected.
