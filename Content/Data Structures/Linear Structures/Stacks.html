Stacks

Overview

A stack stores values so that the most recently added value is the first one removed. This rule is called last-in, first-out, or LIFO.

The active end of a stack is called the top. New values are pushed onto the top, and removals also happen from the top. Values below the top are still stored, but the stack interface does not allow the caller to reach into the middle or remove from the bottom.

This restriction is useful because many programs need to remember unfinished work in reverse order. Function calls, recursion, undo history, expression evaluation, depth-first search, and backtracking all rely on stack behavior. A stack is an abstract data type, so the LIFO rule matters more than the particular storage representation. The same stack behavior can be implemented with an array, a linked list, or a library container.

When we want to represent a stack using text, we typically show the top of the stack at the right side of the list and the bottom at the left side. For example, the stack from the diagram above would be written as [14, 8, 31], with 31 at the top.

Core Operations

The following are the core operations of a stack ADT. The ones in bold are the essential LIFO operations. The exact method signatures and error-handling policies may vary, but the same basic operations are needed to maintain the LIFO abstraction.

Operation Description
push(value) Add value to the top of the stack. If the stack is full, the implementation must report overflow or use a documented full-stack policy.
pop() Remove and return the top value. If the stack is empty, the implementation must report underflow or use a documented empty-stack policy.
peek() Return the top value without removing it. It has the same empty-stack issue as pop().
isEmpty() Report whether the stack currently stores no values.
size() Return the number of values currently stored.
isFull() (Optional) For a bounded stack (i.e. one with fixed capacity), report whether the stack currently stores the maximum number of values.

As seen above, a bounded stack needs an overflow policy for push when it is full, as well as a method isFull() to check its status. Depending on the implementation, this might mean throwing an exception, returning a boolean, or otherwise signaling the overflow condition. On the other hand, a resizable stack can grow its backing storage instead, but even a resizable stack can run out of memory in a real program.

Here are a few more details about the essential operations of a stack.

Push

push makes a new value the top value. It does not disturb the relative order of the values already in the stack; it only places a new item above them. If the stack previously stored [4, 7], then push(9) updates the stack to [4, 7, 9].

The caller should not need to know where that new value is stored physically. An array stack might write into the next open array cell, while a linked stack might allocate a new head node. Both are still the same abstract operation. We will discuss both implementations in more detail later.

Pop

pop removes the current top and returns it to the caller. The operation changes the stack's state. Calling pop() on a stack with contents [4, 7, 9] returns 9 and leaves the stack with contents [4, 7].

Popping from an empty stack is called underflow. A clean stack implementation should define this case explicitly, usually by throwing an exception, returning an optional value (often a boolean), or otherwise reporting an error.

Peek

peek reads the top value without removing it. Calling peek() on [4, 7, 9] returns 9 and leaves the stack unchanged. Peeking at an empty stack should be dealt with in the same way as it was for pop.

peek is useful when the next decision depends on the top value but the value should remain available for later operations. Parentheses matching, for example, often peeks at the most recent unmatched opening delimiter before deciding whether a closing delimiter matches it.

Example Trace of Stack Operations

Below we can see what happens when we start with an empty stack that can hold up to 4 elements (a bounded stack) and perform the following sequence of operations. pop(), push(4), push(7), push(9), pop(), push(2), peek(), push(5), push(8), pop().

Operation Stack Afterward Return Value
pop() [] underflow
push(4) [4] none
push(7) [4, 7] none
push(9) [4, 7, 9] none
pop() [4, 7] 9
push(2) [4, 7, 2] none
peek() [4, 7, 2] 2
push(5) [4, 7, 2, 5] none
push(8) [4, 7, 2, 5] overflow
pop() [4, 7, 2] 5

In the example, we specified return values of underflow and overflow. The most common ways of actually doing this are to return a boolean or throw an exception. In the former case, instead of having "none" as a return value, the calls to push would return true when the push succeeded and false when it failed. pop and peek are more difficult to handle with a boolean return because they also need to return a value when successful. If the stack is storing objects, one option is to return null on underflow, but that only works if null is not a valid stack value. Another option is to return a known invalid value, such as -1, but that only works if the stack is storing integers and -1 is not a valid value. The fallback is to throw an exception, which is more flexible but also more complex to use because the caller must handle the possibility of an exception.

Implementation Options

The stack ADT can be implemented using several representations. The best representation depends on whether the program values compact storage, predictable fixed capacity, avoidance of resizing, or compatibility with a language library. The two standard teaching implementations are an array-backed stack and a linked stack.

Implementation 1: Array-Backed Stack

An array-backed stack stores the bottom at index 0 and the top at index size - 1. The main invariant is that only positions 0 through size - 1 are logically part of the stack. Slots at positions size and beyond are unused capacity.

Placing the top at the logical end of the array is deliberate. Adding or removing at the end avoids shifting elements, whereas using the front of an ordinary array would move every existing value on each operation.

How the Operations Work

Push

If there is free capacity, push writes the new value at index size and then increments size. If the backing array is full, a bounded stack reports overflow, while a resizable stack allocates a larger array and copies the existing values before writing the new one.

Pop

pop first checks whether size is zero. If the stack is nonempty, it decrements size and returns the value that was stored at the old top index. In languages with references, clearing the old slot can help garbage collection, but that clearing is not part of the abstract operation.

Peek, Size, and Empty Checks

peek returns A[size - 1] after checking for an empty stack. size() returns the stored count, and isEmpty() checks whether that count is zero.

Operation Sketch

push(value):
    if size == capacity:
        resize the backing array
    A[size] = value
    size = size + 1

pop():
    if size == 0:
        report underflow
    size = size - 1
    return A[size]

peek():
    if size == 0:
        report underflow
    return A[size - 1]

Implementation Analysis

The normal stack operations touch only the top position. The only expensive event is resizing, and resizing happens infrequently if capacity grows geometrically.

  • push: \(O(1)\) when there is room. With doubling resize, it is \(O(1)\) amortized, though an individual resizing push costs \(O(n)\).
  • pop: \(O(1)\), because it only reads the top slot and updates size.
  • peek, size, and isEmpty: \(O(1)\).
  • Extra space: The backing array may contain unused capacity. The stack does not need a separate node for each value.

Implementation 2: Linked Stack

A linked stack stores the top as the head of a singly linked list. Pushing creates a new head node, and popping removes the current head node. The main invariant is that following next references from head visits the stack from top to bottom.

No tail reference is needed, because all stack work happens at the top. A tail would point to the bottom, but the stack interface never removes from the bottom.

How the Operations Work

Push

push allocates a new node, stores the value in that node, points the new node at the old head, and then makes head refer to the new node. This makes the new value the top without scanning the existing chain.

Pop

pop checks for an empty stack, saves head.value, moves head to head.next, decreases the size, and returns the saved value. Saving the value before changing head prevents the old top from becoming inaccessible before it is returned.

Peek, Size, and Empty Checks

peek reads head.value without changing head. A linked stack can keep a stored size field so size() is constant time; without that field, counting nodes would require traversal.

Operation Sketch

push(value):
    node = Node(value)
    node.next = head
    head = node
    size = size + 1

pop():
    if head == null:
        report underflow
    value = head.value
    head = head.next
    size = size - 1
    return value

Implementation Analysis

A linked stack never shifts values and never copies the whole structure to grow. Each update changes only the head reference and, when tracked, the size field.

  • push: \(O(1)\), because it creates one node and changes a constant number of references.
  • pop: \(O(1)\), because it removes the head node.
  • peek and isEmpty: \(O(1)\). size() is \(O(1)\) if the implementation maintains a size field.
  • Extra space: Each value requires a node and at least one reference in addition to the value itself.

Library Implementations

In everyday programming, it is usually better to use the standard library's stack-like structure than to write a new one. In a data structures course, implementing stacks by hand is still valuable because it clarifies the ADT, boundary cases, and representation tradeoffs.

  • Java: ArrayDeque<E> is usually preferred for stack behavior.
  • C++: std::stack is a container adapter.
  • Python: list.append and list.pop provide efficient stack operations at the end.

Java's older Stack class exists, but modern Java code usually uses ArrayDeque for stack-like behavior because it is more flexible and avoids legacy design issues. In C++, std::stack intentionally hides most operations of its underlying container.

Implementation Cost Summary

Both main implementations make the core stack operations constant time. The practical differences are memory layout, resizing behavior, and per-element overhead.

Operation Array-Backed Stack Linked Stack Notes
push \(O(1)\) amortized \(O(1)\) Array resizing occasionally costs \(O(n)\).
pop \(O(1)\) \(O(1)\) Both remove from the top.
peek \(O(1)\) \(O(1)\) Both read the top.
search \(O(n)\) \(O(n)\) Search is not a core stack operation.
extra space per element low, plus possible unused capacity one node/reference overhead per value Array stacks are usually more compact.

A stack's asymptotic costs are simple, but implementation details still matter. An array stack usually has better locality and less per-item overhead. A linked stack avoids resizing copies, but each pushed value requires a new node allocation.

Applications

Stacks appear whenever the newest unfinished item should be handled next. The structure is small, but the LIFO rule fits many programming and algorithmic situations.

  • Function calls and recursion: Programming languages use a call stack to remember active function calls. Each stack frame stores enough information to resume the caller after the current function returns.
  • Depth-first search and backtracking: A stack stores the most recently discovered unfinished state. This makes it natural for maze search, DFS, N-Queens, subset search, and other algorithms that explore one path deeply before returning to alternatives.
  • Expression processing: Stacks are used for balanced-symbol checking, infix-to-postfix conversion, postfix evaluation, and parse-tree construction. The LIFO rule matches the way nested expressions must close in reverse order.
  • Undo and redo history: Editors often push completed actions onto an undo stack. A redo stack can store actions that were undone, and a new edit usually clears the redo stack because the future history has changed.
  • Explicit recursion simulation: A recursive algorithm can often be rewritten using an explicit stack of pending subproblems. This is useful when recursion depth might exceed the language's call-stack limit or when the program needs more control over traversal order.
  • Nested-structure validation: Compilers, interpreters, and markup processors use stack-like logic to track nested braces, parentheses, tags, or scopes. The next closing symbol should match the most recent unmatched opening symbol.
  • Free lists and resource pools: Some systems maintain reusable objects in a stack-like free list. The most recently released object can be reused first, giving simple constant-time allocation and deallocation from the pool.
  • Monotonic scans: A monotonic stack stores candidates in sorted order while scanning an array. It is a common tool for next-greater-element, stock-span, and histogram-area problems.

Comparisons and Tradeoffs

A stack is intentionally less flexible than a general sequence. That smaller interface is a strength when an algorithm depends on reverse-order processing, but it is a poor fit when the program needs arbitrary access.

  • Stack vs. general list: A stack exposes only top-based operations. This restriction makes algorithms easier to reason about because values leave in exactly the reverse of the order they entered.
  • Stack vs. queue: A stack removes the newest item first, while a queue removes the oldest item first. Use a stack when unfinished work should resume in reverse order; use a queue when arrival order matters.
  • Array stack vs. linked stack: An array stack is compact and usually cache-friendly. A linked stack avoids resizing copies but allocates one node per value and pays reference overhead.
  • Bounded vs. unbounded stack: A bounded stack has a fixed capacity and must define overflow behavior. An unbounded or dynamically resizing stack is easier to use but still has practical memory limits.
  • ADT vs. implementation: The stack ADT does not require iteration, search, or random access. Some backing containers can perform those operations, but exposing them changes the abstraction.
  • Implicit vs. explicit stack: Recursion uses the language runtime's implicit call stack. An explicit stack stores the same kind of pending work in programmer-controlled data.

Variations and Extensions

Many useful structures are stacks with an added constraint, extra operation, or specialized interpretation. These variants are still easiest to understand after the basic LIFO behavior is clear.

  • Bounded stack: A bounded stack has fixed capacity and reports overflow when full. This is common in embedded systems or assignments where memory limits are explicit.
  • Resizable stack: A resizable stack grows its backing storage when it fills. This is the usual dynamic-array approach and gives amortized constant-time pushes.
  • Min stack: A min stack augments each push with enough information to return the current minimum in \(O(1)\). A common implementation stores a second stack of running minima.
  • Call stack: A call stack stores activation records for function calls. It is not just a data-structure exercise; it is part of how ordinary programs execute.
  • Two-stack queue: Two stacks can implement FIFO queue behavior by moving items between an input stack and an output stack. This is a useful example of composing ADTs.
  • Monotonic stack: A monotonic stack maintains increasing or decreasing order among candidates. It solves several linear-time array problems that would otherwise seem to require nested loops.

Common Pitfalls

Most stack mistakes come from breaking the LIFO abstraction or forgetting boundary cases.

  • Ignoring underflow: pop and peek need a defined behavior on an empty stack. Returning an arbitrary value or accessing invalid memory hides a real error.
  • Choosing the wrong array end: The top of an array-backed stack should usually be the logical end of the array. Using the front causes unnecessary shifts on every push or pop.
  • Confusing LIFO and FIFO: A stack and queue can contain the same values but produce different removal orders. Many traversal bugs come from choosing the wrong one.
  • Returning before saving the value: A pop operation must save the top value before changing the structure in a way that makes the value inaccessible. This is especially visible in linked implementations.
  • Overusing stack operations: If an algorithm repeatedly searches inside a stack or removes from the middle, it probably wants a different ADT. The stack abstraction is designed around the top.
  • Forgetting call-stack limits: Recursive code uses stack space even when no explicit stack object appears in the program. Very deep recursion can overflow the call stack.

Summary

A stack stores items so the most recent item is removed first. Its core operations are push, pop, peek, isEmpty, and often size. Array-backed stacks are compact and have amortized constant-time pushes when resizable; linked stacks avoid resizing but use extra node storage. Use a stack when the problem naturally processes unfinished work in reverse order.
