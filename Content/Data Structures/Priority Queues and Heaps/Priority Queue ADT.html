Priority Queue ADT

Overview

A priority queue is an abstract data type for storing items where each item has a priority. Removal is based on priority, not on arrival order. In a min-priority queue, the item with the smallest key is removed first. In a max-priority queue, the item with the largest key is removed first.

Priority queues are different from ordinary FIFO queues. A FIFO queue answers "Who arrived first?" A priority queue answers "Which item is most urgent according to the priority rule?" This distinction is why priority queues are central to algorithms such as Dijkstra's shortest path algorithm, Prim's minimum spanning tree algorithm, event simulation, scheduling, and Huffman coding.

The priority queue is an ADT. It can be implemented with an unsorted array, a sorted array, a linked structure, a binary heap, a balanced search tree, or a more specialized heap family. The ADT describes the behavior; the implementation determines the costs.

Abstract Data Type and Operations

A priority queue stores entries made from an item and a priority key. The key may be the item itself, such as an integer priority, or a separate value attached to a record, such as a graph vertex's current distance estimate.

  • insert(item, priority): Add a new entry to the priority queue.
  • peek(): Return the highest-priority entry without removing it.
  • extract(): Remove and return the highest-priority entry.
  • changePriority(item, newPriority): Change an existing item's priority if the implementation supports it.
  • isEmpty(): Report whether the priority queue contains no entries.
  • size(): Return the number of entries currently stored.

The word highest depends on the convention. In a min-priority queue, a smaller key means higher priority. In a max-priority queue, a larger key means higher priority. Always state the convention before analyzing or tracing an algorithm.

Conceptual Model

Suppose a min-priority queue stores emergency-room patients, where smaller numbers mean more urgent treatment.

Patient Priority
Ana 4
Bo 2
Cy 5
Dia 1

The next extraction returns Dia, not Ana, because priority 1 is the smallest key. The structure does not have to keep all entries sorted visibly; it only has to provide the correct highest-priority entry when asked.

Ties require a policy. Some priority queues allow tied entries to come out in any order. Others are stable, meaning tied priorities leave in the same order they arrived. Stability is not automatic; it usually requires adding an insertion counter or another tie-breaking rule.

Worked Example

Trace this min-priority queue sequence. Lower numbers have higher priority: insert(A, 5), insert(B, 3), insert(C, 7), peek(), insert(D, 2), extract(), extract(), insert(E, 4), extract().

Operation Entries Afterward Return Value
insert(A, 5) {A:5} none
insert(B, 3) {A:5, B:3} none
insert(C, 7) {A:5, B:3, C:7} none
peek() {A:5, B:3, C:7} B
insert(D, 2) {A:5, B:3, C:7, D:2} none
extract() {A:5, B:3, C:7} D
extract() {A:5, C:7} B
insert(E, 4) {A:5, C:7, E:4} none
extract() {A:5, C:7} E

Notice that C arrived before E, but E is extracted first because its priority is better. Arrival order only matters if the priority queue uses it as a tie-breaker.

Core Operations

Insert

insert adds a new entry. The implementation may place it anywhere internally as long as future peek and extract operations still return the correct entry. In an unsorted list, insertion can be trivial. In a heap, insertion adds the entry at the end and repairs the heap property.

Peek

peek returns the current highest-priority entry without removing it. Some implementations maintain the best entry at a known position, making peek constant time. Others must scan to find it.

Extract

extract removes and returns the current highest-priority entry. This is the defining operation of the ADT. If the priority queue is empty, extraction must report underflow, throw an exception, or follow another clearly documented policy.

Changing Priority

Some algorithms need to change the priority of an item already in the queue. Dijkstra's and Prim's algorithms are often described with decreaseKey, which improves an item's key in a min-priority queue. Libraries do not always support this operation directly.

A common workaround is lazy insertion: insert the same item again with the better priority, and when an old stale copy is later extracted, ignore it. This keeps the library interface simple, but the priority queue may temporarily contain multiple entries for the same logical item.

Implementation Options

Implementation 1: Unsorted Array or List

The simplest implementation stores entries in no particular order. Insertion appends to the end. To peek or extract, the implementation scans all entries to find the best priority.

Operation Sketch

insert(item, priority):
    append (item, priority)

peek():
    scan all entries and return the one with best priority

extract():
    scan all entries to find the best priority
    remove that entry
    return it

This version is reasonable when insertions are common but extractions are rare, or when the number of entries is very small.

Implementation 2: Sorted Array or List

A sorted implementation keeps entries ordered by priority. Then peek and extract can look at one end. The cost moves to insertion, because the new entry must be placed in priority order.

If the backing structure is an array, insertion may require shifting many entries. If it is a linked list, insertion still requires a search to find the correct position, but relinking after that position is cheap.

Implementation 3: Binary Heap

A binary heap is the standard introductory implementation because it balances the main costs: insertion and extraction are both logarithmic, while peeking is constant time. The heap stores entries in an array that represents a complete binary tree and maintains the heap property.

The heap property is enough to guarantee that the best entry is at the root. It does not mean the entire array is sorted. This is one of the most important distinctions between a heap and a sorted list.

See Binary Heaps and Heap Operations for the representation and repair operations.

Implementation 4: Balanced Search Tree

A balanced search tree can also implement a priority queue by ordering entries by priority. The best entry is stored at one extreme of the tree, so insertion, extraction, and arbitrary search can all be logarithmic if the tree remains balanced.

This approach is useful when the program needs priority-queue behavior plus ordered traversal, search by key, or removal of a known entry. It usually has more pointer overhead and less cache locality than a binary heap, so a heap is still the simpler default when the only important operation is repeatedly extracting the best item.

Implementation 5: Specialized Heaps

Heap families such as binomial heaps, Fibonacci heaps, pairing heaps, and d-ary heaps make different tradeoffs. Some improve merging, some improve decrease-key bounds, and some improve practical constants by changing the branching factor. These structures are usually introduced after binary heaps because they solve more specialized performance problems.

Library Implementations

  • Java: PriorityQueue<E> is a min-priority queue by natural ordering or comparator.
  • C++: std::priority_queue is max-oriented by default, though custom comparators can change the rule.
  • Python: heapq provides heap operations on lists and is min-oriented.

Library priority queues usually support insertion, peek, and extraction. They may not support efficient arbitrary removal or change-priority operations. When an algorithm textbook uses decreaseKey, check whether the chosen language library actually provides it.

A binary heap is a good default because it avoids the extreme tradeoff of the unsorted and sorted versions. It does not make every operation constant time, but it keeps the two most important update operations logarithmic.

Applications

  • Dijkstra's algorithm: A min-priority queue repeatedly selects the unsettled vertex with the smallest current distance estimate.
  • Prim's algorithm: A min-priority queue selects the next vertex or edge that most cheaply expands the growing minimum spanning tree.
  • Huffman encoding: A min-priority queue repeatedly extracts the two lowest-frequency trees and merges them.
  • Event simulation: Future events are processed in increasing time order.
  • Scheduling: Jobs can be selected by urgency, deadline, shortest processing time, or another priority rule.
  • Top-\(k\) problems: A heap-backed priority queue can keep only the best \(k\) items seen so far.
  • Search algorithms: Best-first search and A* use priorities to choose which state to explore next.
  • Heapsort: A binary heap can repeatedly extract the maximum to produce sorted order; this is an application of the heap implementation rather than the abstract priority queue alone.

Comparisons and Tradeoffs

  • Priority queue vs. queue: A queue removes by arrival order. A priority queue removes by priority.
  • Priority queue vs. sorted list: A sorted list keeps all entries ordered. A heap keeps only enough order to find the best entry efficiently.
  • Priority queue vs. map: A map finds entries by key. A priority queue finds the entry with the best priority key.
  • Heap vs. balanced tree: A heap gives very fast access to the best item but poor arbitrary search. A balanced tree supports ordered search and traversal, often with more overhead.
  • Binary heap vs. specialized heaps: Binary heaps are simple and fast in practice. Binomial, Fibonacci, pairing, and d-ary heaps are considered when merge, decrease-key, or branching-factor tradeoffs dominate.
  • Stable vs. unstable priority queues: Stability makes ties predictable but may require storing an extra insertion counter.
  • Lazy insertion vs. decrease-key: Lazy insertion is easy with simple libraries but can leave stale entries in the queue. Decrease-key is cleaner conceptually but requires direct access to an item's heap position.

Common Pitfalls

  • Mixing min and max conventions: Be explicit about whether smaller or larger keys mean higher priority.
  • Assuming FIFO behavior: Equal priorities may not preserve arrival order unless the priority queue is designed to be stable.
  • Assuming a heap is sorted: A heap guarantees the best item at the root, not a fully sorted array.
  • Ignoring stale entries: Algorithms that use lazy insertion must check whether an extracted entry is still current.
  • Forgetting empty cases: peek and extract need defined behavior when the priority queue is empty.
  • Using the wrong library default: Java and Python are min-oriented by default; C++ std::priority_queue is max-oriented by default.
  • Leaving priorities ambiguous: Complex records need a clear comparator or priority field. Otherwise different implementations may order them differently.

Summary

A priority queue stores entries with priorities and removes the highest-priority entry first. The ADT is defined by operations such as insert, peek, and extract. Binary heaps are the standard implementation because they provide constant-time peek and logarithmic insertion and extraction, while simpler sorted or unsorted structures trade one operation's cost against another.
