Binary Heaps

Overview

A binary heap is a compact tree-shaped structure used to implement priority queues. It stores elements in a complete binary tree and maintains a simple parent-child ordering rule called the heap property.

In a max-heap, every parent is at least as large as its children, so the maximum value is at the root. In a min-heap, every parent is at most as large as its children, so the minimum value is at the root. This page uses max-heaps as the default example; min-heaps reverse the comparisons.

Binary heaps are the standard introductory heap implementation because they are simple, array-based, and fast in practice. They support efficient insert, peekMax, extractMax, heapify, and buildMaxHeap operations. The detailed mechanics and code for those operations are on the Heap Operations page.

Relationship to Priority Queues

A priority queue is an abstract data type: it says that entries are removed by priority. A binary heap is one implementation of that ADT. The heap property guarantees that the next entry to remove is easy to find, while the complete-tree shape keeps the structure compact.

A heap does not keep all entries sorted. It keeps just enough order to make the best entry available at the root. That weaker ordering is the reason heaps can insert and extract in \(O(\log n)\) time without paying the full cost of maintaining a completely sorted list.

For the broader ADT view, including unsorted lists, sorted lists, balanced search trees, and specialized heap families, see the Priority Queue ADT page.

Heap Property

The heap property is a local rule between each parent and its children.

  • Max-heap property: every parent is greater than or equal to each child.
  • Min-heap property: every parent is less than or equal to each child.

The rule is local, but it implies a useful global fact: the maximum value in a max-heap is at the root, and the minimum value in a min-heap is at the root. It does not imply that siblings are ordered, that the left child is smaller than the right child, or that an inorder traversal produces sorted output. 

        90
      /    \
    72      85
   /  \    /  \
  31  60  40  10

This is a valid max-heap. The root is the maximum. The left and right subtrees are not sorted lists; they only obey the parent-child rule.

Complete Binary Tree Shape

A binary heap is not just any binary tree. It is a complete binary tree: every level is full except possibly the last, and the last level is filled from left to right.

Completeness matters because it keeps the height small. A complete binary tree with \(n\) nodes has height \(\lfloor \log n \rfloor\), so operations that move one level at a time take \(O(\log n)\) steps in the worst case.

Completeness also makes the array representation possible. Since there are no gaps before the final node in level order, the heap can be stored compactly without explicit node objects or child references.

Array Representation

Binary heaps are usually stored in an array. The array lists the tree nodes in level-order: root first, then the next level from left to right, and so on.

With zero-based indexing, the relatives of the node at index \(i\) are:

  • left child: \(2i + 1\)
  • right child: \(2i + 2\)
  • parent: \(\lfloor(i - 1)/2\rfloor\)

These formulas are the main reason binary heaps are so compact. The implementation does not store left, right, or parent references. It computes those positions from the index.

In practice, a heap often stores a backing array and a separate heapSize. The physical array may have extra capacity, but only indices 0 through heapSize - 1 are currently part of the heap. Every child-index calculation must be checked against heapSize, not just the physical array length.

Worked Example

Suppose the array representation is: 

index:  0   1   2   3   4   5   6
value: 90  72  85  31  60  40  10

The tree view is: 

        90
      /    \
    72      85
   /  \    /  \
  31  60  40  10

For the value at index 2, the left child is at \(2(2) + 1 = 5\), the right child is at \(2(2) + 2 = 6\), and the parent is at \(\lfloor(2 - 1)/2\rfloor = 0\). In the array, those values are 40, 10, and 90.

The heap property holds because every parent is at least as large as its children. The array itself is not sorted: 72 appears before 85, but that is allowed because they are siblings in different subtrees.

Operation Overview

Heap operations preserve two invariants: the tree must stay complete, and the heap property must hold. Updates usually repair the heap by moving one value up or down a root-to-leaf path.

  • peekMax: return A[0] after checking that the heap is nonempty.
  • heapify(A, i, heapSize): repair a local violation by moving a value downward.
  • buildMaxHeap(A): turn an arbitrary array into a heap by heapifying internal nodes from bottom to top.
  • insert(A, value, heapSize): add the value at the next open leaf position and bubble it upward.
  • extractMax(A, heapSize): remove the root, move the last heap item to the root, shrink the heap, and heapify downward.

The Heap Operations page contains the operation demos, pseudocode, Java/C++/Python code, and per-operation time/space analysis.

Operation Costs

The height of a binary heap with \(n\) elements is \(\lfloor \log n \rfloor\), which determines the worst-case cost of the operations that move along one path.

Operation Binary Max-Heap Cost Reason
peekMax \(O(1)\) The maximum is always stored at the root.
insert \(O(\log n)\) The new value may bubble from a leaf to the root.
extractMax \(O(\log n)\) The replacement root may move down one path.
heapify \(O(\log n)\) The repair travels downward through at most the heap height.
buildMaxHeap \(O(n)\) Most heapify calls occur near the bottom on small subtrees.
extra space \(O(n)\) storage, \(O(1)\) auxiliary for iterative operations The array stores the heap compactly without node references.

Heapsort Connection

Heapsort is a sorting algorithm built directly from binary heap operations. First, build a max-heap from the input array. Then repeatedly swap the root with the last item in the current heap, shrink the heap size, and heapify the root. Each extraction places the next largest value into its final sorted position.

This is why heapSize is useful: during Heapsort, the same physical array contains a heap prefix and a sorted suffix. The heap operations must treat only the prefix as active. See Heapsort for the full sorting algorithm.

Applications

  • Priority queues: Binary heaps are the usual first implementation choice for repeated insert and extract operations.
  • Graph algorithms: Dijkstra's algorithm and Prim's algorithm commonly use min-heaps to select the next vertex or edge.
  • Selection problems: A heap can keep the top \(k\) or bottom \(k\) items seen so far.
  • Event simulation: A min-heap can process future events in increasing time order.
  • Huffman encoding: A min-heap repeatedly provides the two lowest-frequency trees.
  • Heapsort: A max-heap supports in-place \(O(n \log n)\) sorting by repeated extraction.

Comparisons and Tradeoffs

  • Heap vs. sorted array: A sorted array gives easy access to the maximum or minimum, but inserting into sorted position can require shifting. A heap keeps insertion logarithmic.
  • Heap vs. unsorted array: An unsorted array can insert cheaply, but finding the best item requires scanning. A heap makes extraction logarithmic.
  • Heap vs. binary search tree: A heap gives fast access to the extreme value but does not support ordered search well. A balanced search tree supports ordered search and traversal, often with more overhead.
  • Array heap vs. linked tree: Array heaps avoid pointer overhead and have good cache locality because elements are packed contiguously.
  • Binary heap vs. specialized heaps: Binary heaps are simple and usually fast in practice. Binomial, Fibonacci, pairing, and d-ary heaps target more specialized operation mixes.

Variations and Extensions

  • Min-heaps: Reverse the comparisons so the minimum value is stored at the root.
  • d-ary heaps: Give each node more than two children, reducing height but increasing the number of child comparisons during downward repair.
  • Binomial heaps: Organize elements as a forest of heap-ordered trees and support efficient merging.
  • Fibonacci heaps: Improve some theoretical bounds, especially for repeated decrease-key operations.
  • Pairing heaps: A practical meldable heap family often discussed as a simpler alternative to Fibonacci heaps.

Common Pitfalls

  • Assuming the heap array is sorted: Only parent-child order is guaranteed.
  • Using the wrong index formulas: The formulas on this page are for zero-based indexing; one-based heaps use different formulas.
  • Checking against array length instead of heapSize: Extra capacity or a Heapsort suffix may exist outside the active heap.
  • Confusing min-heaps and max-heaps: The direction of every comparison must match the priority convention.
  • Forgetting empty-heap behavior: peekMax and extractMax need a defined policy when the heap has no elements.
  • Breaking the shape invariant: Heap insertion must add at the next open leaf position, not at an arbitrary tree location.

Summary

A binary heap is a complete binary tree stored compactly in an array. The heap property puts the maximum or minimum at the root without fully sorting the structure. This gives binary heaps their useful balance: constant-time peek, logarithmic insertion and extraction, and linear-time heap construction.

Reading Comprehension Questions

  1. Shape vs. Order: Why does the completeness of the binary tree make an array implementation possible?
  2. Heap Property: What does a max-heap guarantee about each parent and its children?
  3. Not Fully Sorted: What ordering information does a max-heap guarantee, and what does it not guarantee?
  4. Index Formulas: In a zero-based heap array, where are the children of index i?
  5. Heap Size: Why might heapSize be smaller than the physical array length?
  6. Heapsort: Why is it useful for Heapsort to keep a heap prefix and a sorted suffix in the same array?
  1. Answer: A complete binary tree has no internal gaps, so its nodes can be listed level by level in a contiguous array while preserving fixed parent-child index formulas.
  2. Answer: Every parent is greater than or equal to each of its children.
  3. Answer: A max-heap guarantees that every parent is at least as large as its children, so the root is the maximum. It does not guarantee that siblings or nodes in different branches are in sorted order.
  4. Answer: The left child is at \(2i + 1\), and the right child is at \(2i + 2\).
  5. Answer: The array may have extra capacity, or part of the array may hold values outside the active heap, as in Heapsort.
  6. Answer: It lets the algorithm sort in place: the active heap shrinks while extracted maximum values accumulate in final sorted positions.

In-Class Activities

  1. Index Mapping Practice: Give students an array representation of a heap and ask them to label the parent, left child, and right child of selected indices.
  2. Heap or Not: Give several arrays and ask students which ones represent valid max-heaps.
  3. Tree-to-Array: Draw a complete binary tree and convert it into level-order array storage.
  4. Not Sorted: Ask students to construct two different valid max-heaps using the same set of values.
  5. Application Match: Decide whether a heap, sorted array, unsorted array, or balanced search tree fits several priority-queue workloads.

Homework Problems

  1. Min-Heap Conversion: Rewrite the heap property and index-based operation descriptions for a min-heap.
  2. Array Validation: Write an algorithm that determines whether an array represents a valid max-heap.
  3. Height Bound: Explain why a complete binary tree with \(n\) nodes has height \(\Theta(\log n)\).
  4. Heapsort Trace: Starting from an unsorted array, build a max-heap and then trace the repeated root swaps used by Heapsort.
  5. Implementation Comparison: Compare binary heaps, sorted arrays, and balanced search trees for a workload with many inserts, many extracts, and occasional searches by key.
  6. Variant Research: Choose binomial heaps, Fibonacci heaps, pairing heaps, or d-ary heaps and summarize what operation tradeoff motivates that structure.
