Heap Operations

Overview

This page explains the operations that maintain and use a binary max-heap: peekMax, heapify, buildMaxHeap, insert, and extractMax. Each operation preserves the heap's two invariants: the tree remains complete, and every parent is at least as large as its children.

The page assumes the representation from Binary Heaps: a complete binary tree stored in an array, with a separate heapSize marking how much of the array belongs to the active heap.

We focus on max-heaps to keep the comparisons consistent. A min-heap uses the same structure and the same movement patterns, but reverses the comparisons.

Representation Reminder

With zero-based indexing, the relatives of the node at index \(i\) are:

  • left child: \(2i + 1\)
  • right child: \(2i + 2\)
  • parent: \(\lfloor(i - 1)/2\rfloor\)

The active heap contains indices 0 through heapSize - 1. Any index greater than or equal to heapSize is outside the heap, even if it is still inside the physical array.

Operation Summary

The operations differ in where a possible heap-property violation begins and which direction the repair moves.

Operation Main Idea Worst-Case Time
peekMax Read the root. \(O(1)\)
heapify Move a too-small parent downward. \(O(\log n)\)
buildMaxHeap Call heapify bottom-up on internal nodes. \(O(n)\)
insert Add at the next leaf and bubble upward. \(O(\log n)\)
extractMax Remove the root, replace it with the last item, then heapify downward. \(O(\log n)\)

peekMax(A, heapSize)

The maximum value in a max-heap is always at the root, which is stored at index 0. Therefore peekMax only needs to check that the heap is nonempty and return A[0]. 

peekMax(A, heapSize):
    if heapSize == 0:
        report underflow
    return A[0]

Time and space analysis: The operation does constant work and uses constant extra space, so it runs in \(O(1)\) time and uses \(O(1)\) auxiliary space.

heapify(A, i, heapSize)

heapify restores the max-heap property at index \(i\), assuming both child subtrees are already valid heaps. The value at \(i\) may be too small. The operation compares it with its children, swaps it with the larger child if needed, and repeats from the child position.

  1. Compute the left and right child indices.
  2. Find the largest value among the parent and the existing children.
  3. If the parent is already largest, stop.
  4. Otherwise, swap with the larger child and continue heapifying from that child.

Implementation in Java, C++, and Python

The Java and C++ snippets assume a helper named swap. The Python version swaps directly. 

public void heapify(int[] A, int i, int heapSize) {
    int largest = i;
    int left = 2 * i + 1;
    int right = 2 * i + 2;

    if (left < heapSize && A[left] > A[largest]) {
        largest = left;
    }
    if (right < heapSize && A[right] > A[largest]) {
        largest = right;
    }

    if (largest != i) {
        swap(A, i, largest);
        heapify(A, largest, heapSize);
    }
}

def heapify(A, i, heapSize):
    largest = i
    left = 2 * i + 1
    right = 2 * i + 2

    if left < heapSize and A[left] > A[largest]:
        largest = left
    if right < heapSize and A[right] > A[largest]:
        largest = right

    if largest != i:
        A[i], A[largest] = A[largest], A[i]
        heapify(A, largest, heapSize)

Time and space analysis: Each recursive call moves one level down the heap. Since a complete binary tree has height \(\lfloor \log n \rfloor\), heapify runs in \(O(\log n)\) time. The recursive version uses \(O(\log n)\) call-stack space in the worst case; an iterative version uses \(O(1)\) auxiliary space.

buildMaxHeap(A)

buildMaxHeap converts an arbitrary array into a valid max-heap. It starts at the last internal node, index \(\lfloor n/2 \rfloor - 1\), and calls heapify while moving backward to the root. This order works because every child subtree is repaired before its parent is processed.

Implementation in Java, C++, and Python

public void buildMaxHeap(int[] A) {
    int n = A.length;
    for (int i = n / 2 - 1; i >= 0; i--) {
        heapify(A, i, n);
    }
}
void buildMaxHeap(int A[], int n) {
    for (int i = n / 2 - 1; i >= 0; i--) {
        heapify(A, i, n);
    }
}
def build_max_heap(A):
    n = len(A)
    for i in range(n // 2 - 1, -1, -1):
        heapify(A, i, n)

Time and space analysis: At first glance, buildMaxHeap looks like \(O(n \log n)\), because it makes about \(n/2\) calls to heapify. The sharper bound is \(O(n)\): most nodes are near the bottom of the heap, where heapify can move only a short distance. The iterative loop uses \(O(1)\) auxiliary space, aside from any stack space used by heapify.

insert(A, value, heapSize)

Insertion preserves the complete-tree shape by placing the new value at the next open leaf position, index heapSize. The value may be larger than its parent, so the operation then bubbles up: repeatedly swap the value with its parent until the parent is larger or the value reaches the root.

Implementation in Java, C++, and Python

These snippets show the operation core and return the updated heapSize. A full heap class would usually store heapSize as a field and resize the backing array when it is full. 

public int insert(int[] A, int value, int heapSize) {
    A[heapSize] = value;
    heapSize++;
    int i = heapSize - 1;

    while (i > 0 && A[(i - 1) / 2] < A[i]) {
        int parent = (i - 1) / 2;
        swap(A, i, parent);
        i = parent;
    }
    return heapSize;
}

}
def insert(A, value, heapSize):
    A.append(value)
    heapSize += 1
    i = heapSize - 1

    while i > 0:
        parent = (i - 1) // 2
        if A[parent] < A[i]:
            A[i], A[parent] = A[parent], A[i]
            i = parent
        else:
            break
    return heapSize

Time and space analysis: In the worst case, the new value moves from a leaf to the root, one level per swap. The running time is \(O(\log n)\). The operation uses \(O(1)\) auxiliary space.

extractMax(A, heapSize)

extractMax removes and returns the root. Removing the root creates a hole at the top, but the heap shape must stay complete. The standard fix is to move the last heap element into the root, decrement heapSize, and call heapify(A, 0, heapSize).

  1. Save A[0] as the maximum value to return.
  2. Move A[heapSize - 1] into A[0].
  3. Decrement heapSize.
  4. Call heapify at the root to repair the heap property.

Implementation in Java, C++, and Python

These snippets return both the extracted value and the updated heap size. A class-based implementation would usually update a heapSize field directly. 

public static class ExtractResult {
    public final int value;
    public final int heapSize;

    public ExtractResult(int value, int heapSize) {
        this.value = value;
        this.heapSize = heapSize;
    }
}

public ExtractResult extractMax(int[] A, int heapSize) {
    if (heapSize == 0) {
        throw new IllegalStateException("empty heap");
    }

    int max = A[0];
    A[0] = A[heapSize - 1];
    heapSize--;
    heapify(A, 0, heapSize);
    return new ExtractResult(max, heapSize);
}
struct ExtractResult {
    int value;
    int heapSize;
};

ExtractResult extractMax(int A[], int heapSize) {
    if (heapSize == 0) {
        throw runtime_error("empty heap");
    }

    int maxValue = A[0];
    A[0] = A[heapSize - 1];
    heapSize--;
    heapify(A, 0, heapSize);
    return {maxValue, heapSize};
}
def extract_max(A, heapSize):
    if heapSize == 0:
        raise IndexError("empty heap")

    max_value = A[0]
    A[0] = A[heapSize - 1]
    heapSize -= 1
    heapify(A, 0, heapSize)
    return max_value, heapSize

Time and space analysis: Moving the last value to the root is constant time. The repair call may travel from the root to a leaf, so the running time is \(O(\log n)\). The operation uses \(O(1)\) auxiliary space if heapify is iterative, or \(O(\log n)\) call-stack space with the recursive version above.

Heapsort Connection

Heapsort combines buildMaxHeap with repeated root removal. After building a max-heap, the algorithm swaps the root with the last active heap element, decreases heapSize, and heapifies the root. The removed maximum is now in its final sorted position at the end of the array.

This use of heapSize is important. During Heapsort, the same physical array holds two regions: an active heap prefix and a sorted suffix. Child-index checks must use heapSize so the sorted suffix is not pulled back into the heap.

See Heapsort for the full sorting page.

Min-Heap Notes

To adapt these max-heap operations to a min-heap, reverse the comparisons:

  • peekMin returns the root.
  • heapify swaps with the smaller child instead of the larger child.
  • insert bubbles upward while the new value is smaller than its parent.
  • extractMin removes the root, moves the last value to the root, and repairs downward.

This is why priority-queue libraries often expose a comparator: the structural algorithm is the same, but the meaning of "higher priority" changes.

Common Pitfalls

  • Using physical length instead of heapSize: Extra capacity or sorted Heapsort elements should not be treated as heap nodes.
  • Forgetting to update size: A standalone insert or extract snippet must return the new size or update a field in a heap object.
  • Choosing the wrong child in heapify: A max-heap must swap with the larger child, not just the left child.
  • Calling heapify where bubble-up is needed: Insertion creates a possible violation between a new node and its parent, so the repair moves upward.
  • Building a heap top-down: Repeated insertion works but costs \(O(n \log n)\). Bottom-up heap construction costs \(O(n)\).
  • Ignoring empty heap cases: peekMax and extractMax need defined behavior when heapSize == 0.

Summary

Heap operations maintain a complete-tree shape and the heap property. peekMax is constant time, insert repairs upward, extractMax repairs downward, heapify fixes a local downward violation, and buildMaxHeap uses bottom-up repair to construct a heap in linear time.

Reading Comprehension Questions

  1. Peek: Why is peekMax constant time?
  2. Heapify Direction: Why does heapify move downward rather than upward?
  3. Build Order: Why does buildMaxHeap start at the last internal node?
  4. Insertion: Why does insertion bubble upward?
  5. Extraction: Why is the last heap element moved to the root after removing the maximum?
  6. Heapsort: Why must Heapsort use heapSize instead of the full array length after the first extraction?
  1. Answer: The max-heap property guarantees that the maximum is at the root, index 0.
  2. Answer: Its precondition says the child subtrees are already heaps; only the value at the current root may need to move downward.
  3. Answer: Leaves are already heaps, so the first possible violations are at internal nodes. Processing bottom-up ensures child subtrees are fixed before their parent.
  4. Answer: The new item is added as a leaf, so the only possible violation is between it and its ancestors.
  5. Answer: Moving the last item preserves the complete-tree shape before the heap property is repaired.
  6. Answer: The end of the array begins holding sorted values that are no longer part of the active heap.

In-Class Activities

  1. Manual Heapify: Start with one local heap-property violation and trace every comparison and swap.
  2. Build a Heap by Hand: Apply bottom-up buildMaxHeap to an unsorted array.
  3. Insert Trace: Insert a value that bubbles all the way to the root, then insert one that does not move.
  4. Extract Trace: Remove the maximum from a heap and trace the downward repair.
  5. Min-Heap Conversion: Rewrite one operation for a min-heap and identify every comparison that changes.
  6. Heapsort Regions: Draw an array during Heapsort and label the active heap prefix and sorted suffix after each extraction.
