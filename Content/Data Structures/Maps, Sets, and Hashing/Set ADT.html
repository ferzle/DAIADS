Set ADT

Overview

A set is an abstract data type that stores a collection of distinct keys. The central question a set answers is simple: is this key present? If a value is already in the set, adding it again does not create a second copy.

Sets are useful whenever a program cares about membership instead of position or frequency. They remove duplicates, track visited vertices in graph algorithms, record banned words, test whether a user has a permission, and represent mathematical collections such as \(A\), \(B\), and \(A \cap B\).

The set is an ADT, not a single implementation. A set can be implemented with a list, sorted array, balanced search tree, hash table, bit vector, or skip list. The ADT describes the visible behavior; the implementation determines the cost of each operation and whether traversal happens in any useful order.

Abstract Data Type and Operations

A set stores keys with no duplicates. A typical set interface includes these operations:

  • add(key): Insert key if it is not already present.
  • remove(key): Delete key if it is present.
  • contains(key): Return whether key is present.
  • size(): Return the number of distinct keys in the set.
  • isEmpty(): Return whether the set contains no keys.
  • clear(): Remove all keys from the set.

Some interfaces return a Boolean from add and remove. For example, add(key) might return true only when the key was newly inserted, and remove(key) might return true only when a key was actually removed. Other interfaces simply perform the operation and leave the caller to use contains if it needs that information.

Conceptual Model

Think of a set as a bag of labels where each label may appear at most once. 

S = { "red", "green", "blue" }

The expression contains("green") returns true, while contains("orange") returns false. Calling add("red") does not make two copies of "red"; the set still contains exactly three keys.

The set ADT does not promise that keys have positions such as index 0 or index 1. Some implementations let you iterate over the keys, but the order depends on the implementation. A hash set may produce an order that seems arbitrary. A tree set may produce sorted order. A linked hash set may preserve insertion order.

Worked Example

Suppose a program scans these course IDs and wants to report the distinct courses mentioned: 

CS1, CS2, MATH, CS2, HIST, CS1

Start with an empty set and add each key in order.

Operation Set Afterward Did the Size Change?
add("CS1") {CS1} yes
add("CS2") {CS1, CS2} yes
add("MATH") {CS1, CS2, MATH} yes
add("CS2") {CS1, CS2, MATH} no
add("HIST") {CS1, CS2, MATH, HIST} yes
add("CS1") {CS1, CS2, MATH, HIST} no

The final set has size 4, even though the input list had six entries. A list keeps all occurrences. A set keeps only distinct keys.

Core Operations

Membership

contains(key) is the defining set operation. The implementation searches for an equal key and returns whether it exists. Sets are often chosen specifically because membership should be faster or cleaner than scanning a list.

Insertion

add(key) must preserve uniqueness. If the key is absent, the operation inserts it and increases the size by one. If an equal key is already present, the visible set does not change. 

add(key):
    if contains(key):
        do not insert a duplicate
    else:
        store key
        size = size + 1

Many real implementations combine the search and insertion work instead of literally calling contains first. The sketch shows the behavior, not necessarily the most efficient code structure.

Deletion

remove(key) deletes the key if it is present. Removing a key that is not present is usually not an error; it simply leaves the set unchanged or returns false.

Iteration

Many sets support iteration over all stored keys. Iteration is useful for printing, copying, or applying a function to every key. It is not the same as indexed access. Unless the set interface promises an order, client code should not rely on the traversal order.

Set Algebra Operations

Sets also support operations from mathematics. These operations create or update sets based on membership in other sets.

Operation Meaning Example with \(A=\{1,2,3\}\), \(B=\{3,4\}\)
Union, \(A \cup B\) Keys in \(A\), in \(B\), or in both. \(\{1,2,3,4\}\)
Intersection, \(A \cap B\) Keys that appear in both sets. \(\{3\}\)
Difference, \(A - B\) Keys in \(A\) that are not in \(B\). \(\{1,2\}\)
Subset test, \(A \subseteq B\) Every key in \(A\) also appears in \(B\). false

These operations can be built from contains, add, and iteration. For example, to compute an intersection, iterate over the smaller set and add a key to the result only if the other set contains that key. 

intersection(A, B):
    result = empty set
    for each key in A:
        if B.contains(key):
            result.add(key)
    return result

Implementation Options

Different implementations provide the same set behavior with different costs and ordering guarantees.

Implementation Membership Add/Remove Ordering Best Fit
Unsorted array or linked list \(O(n)\) \(O(n)\) because duplicates must be checked Can preserve insertion order Small sets and first implementations
Sorted array \(O(\log n)\) \(O(n)\) because elements may shift Sorted Mostly-static sets with many membership tests
Balanced search tree \(O(\log n)\) \(O(\log n)\) Sorted Ordered traversal and range-style queries
Hash table Expected \(O(1)\) Expected \(O(1)\) Usually not sorted Fast exact membership, insertion, and deletion
Bit vector \(O(1)\) \(O(1)\) By numeric key order Dense sets over a small integer universe

Hash tables are a common default for general-purpose sets, but they are not the only choice. If a program needs the keys in sorted order, a tree-based set may be more appropriate. If keys are small nonnegative integers, a bit vector can be extremely compact and fast.

Relationship to Maps

Sets and maps are closely related. A set stores keys only. A map stores keys with associated values. One common way to implement a set is to use a map from each key to a simple marker value. 

set contains key  <==>  markerMap contains key

This relationship is useful, but the ADTs answer different questions. A set asks whether a key is present. A map asks what value is associated with a key. If the program needs counts, labels, records, or distances, use a map. If it only needs membership, use a set.

Edge Cases and Pitfalls

  • Duplicate insertion: Adding an existing key should not increase the size.
  • Removing a missing key: The interface should define whether this returns false, does nothing, or reports an error.
  • Equality rules: The set needs a clear definition of when two keys count as the same key.
  • Mutable keys: Changing a key after insertion can break hash-based or tree-based lookup if the key's hash, equality, or ordering changes.
  • Iteration order: Do not assume sorted or insertion order unless the implementation promises it.
  • Using a set when counts matter: A set loses frequency information. Use a map from key to count when occurrences matter.

Applications

  • Duplicate detection: Insert each item into a set and report a duplicate when insertion finds an existing key.
  • Visited tracking: Graph searches use a set to avoid processing the same vertex repeatedly.
  • Filtering: A set of allowed or blocked keys can decide whether an item should be kept.
  • Membership joins: Given two collections, convert one to a set so membership tests against it are fast.
  • Feature flags and permissions: A user can have a set of permissions, enabled features, or roles.
  • Mathematical modeling: Union, intersection, and difference model overlap between groups.

Summary

A set stores distinct keys and makes membership a primary operation. Adding an existing key leaves the set unchanged, while removing a missing key usually has no effect or returns false. Sets can be implemented in several ways, including lists, sorted arrays, balanced trees, hash tables, and bit vectors. The right implementation depends on whether the program needs fast exact lookup, sorted traversal, compact integer storage, or simple teaching code.

Reading Comprehension Questions

  1. Definition: What makes a set different from a list?
  2. Insertion: What should happen if add(key) is called when key is already present?
  3. Membership: Why is contains the central operation for a set?
  4. Implementation: Why might a hash set be preferred over an unsorted list for a large collection?
  5. Ordering: Why should client code avoid assuming that a general set iterates in sorted order?
  6. Set algebra: How can contains be used to compute an intersection?
  1. Answer: A set stores distinct keys and does not keep duplicate occurrences. A list may contain repeated values and has positions.
  2. Answer: The set should not insert a duplicate. The size should stay the same.
  3. Answer: A set is mainly used to answer whether a key is present, so membership is the defining query.
  4. Answer: A hash set usually supports expected constant-time membership, insertion, and deletion, while an unsorted list requires a linear scan.
  5. Answer: Different implementations use different internal layouts. Only ordered set implementations promise sorted traversal.
  6. Answer: Iterate through one set and add a key to the result only when the other set contains that key.

In-Class Activities

  1. Trace the set: Starting from an empty set, trace add(5), add(2), add(5), remove(7), add(7), and contains(2). Record the set and size after each operation.
  2. Choose the structure: For each task, decide whether a list, set, or map is the best ADT: storing a playlist order, checking whether an ID is banned, counting word frequencies, and tracking visited maze cells.
  3. Set algebra by hand: Given \(A=\{a,b,c,d\}\) and \(B=\{b,d,e\}\), compute \(A \cup B\), \(A \cap B\), \(A-B\), and \(B-A\).
  4. Implementation debate: Compare a hash set and a tree set for a spell-check dictionary. Which operations matter, and does sorted order help?

Homework Problems

  1. First duplicate: Write an algorithm that scans an array and returns the first value that appears twice. Explain how a set helps.
  2. Unique words: Given a text file, report how many distinct words appear. State what should count as the same word when capitalization or punctuation differs.
  3. Intersection algorithm: Write pseudocode for computing the intersection of two sets. Analyze the running time assuming contains is expected \(O(1)\).
  4. Implementation comparison: Compare an unsorted-list set, sorted-array set, hash set, and tree set for membership-heavy workloads.
  5. Mutable-key failure: Explain why changing an object after it has been inserted into a hash set can make later membership tests fail.
