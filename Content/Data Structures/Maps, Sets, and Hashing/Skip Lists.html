Skip Lists

Overview

A skip list is an ordered data structure for implementing maps and sets. It keeps keys in sorted order like a binary search tree, but it uses layers of linked lists instead of tree rotations.

The bottom layer contains all keys in sorted order. Higher layers contain selected keys that act like express lanes. Searching moves right while the next key is not too large, then drops down a level when moving right would overshoot the target.

Skip lists are randomized. When a key is inserted, coin flips or random choices decide how many levels that key appears in. With a good promotion rule, search, insertion, and deletion take expected \(O(\log n)\) time, while the structure uses expected \(O(n)\) space.

Ordered Dictionary Problem

A hash table is excellent for exact-key lookup, but it usually does not keep keys in sorted order. A skip list solves the ordered dictionary problem: store keys so that lookup, insertion, deletion, and sorted traversal are all efficient.

A skip list can support set operations such as contains(key), add(key), and remove(key). It can also support map operations such as get(key) and put(key, value) by storing a value with each bottom-level key.

The structure is especially useful as a comparison point: it gives tree-like expected logarithmic performance without maintaining heights, colors, or rotations.

Layered Linked-List Model

A skip list can be drawn as several linked lists stacked on top of each other. The bottom list contains every key. Each higher list contains a subset of the keys from the level below it. 

Level 3:  -inf ----------------------------- +inf
Level 2:  -inf -------- 17 ----------------- +inf
Level 1:  -inf ---- 9 -- 17 ------ 31 ------ +inf
Level 0:  -inf -- 4 -- 9 -- 17 -- 22 -- 31 -- 40 -- +inf

The -inf and +inf entries are sentinels. They are not real keys from the set; they simplify the edge cases at the beginning and end of each level.

A key that appears in multiple levels forms a vertical tower. In the diagram, key 17 appears on levels 0, 1, and 2. Key 4 appears only at the bottom level.

Search

Search starts at the top-left sentinel. At each level, move right while the next key is less than or equal to the target. If moving right would pass the target, move down one level. When the search reaches the bottom level, the next key determines whether the target is present. 

search(key):
    current = top-left sentinel
    while current has a lower level:
        while current.next.key <= key:
            current = current.next
        current = current.down
    while current.next.key <= key:
        current = current.next
    return current.key == key

Some implementations use nodes with arrays of forward pointers instead of explicit down pointers. The movement idea is the same: scan right at the current level, then continue at the next lower level.

Worked Example

Search for key 22 in this skip list: 

Level 2:  -inf -------- 17 ----------------- +inf
Level 1:  -inf ---- 9 -- 17 ------ 31 ------ +inf
Level 0:  -inf -- 4 -- 9 -- 17 -- 22 -- 31 -- 40 -- +inf
  1. Start at -inf on level 2.
  2. Move right to 17, because 17 <= 22.
  3. The next key is +inf, so drop to level 1.
  4. The next key is 31, which is too large, so drop to level 0.
  5. Move right to 22. The target is found.

The search skips over keys 4 and 9 at the bottom level because the higher-level link reached 17 first. This is the purpose of the express lanes.

Insertion

Insertion has two phases. First, search for the position where the key belongs and remember the last node visited at each level. Second, create the new key's tower and splice it into the relevant levels. 

insert(key, value):
    update = array of predecessors, one per level
    current = top-left sentinel

    for level from top down to 0:
        while current.next.key < key:
            current = current.next
        update[level] = current
        if level > 0:
            current = current.down

    if update[0].next.key == key:
        update the existing value
        return

    height = randomHeight()
    for level from 0 to height:
        splice a new node after update[level]

The random height is usually chosen by repeated coin flips. Start with height 0. Each time the coin comes up heads, promote the key one more level. With promotion probability \(p=1/2\), about half the keys appear on level 1, about one quarter appear on level 2, and so on.

If a new key's tower is taller than the current list, the implementation creates new top-level sentinels before linking the new nodes.

Deletion

To delete a key, search for its predecessors on every level. Whenever the next node at that level contains the key, unlink it. 

delete(key):
    current = top-left sentinel
    found = false

    for level from top down to 0:
        while current.next.key < key:
            current = current.next
        if current.next.key == key:
            current.next = current.next.next
            found = true
        if level > 0:
            current = current.down

    remove empty top levels if needed
    return found

Deletion does not need rotations or rebalancing. It only removes the nodes in the deleted key's tower. Empty top levels can be removed so future searches do not start above any useful links.

Why Search Is Fast

Skip lists are fast because each higher level is expected to contain a constant fraction of the level below it. With promotion probability \(p=1/2\), the expected number of keys on each level roughly halves:

Level Expected Number of Keys
0 \(n\)
1 \(n/2\)
2 \(n/4\)
3 \(n/8\)

After about \(\log_2 n\) levels, the expected number of keys on the top level is constant. A search does a small expected amount of horizontal movement per level and then moves down. This gives expected \(O(\log n)\) search time.

The guarantee is expected rather than absolute. Random choices can produce an unlucky shape, but the probability of a badly unbalanced skip list is small when the random choices are independent.

Time and Space Complexity

Operation Expected Cost Worst Case Reason
Search \(O(\log n)\) \(O(n)\) Expected logarithmic height, but an unlucky structure can have poor shortcuts.
Insert \(O(\log n)\) \(O(n)\) Search for predecessors, then splice a random-height tower.
Delete \(O(\log n)\) \(O(n)\) Search for predecessors and unlink the tower.
Sorted traversal \(O(n)\) \(O(n)\) Walk the bottom level from left to right.
Space \(O(n)\) \(O(nL)\) with maximum level \(L\) With geometric promotion, the expected total number of node levels is linear.

Real implementations usually impose a maximum level, often based on the expected maximum number of entries, to avoid unbounded tower heights.

Implementation Notes

A skip-list node usually stores a key, an optional value, and forward references. Two common representations are:

  • Stacked nodes: Each level has a node, with right and down pointers.
  • Forward-pointer array: One node stores an array of next pointers, one for each level in its tower.

The forward-pointer array representation is common because one logical entry has one object containing all of its level links. 

Node:
    key
    value
    forward[0..height]

Duplicate-key behavior depends on whether the skip list is implementing a set or a map. A set should ignore or reject a duplicate key. A map should update the existing key's value.

Comparison with Other Structures

Structure Lookup Ordering Main Tradeoff
Hash table Expected \(O(1)\) Usually not sorted Fast exact lookup, but poor fit for sorted traversal and range queries.
Balanced search tree Worst-case \(O(\log n)\) Sorted Strong worst-case bounds, but balancing rules and rotations add implementation complexity.
Skip list Expected \(O(\log n)\) Sorted Simpler randomized balancing, but expected rather than worst-case guarantees.

If the program only needs exact lookup, a hash table is often the fastest default. If the program needs keys in order, a skip list or balanced tree is a better match.

Common Pitfalls

  • Forgetting the bottom level: Every key must appear at level 0.
  • Using inconsistent comparisons: The ordering rule must treat keys consistently across all levels.
  • Not remembering predecessors: Insertion and deletion need the predecessor at each level that will be changed.
  • Promoting deterministically by accident: A predictable bad promotion rule can destroy the expected balance.
  • Ignoring maximum level: Practical implementations usually cap the height.
  • Expecting hash-table behavior: Skip lists keep sorted order, but they do not give expected constant-time exact lookup.

Summary

A skip list is a randomized ordered structure built from layers of linked lists. Search moves right and down through express lanes until it reaches the target position. Insertion chooses a random tower height and splices the new key into each selected level. Deletion unlinks the key's tower. With independent random promotion, skip lists support expected \(O(\log n)\) search, insertion, and deletion with expected \(O(n)\) space.

In-Class Activities

  1. Trace search: Given a three-level skip list, trace the right and down moves used to search for three target keys: one present, one missing between two keys, and one larger than every key.
  2. Build towers: Flip coins for the keys 5, 8, 12, 19, and 25. Draw the resulting towers and bottom-level order.
  3. Insertion path: For a given skip list, mark the predecessor nodes that would be stored in the update array before inserting key 18.
  4. Structure choice: Compare a hash table, balanced BST, and skip list for a word dictionary that must support exact lookup and alphabetical printing.
