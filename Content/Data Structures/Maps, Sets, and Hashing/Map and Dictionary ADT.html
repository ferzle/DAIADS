Map and Dictionary ADT

Overview

A map is an abstract data type that stores associations between keys and values. Given a key, the map finds the value associated with that key. This is why maps are often called dictionaries or associative arrays: they let a program look up information by name, identifier, word, or other key.

The key idea is that keys are unique. A map should not contain two separate entries with the same key. Values do not have to be unique. Many different keys may map to the same value.

This page treats map, dictionary, and associative array as closely related names for the same keyed-lookup ADT. Different textbooks and programming languages choose different names, but the central behavior is the same: store a key-value association, then retrieve or update it by key.

Abstract Data Type and Operations

A map stores entries. Each entry has a key and a value. 

key -> value

A typical map interface includes these operations:

  • put(key, value): Store an association from key to value.
  • get(key): Return the value associated with key.
  • containsKey(key): Return whether key is present.
  • remove(key): Delete the entry with key key, if present.
  • size(): Return the number of key-value entries.
  • isEmpty(): Return whether the map contains no entries.
  • keys(): Iterate over the keys.
  • values(): Iterate over the values.
  • entries(): Iterate over key-value pairs.

Some interfaces use names such as set, insert, lookup, or delete instead of put, get, and remove. The names vary, but the ADT contract should still make the behavior clear.

Conceptual Model

A map can be pictured as a two-column table where the left column contains keys and the right column contains the associated values.

Key Value
"A100" "Mina"
"A101" "Cal"
"A102" "Mina"

The keys "A100", "A101", and "A102" are unique. The value "Mina" appears twice, and that is allowed because values are not used to identify entries.

Calling get("A101") returns "Cal". Calling containsKey("A103") returns false. The map searches by key, not by value.

Worked Example

Suppose a program counts word frequencies in this input: 

to be or not to be

The map stores each distinct word as a key and the number of times that word has appeared as the value.

Word Read Operation Map Afterward
to put("to", 1) {to: 1}
be put("be", 1) {to: 1, be: 1}
or put("or", 1) {to: 1, be: 1, or: 1}
not put("not", 1) {to: 1, be: 1, or: 1, not: 1}
to put("to", get("to") + 1) {to: 2, be: 1, or: 1, not: 1}
be put("be", get("be") + 1) {to: 2, be: 2, or: 1, not: 1}

Notice that the second occurrence of to does not create another to entry. It updates the value associated with the existing key.

Core Operations

Lookup

get(key) searches for the entry with the given key and returns its value. This is the defining map operation. A map is useful because the caller does not need to scan every entry manually to find the matching key.

A map interface must define what happens when the key is missing. Common choices include returning a special value such as null, throwing an exception, returning an optional result, or requiring the caller to check containsKey first.

Insertion and Update

put(key, value) has two cases. If the key is absent, the operation inserts a new entry and increases the size by one. If the key is already present, the operation replaces the old value and the size does not change. 

put(key, value):
    if key is already present:
        replace the old value with value
    else:
        store a new (key, value) entry
        size = size + 1

Some map APIs separate these cases with operations such as putIfAbsent, replace, or insert that fails when the key already exists. The important point is that the interface must make the update rule explicit.

Deletion

remove(key) deletes the entry with the given key if it exists. A useful version returns the removed value or a Boolean that says whether anything was removed. Removing a missing key usually leaves the map unchanged.

Iteration

A map may let clients iterate over keys, values, or entries. Iterating over entries is useful when both parts are needed, such as printing a frequency table. Iterating over values alone can be useful, but remember that values are not required to be unique.

Missing Keys and Default Values

Missing keys are one of the most common sources of map bugs. In the word-frequency example, the program cannot simply write put(word, get(word) + 1) for the first occurrence of a word unless get has a defined default.

A common pattern is to check for membership first: 

if counts.containsKey(word):
    counts.put(word, counts.get(word) + 1)
else:
    counts.put(word, 1)

Many languages provide helper operations for this pattern, such as "get with default," "compute if absent," or dictionary indexing that creates a default value. Those helpers are conveniences built on top of the same ADT idea: missing keys need a deliberate policy.

Implementation Options

Maps can be implemented many ways. The right choice depends on operation costs, ordering needs, and the types of keys being stored.

Implementation Typical Lookup Typical Put/Remove Ordering Best Fit
Unsorted list of entries \(O(n)\) \(O(n)\), because the key must be searched Can preserve insertion order Small maps and simple first implementations
Sorted array of entries \(O(\log n)\) \(O(n)\), because entries may shift Sorted by key Mostly-static dictionaries with many lookups
Balanced search tree \(O(\log n)\) \(O(\log n)\) Sorted by key Ordered maps, range queries, predecessor/successor operations
Hash table Expected \(O(1)\) Expected \(O(1)\) Usually not sorted Fast exact lookup, insertion, and deletion by key
Skip list Expected \(O(\log n)\) Expected \(O(\log n)\) Sorted by key Ordered dictionaries with randomized balancing

Hash tables are a common default when the program needs exact lookup by key and does not need sorted order. Tree-based maps and skip lists are better matches when order matters, such as printing keys alphabetically or finding all keys in a range.

Relationship to Sets

A set stores keys only. A map stores keys and values. Because every map entry has a unique key, the keys of a map form a set. 

map entries:  {A: 4, B: 9, C: 4}
key set:      {A, B, C}

This relationship explains why maps and sets often have similar implementations. A hash set and a hash map both need hashing, collision handling, resizing, and key equality. The map stores an extra value with each key.

Key Rules

A map implementation needs a reliable way to decide whether two keys are the same. Hash-based maps use equality and hash codes. Tree-based maps use a comparison order. Either way, the key rule must stay consistent while the key is stored in the map.

  • Keys must be comparable or hashable: The implementation needs a lookup rule.
  • Keys should be stable: Mutating a key after insertion can make future lookup fail.
  • Keys identify entries: Two equal keys should not create two entries.
  • Values may repeat: Duplicate values are allowed because values are not used as identifiers.

For example, using a student's ID number as a key is usually safer than using a student object whose fields may change. If the object changes in a way that affects equality, hashing, or ordering, the map may no longer be able to find the entry correctly.

Common Uses

  • Frequency tables: Map each item to a count.
  • Symbol tables: Map variable names to type, scope, or memory-location information.
  • Caches: Map a request or input to a previously computed result.
  • Indexes: Map an ID, word, or key field to records where it appears.
  • Graph algorithms: Map vertices to distances, parents, colors, discovery times, or adjacency lists.
  • Sparse arrays: Map meaningful indices to values and omit default entries.
  • Configuration data: Map option names to option values.

Common Pitfalls

  • Confusing keys and values: A map searches by key. Values are the data retrieved after the key is found.
  • Expecting duplicate keys: Inserting an existing key updates or rejects the old entry depending on the API; it does not create a second equal key.
  • Ignoring missing keys: Calling get on a missing key must be handled deliberately.
  • Assuming sorted order: Many maps, especially hash maps, do not iterate in sorted key order.
  • Mutating keys: A key that changes after insertion can become impossible to find.
  • Using a map when a set is enough: If no value is needed, a set usually communicates the intent more clearly.
  • Using a set when a map is needed: If counts, labels, distances, or records matter, the value part is essential.

Summary

A map stores key-value associations. Keys are unique and identify entries; values are the data associated with those keys and may repeat. The main operations are lookup, insertion or update, deletion, and iteration over keys, values, or entries. Lists, sorted arrays, balanced trees, hash tables, and skip lists can all implement the same map ADT with different cost and ordering tradeoffs.

Reading Comprehension Questions

  1. Definition: What does a map store?
  2. Keys and values: Which part of a map entry must be unique?
  3. Update: What should happen when put(key, value) is called for a key that is already present?
  4. Missing keys: Why is calling get on a missing key a boundary case?
  5. Implementation: Why might a hash map be preferred for exact lookup, but a tree map be preferred for sorted traversal?
  6. Set relationship: How are the keys of a map related to a set?
  1. Answer: A map stores key-value associations.
  2. Answer: The keys must be unique. Values may repeat.
  3. Answer: The existing value should be replaced, or the operation should follow another explicitly documented update policy.
  4. Answer: The map has no associated value to return, so the interface must define whether it returns a special value, throws an exception, returns an optional result, or requires a membership check first.
  5. Answer: A hash map usually gives expected constant-time exact lookup, while a tree map keeps keys ordered for sorted iteration and range-style operations.
  6. Answer: The keys of a map form a set because each key appears at most once.

In-Class Activities

  1. Trace the map: Starting from an empty map, trace put(A, 3), put(B, 5), put(A, 7), remove(C), put(C, 5), and get(A). Record the entries and size after each operation.
  2. Classify the ADT: Decide whether each task needs a set, map, list, or priority queue: count word frequencies, remove duplicates, store students in seating order, and repeatedly remove the most urgent task.
  3. Missing-key policy: Design three possible behaviors for get when a key is absent. Discuss which one is easiest for a beginning programmer to use correctly.
  4. Implementation choice: Choose between a hash map and tree map for a phone book, a compiler symbol table, a leaderboard by username, and printing all usernames alphabetically.

Homework Problems

  1. Frequency table: Write pseudocode that uses a map to count how many times each word appears in a list of words.
  2. Index construction: Given a list of records with student ID and name, build a map from ID to name. State what your algorithm does if the same ID appears twice.
  3. Invert a map: Given a map from student ID to major, build a map from major to the set of student IDs in that major.
  4. Compare implementations: Compare an unsorted-list map, sorted-array map, hash map, and tree map for lookup-heavy workloads.
  5. Mutable-key failure: Explain why changing a key object after insertion can make lookup fail in a hash map or tree map.
  6. Design tests: List at least six tests for a map implementation, including duplicate-key updates and missing-key behavior.
