Separate Chaining

Overview

Separate chaining is a collision-resolution strategy for hash tables. Each table position is a bucket, and each bucket stores all entries whose keys hash to that position. The bucket is often shown as a linked list, but it can also be a dynamic array, small vector, or another local collection.

Chaining is conceptually direct: if several keys choose the same table index, keep them together in that bucket. Searching for a key means hashing to the bucket and then searching only that bucket.

Chaining can implement both sets and maps. A hash set stores keys in the buckets. A hash map stores key-value entries in the buckets.

Bucket Model

A chained hash table has an array of \(m\) bucket references. The table stores \(n\) entries total, spread across those buckets. 

table[0] -> bucket of entries whose hash index is 0
table[1] -> bucket of entries whose hash index is 1
table[2] -> bucket of entries whose hash index is 2
...
table[m-1] -> bucket of entries whose hash index is m-1

If \(h(k)=k \bmod m\), then key 17 belongs in bucket 17 mod m. If another key lands in the same bucket, it joins the same bucket collection instead of searching for another array position.

Worked Example

Use a table of size \(m=5\) and hash function \(h(k)=k \bmod 5\). Insert keys 12, 22, 7, 4, and 17.

Key Bucket Table State After Insertion
12 2 2: [12]
22 2 2: [12, 22]
7 2 2: [12, 22, 7]
4 4 2: [12, 22, 7], 4: [4]
17 2 2: [12, 22, 7, 17], 4: [4]

A search for 17 hashes to bucket 2, then searches that bucket. It does not inspect buckets 0, 1, 3, or 4.

Core Operations

Lookup

Lookup hashes the key to a bucket, then scans that bucket for an equal key. In a map, lookup returns the value associated with the matching key. In a set, lookup reports whether the key is present. 

lookup(key):
    i = hash(key) mod table.length
    for each entry in table[i]:
        if entry.key equals key:
            return entry
    report missing key

Insert or Put

In a map, insertion must first check whether the key is already in the bucket. If it is, the map updates the existing value. If not, it adds a new entry to the bucket. In a set, adding an existing key leaves the set unchanged. 

put(key, value):
    i = hash(key) mod table.length
    for each entry in table[i]:
        if entry.key equals key:
            entry.value = value
            return
    add (key, value) to table[i]
    size = size + 1

If the bucket is a linked list, the new entry is often inserted at the head because that takes constant time. Inserting at the tail is also possible if the bucket maintains a tail reference.

Remove

Removal hashes to the bucket, searches for the matching key, and unlinks or removes that entry from the bucket. Unlike open addressing, chaining does not need tombstones because deleting from one bucket does not break the search path for another bucket. 

remove(key):
    i = hash(key) mod table.length
    search table[i] for key
    if found:
        remove that entry from the bucket
        size = size - 1

Load Factor

For separate chaining, the load factor is still \(\alpha=n/m\), where \(n\) is the number of entries and \(m\) is the number of buckets. Unlike open addressing, \(\alpha\) can be greater than 1, because each bucket can hold more than one entry.

If the hash function distributes entries uniformly, the average bucket length is about \(\alpha\). That is why lookup and deletion are often described as expected \(O(1+\alpha)\): constant time to find the bucket, plus time proportional to the bucket length.

Implementations usually resize when the load factor gets too high. Resizing creates more buckets and rehashes existing entries, which lowers the average bucket length.

Time and Space Complexity

Operation Expected Cost Worst Case Reason
Lookup \(O(1+\alpha)\) \(O(n)\) Search one bucket; worst case all keys are in one bucket.
Put \(O(1+\alpha)\) \(O(n)\) Must check whether the key already exists before adding or updating.
Remove \(O(1+\alpha)\) \(O(n)\) Must find the key in its bucket before removing it.
Iterate all entries \(O(n+m)\) \(O(n+m)\) Scan all buckets and all stored entries.

If resizing keeps \(\alpha\) bounded by a constant, the expected costs for lookup, put, and remove are expected \(O(1)\). The space usage is \(O(n+m)\): storage for the bucket array plus storage for the entries.

Bucket Choices

The classic teaching version uses linked lists for buckets, but the bucket representation is a design choice.

  • Linked list: Simple insertion and deletion, but pointer overhead and poor cache locality.
  • Dynamic array or small vector: Better cache locality for short buckets, but deletion may require shifting or swapping.
  • Tree bucket: Can protect against long chains, but adds complexity and overhead for small buckets.

For a CS2 implementation, linked lists make the collision behavior easiest to see. Production libraries may use more specialized bucket strategies when chains become long.

Advantages and Tradeoffs

  • Deletion is straightforward: Removing an entry from a bucket does not disturb other buckets.
  • Load factor can exceed one: The table can still function even when there are more entries than buckets.
  • Bucket overhead exists: Each entry may need node or reference storage beyond the key and value.
  • Cache locality can be weaker: Linked nodes may be scattered in memory.
  • Long chains are possible: A poor hash function or adversarial input can make one bucket very long.

Common Pitfalls

  • Adding duplicates: A map or set implementation must search the bucket before inserting a duplicate key.
  • Forgetting to update size: Size changes only when a new key is inserted or an existing key is removed.
  • Assuming each bucket has at most one entry: Collisions are exactly why buckets can hold multiple entries.
  • Ignoring load factor: Chains grow longer when the table has too few buckets for the number of entries.
  • Rehashing incorrectly: Existing entries must be placed according to the new table size, not copied to the same indices.

Summary

Separate chaining resolves collisions by storing all entries with the same hash index in the same bucket. The implementation hashes to a bucket, then searches within that bucket. When hashing distributes keys well and resizing keeps the load factor bounded, lookup, insertion, and deletion are expected constant time. Chaining is easy to understand and handles deletion cleanly, but it uses extra bucket storage and can degrade if chains become long.

In-Class Activities

  1. Build chains: Insert 6, 11, 16, 2, and 7 into a chained table of size 5 using \(h(k)=k \bmod 5\). Draw the buckets after each insertion.
  2. Trace lookup: Using your table from the first activity, trace searches for 16, 7, and 12.
  3. Delete carefully: Remove one key from the middle of a chain and show the bucket before and after deletion.
  4. Resize discussion: Rehash the same entries into a table of size 11. Which chains get shorter?

Homework Problems

  1. Pseudocode: Write contains, add, and remove for a chained hash set.
  2. Map update: Write put for a chained hash map and make sure updating an existing key does not increase the size.
  3. Analysis: If a table stores 120 entries in 40 buckets, what is the load factor? What is the expected bucket length under uniform hashing?
  4. Bad hashing: Describe a hash function that would put every integer key into the same bucket. What would happen to lookup cost?
  5. Bucket choice: Compare linked-list buckets and dynamic-array buckets for memory overhead, deletion, and cache locality.
