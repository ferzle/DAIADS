Hash Tables

Overview

A hash table is a data structure for implementing maps and sets. Its goal is fast exact lookup: given a key, find the stored key or key-value entry without scanning the whole collection.

The central idea is to use a hash function to turn a key into an array index. If the hash function spreads keys out well and the table is kept at a reasonable load factor, lookup, insertion, and deletion are expected \(O(1)\).

Hash tables are implementation structures. The same table machinery can store a set of keys or a map of key-value entries. The ADT describes what the client sees; the hash table describes how the implementation organizes the entries internally.

The Table Model

A hash table uses an array of \(m\) positions. The table currently stores \(n\) entries. The load factor is 

alpha = n / m

The load factor measures how full the table is. A higher load factor means the table uses space more tightly, but it also usually means more collisions and slower operations.

To place a key in the table, the implementation computes a hash code and converts it into a table index: 

index = hash(key) mod tableSize

The actual hash function may be more complicated than this sketch, especially for strings or objects. In introductory examples, integer keys and modulo arithmetic make the table behavior easier to trace.

Collisions

A collision occurs when two different keys are assigned to the same table position. Collisions are unavoidable because there are usually many more possible keys than array positions.

For example, if \(h(k)=k \bmod 7\), then keys 10, 17, and 24 all map to index 3. A hash table is not complete until it says what to do when this happens.

The two collision-resolution families covered in this section are:

  • Separate Chaining: Each array position stores a small collection, often a linked list, of entries that hash to that position.
  • Open Addressing: Entries are stored directly in the table array, and collisions are handled by probing other slots.

Worked Example

Suppose a table has \(m=7\) positions and uses \(h(k)=k \bmod 7\). Insert keys 10, 3, 17, 8, and 24.

Key Hash Index Observation
10 3 Bucket or slot 3 is selected.
3 3 Collision with 10.
17 3 Another collision at index 3.
8 1 No collision if index 1 is empty.
24 3 Collision again at index 3.

This example has many collisions because the selected keys share the same remainder modulo 7. A real hash table tries to avoid patterns like this by using a better hash function and resizing when the load factor becomes too high.

Core Operations

Search or Lookup

To search for a key, the table hashes the key to find the first place to look. Then it follows the table's collision strategy until it either finds the key or determines that the key is absent. 

lookup(key):
    index = tableIndex(key)
    search according to the collision strategy
    return the matching entry or report that key is absent

Insert or Put

In a set, insertion stores a key. In a map, insertion stores a key-value entry. If the key already exists, a map usually updates the existing value and a set leaves the key unchanged.

Remove

Removal first finds the key, then deletes the entry according to the collision strategy. Deletion is especially delicate in open addressing because simply clearing a slot can break later searches. That is why open-addressed tables often use tombstones.

Hash Functions

A good hash function should be deterministic, fast to compute, and spread likely keys across the table. It does not need to be random, but it should avoid predictable clustering for the data being stored.

  • Deterministic: The same key must hash to the same value while it is stored.
  • Uses the key's meaningful data: Equal keys must be treated consistently.
  • Spreads common inputs: Similar keys should not all land in the same small part of the table.
  • Fast: Hashing should not dominate the cost of lookup.

For object keys, the hash function and equality rule must agree: if two keys are equal, they must have the same hash code. The reverse is not required. Different keys may have the same hash code, which is exactly why collision handling is necessary.

Load Factor and Resizing

Hash tables resize to keep operations fast. When the load factor crosses a chosen threshold, the implementation allocates a larger table and inserts all existing entries into the new table. This process is called rehashing. 

resize():
    oldTable = table
    table = new larger empty table
    size = 0
    for each entry in oldTable:
        insert entry into the new table

Rehashing is expensive when it happens because every entry is moved. If the table grows geometrically, such as doubling in size, the cost is spread across many insertions and the average insertion cost can still be treated as expected \(O(1)\).

Time and Space Complexity

Operation Expected Cost Worst Case Why
Lookup \(O(1)\) \(O(n)\) Expected constant if entries are well distributed; linear if many keys collide.
Insert or put \(O(1)\) amortized expected \(O(n)\) Usually constant, but may scan a long collision group or trigger a resize.
Remove \(O(1)\) \(O(n)\) Deletion first has to find the key.
Iterate all entries \(O(n + m)\) \(O(n + m)\) The implementation may need to scan table positions as well as entries.

The space usage is usually \(O(n+m)\), where \(n\) is the number of entries and \(m\) is the table capacity. Some collision strategies add extra pointer or marker overhead.

Common Pitfalls

  • Assuming guaranteed constant time: Hash tables usually provide expected constant time, not worst-case constant time.
  • Ignoring collisions: A hash function alone is not a hash table implementation.
  • Letting the table get too full: High load factor makes collisions more expensive.
  • Mutating keys: If a key's hash or equality result changes after insertion, lookup may fail.
  • Assuming sorted order: Most hash tables do not keep keys in sorted order.
  • Using hashing for range questions: Hash tables are designed for exact-key lookup, not predecessor, successor, or range queries.

Applications

  • Hash sets: Fast membership testing and duplicate removal.
  • Hash maps: Fast key-value lookup, including frequency tables and indexes.
  • Caches: Store previously computed results by input or request key.
  • Graph algorithms: Store visited vertices, distances, parents, and adjacency lists keyed by vertex.
  • Compilers: Symbol tables map identifiers to declaration information.
  • Sparse data: Store only meaningful positions in a large logical array or matrix.

Summary

A hash table implements maps and sets by using a hash function to choose where entries belong in an array. Collisions are unavoidable, so every hash table needs a collision-resolution strategy. Separate chaining stores colliding entries in bucket collections. Open addressing keeps entries in the array and searches through a probe sequence. Load factor and resizing are what keep expected operation costs close to constant.

In-Class Activities

  1. Compute hash indices: For \(m=11\) and \(h(k)=k \bmod 11\), compute the table index for 21, 32, 43, 17, and 28.
  2. Find collisions: Choose five integer keys that all collide when \(m=5\). Explain why they collide.
  3. Load-factor discussion: Compare two tables storing \(n=20\) entries, one with \(m=25\) and one with \(m=100\). What tradeoff is being made?
  4. Collision strategy preview: For a colliding key, describe where it would go under separate chaining and where it might go under open addressing.
