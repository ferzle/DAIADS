Maps, Sets, and Hashing

Overview

Maps, sets, and hash tables belong together because they all center on one question: How can a program quickly find whether a key is present? A set stores keys. A map stores keys together with associated values. Hashing is one of the most important implementation techniques for making lookup, insertion, and deletion fast.

These structures appear everywhere in programming. They count frequencies, remove duplicates, cache previous results, group records by identifier, test membership, store symbol tables, represent sparse data, and support many graph and string algorithms. Students often first meet them as library types such as Java HashMap and HashSet, C++ unordered_map and unordered_set, or Python dict and set.

This section treats the ADTs first and the implementations second. That separation matters: a map or set can be implemented with a hash table, a balanced search tree, a sorted array, a linked list, or a skip list. The ADT describes the behavior; the implementation determines the costs and ordering guarantees.

Why These Topics Are Together

Sets and maps are closely related ADTs. A set answers membership questions: contains(key). A map answers lookup questions: get(key). In fact, a set can often be implemented as a map from each key to a simple marker value.

Hashing is grouped with them because hash tables are the standard implementation when the goal is fast average lookup by key. A hash function maps a key to an array index. If the hash function spreads keys well and the table is not too full, common operations are expected \(O(1)\).

The same section also includes skip lists because they solve a related dictionary problem with different tradeoffs. A skip list keeps keys in sorted order and supports expected \(O(\log n)\) search, insert, and delete. It is not a hashing method, but it is a useful comparison point for ordered maps and sets.

Core Ideas

Keys

A key is the value used for lookup. In a student-record map, the key might be a student ID. In a word-frequency table, the key is a word. In a visited set for graph search, the key is a vertex.

Membership

A membership query asks whether a key is present. Sets make membership the central operation. Maps also use membership internally, because get(key) must find whether the key exists before it can return the associated value.

Key-Value Association

A map stores an association between a key and a value. The key identifies the entry; the value is the data retrieved or updated. A map should not contain two separate entries with the same key. Inserting an existing key usually updates its value or reports that the key is already present, depending on the chosen interface.

Collisions

A collision occurs when two different keys map to the same hash-table location. Collisions are unavoidable when many possible keys are mapped into a finite array. Hash-table implementations differ mostly in how they resolve collisions and when they resize.

Worked Example

Suppose a program scans the words: 

to be or not to be

A set can record which distinct words have appeared: 

{to, be, or, not}

A map can record how many times each word appears:

Key Value
to 2
be 2
or 1
not 1

A hash table can implement either structure. For each word, the hash function chooses a bucket. A set stores the word in that bucket. A map stores the pair (word, count). If another word hashes to the same bucket, the collision-resolution strategy decides where that entry goes.

Implementation Options

Implementation Typical Search/Insert/Delete Ordering Best Fit
Unsorted list \(O(n)\) Insertion order can be preserved Very small collections or simple teaching examples
Sorted array \(O(\log n)\) search, \(O(n)\) insert/delete Sorted Mostly-static data with many searches
Balanced search tree \(O(\log n)\) Sorted Ordered maps/sets and range queries
Hash table Expected \(O(1)\) Usually not sorted Fast lookup, insertion, and deletion by key
Skip list Expected \(O(\log n)\) Sorted Ordered dictionaries with randomized balancing

No implementation dominates every workload. Hash tables are often the fastest default for exact-key lookup, but sorted structures are better when the program needs keys in order, range queries, predecessor/successor operations, or predictable worst-case bounds.

Hashing Preview

A hash table uses an array of buckets. A hash function turns a key into an integer, and the table converts that integer into a bucket index, often with modulo arithmetic: 

index = hash(key) mod tableSize

The table should keep its load factor under control. The load factor is usually \(\alpha = n/m\), where \(n\) is the number of stored entries and \(m\) is the number of buckets or slots. As the table gets too full, collisions become more frequent and operations slow down. Resizing creates a larger table and re-inserts the existing entries.

The two main collision-resolution families in this section are separate chaining and open addressing. Separate chaining stores a small collection at each bucket. Open addressing stores entries directly in the array and follows a probe sequence when a collision occurs.

Applications

  • Counting frequencies: A map from item to count records how many times each key appears.
  • Removing duplicates: A set records which keys have already been seen.
  • Caching and memoization: A map stores previous results so repeated requests can be answered quickly.
  • Symbol tables: Compilers and interpreters map variable names to information about declarations and types.
  • Graph algorithms: Sets track visited vertices, and maps store distances, parents, colors, or adjacency information.
  • Indexes: Databases and search tools map keys or terms to records where they occur.
  • Sparse data: A map can store only the nonzero or meaningful entries in a large logical array or matrix.

Common Pitfalls

  • Confusing keys and values: Maps search by key. Values are the data associated with keys and are not necessarily unique.
  • Assuming hash tables are sorted: Most hash tables do not keep keys in sorted order.
  • Ignoring collisions: A hash function alone is not a complete hash table. The implementation also needs a collision strategy.
  • Letting the table get too full: High load factor increases collision cost and can make operations much slower.
  • Mutating keys after insertion: If a key's hash or equality behavior changes while it is in a table, later lookups may fail.
  • Expecting worst-case constant time: Hash-table operations are usually expected \(O(1)\), not guaranteed \(O(1)\) under every possible input.
  • Choosing hashing for ordered questions: If the program needs the next largest key or all keys in a range, an ordered structure is usually a better fit.

Summary

Sets store keys; maps store key-value associations. Hash tables are a major implementation technique for both because they provide fast expected lookup, insertion, and deletion. The rest of this section separates the ADTs from their implementations, then studies how hashing handles collisions, resizing, and tradeoffs with ordered alternatives such as skip lists and balanced trees.

In-Class Activities

  1. Classify the ADT: For several programming tasks, decide whether a set, map, or neither is the best match.
  2. Frequency Table: Build a word-count map by hand for a short sentence.
  3. Collision Trace: Given a table size and hash values, identify which keys collide.
  4. Implementation Choice: Choose between a hash table, sorted array, balanced tree, and skip list for several workloads.
  5. Mutable Key Discussion: Explain what can go wrong if a key object changes after insertion into a hash table.

Homework Problems

  1. Duplicate Detection: Use a set to report whether an array contains any duplicate value.
  2. Word Frequencies: Use a map to count words in a text file and return the most common word.
  3. Two Sum: Use a set or map to determine whether any two array values add to a target.
  4. Implementation Comparison: Compare hash tables and balanced search trees for exact lookup, sorted traversal, and range queries.
  5. Collision Policies: Preview separate chaining and open addressing by describing where colliding keys are stored in each approach.
  6. Load Factor Experiment: Simulate inserting keys into tables of different sizes and record how often collisions occur.
