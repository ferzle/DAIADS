Open Addressing

Overview

Open addressing is a collision-resolution strategy for hash tables where every entry is stored directly inside the table array. There are no separate bucket lists. If a key's first-choice slot is already occupied, the table follows a probe sequence until it finds the key or a usable slot.

Open addressing is space efficient because it avoids per-bucket linked structures. The tradeoff is that lookup, insertion, and deletion depend heavily on the probe rule and on keeping the load factor below 1.

This page covers the three standard probe families: linear probing, quadratic probing, and double hashing.

Table States

In open addressing, each table slot has a state:

  • Empty: The slot has never held an entry.
  • Occupied: The slot currently stores an entry.
  • Deleted: The slot used to store an entry but now holds a tombstone marker.

The deleted state is important. During lookup, a tombstone cannot stop the search, because the key being searched for may have been placed farther along the same probe sequence. During insertion, a tombstone may be reused as a slot for a new entry.

Probe Sequences

A probe sequence is the order of table slots checked for a key. The general form is 

h_0(key), h_1(key), h_2(key), ...

The first probe is the key's home slot. Later probes are alternate positions. A correct open-addressed table must use the same probe sequence for insertion, lookup, and deletion. 

lookup(key):
    for i from 0 to table.length - 1:
        j = probeIndex(key, i)
        if table[j] is empty:
            report missing key
        if table[j] is occupied by key:
            return table[j]
        otherwise continue probing

The search stops at an empty slot because an empty slot means the key would have been inserted earlier in that same probe sequence if it were present. A tombstone does not stop the search.

Linear Probing

Linear probing checks consecutive slots after the home slot: 

h_i(key) = (h(key) + i) mod m

If the home slot is full, try the next slot, then the next, wrapping around at the end of the array.

Example

With \(m=10\), if a key hashes to 6, its linear-probing sequence is: 

6, 7, 8, 9, 0, 1, 2, 3, 4, 5

Advantages

  • Very simple to implement.
  • Good cache locality because nearby slots are checked together.
  • Uses no extra hash function.

Tradeoff: Primary Clustering

Linear probing suffers from primary clustering. Once a run of occupied slots forms, any key that hashes into that run must move to the end of it, making the run longer. Longer runs make future operations slower and attract even more keys.

Quadratic Probing

Quadratic probing spreads probes using a quadratic function of the probe number. A common form is: 

h_i(key) = (h(key) + c1 * i + c2 * i * i) mod m

In a simplified version, the offsets are square numbers: 

h_i(key) = (h(key) + i * i) mod m

Example

With \(m=11\) and home slot 3, square offsets give this sequence: 

3, 4, 7, 1, 8, 6, ...

The probes jump away from the home slot instead of checking one long consecutive run.

Advantages

  • Reduces the primary clustering problem of linear probing.
  • Still uses a predictable formula and one main hash value.
  • Can perform well when the table size and constants are chosen carefully.

Tradeoff: Secondary Clustering

Quadratic probing can still suffer from secondary clustering. If two keys have the same home slot, they follow the same probe sequence. Also, not every choice of table size and constants guarantees that the sequence can reach every slot, so implementations must choose parameters carefully and resize before the table becomes too full.

Double Hashing

Double hashing uses a second hash function to choose the step size for the probe sequence: 

h_i(key) = (h1(key) + i * h2(key)) mod m

The first hash function chooses the home slot. The second hash function chooses how far to move on each probe. The step size must not be 0, and it should be relatively prime to the table size so the probe sequence can visit every slot.

Example

Let \(m=11\), \(h_1(k)=k \bmod 11\), and \(h_2(k)=1+(k \bmod 10)\). For key 25: 

h1(25) = 3
h2(25) = 6
probe sequence: 3, 9, 4, 10, 5, 0, 6, 1, 7, 2, 8

Advantages

  • Usually reduces clustering more than linear or simple quadratic probing.
  • Keys with the same home slot can still have different probe sequences.
  • Can approximate uniform probing when the hash functions are well chosen.

Tradeoffs

  • Requires computing a second hash value.
  • The second hash function must be designed so the step size works with the table size.
  • It is slightly more complex to explain and implement correctly.

Worked Example

Use a table of size \(m=7\), hash function \(h(k)=k \bmod 7\), and linear probing. Insert 10, 17, 24, and 8.

Key Home Slot Probes Final Slot
10 3 3 is empty 3
17 3 3 occupied, 4 empty 4
24 3 3 occupied, 4 occupied, 5 empty 5
8 1 1 is empty 1

Searching for 24 repeats the same sequence: check 3, then 4, then 5. It cannot jump directly to 5 unless it remembers or recomputes the same probing rule.

Insertion and Deletion Details

Insertion

To insert, follow the probe sequence until finding either the key, an empty slot, or a reusable tombstone. If the key is found, update the existing map value or leave the set unchanged. If the key is absent, store the new entry in the first valid slot. 

put(key, value):
    firstDeleted = none
    for i from 0 to table.length - 1:
        j = probeIndex(key, i)
        if table[j] is occupied by key:
            table[j].value = value
            return
        if table[j] is deleted and firstDeleted is none:
            firstDeleted = j
        if table[j] is empty:
            if firstDeleted exists:
                table[firstDeleted] = (key, value)
            else:
                table[j] = (key, value)
            size = size + 1
            return
    resize and try again

Deletion

Deletion should usually mark a slot as deleted instead of making it empty. If the slot were made empty, later lookups might stop too soon and fail to find keys placed farther along the same probe sequence.

Too many tombstones slow down probing. Implementations may rebuild the table when tombstones accumulate, even if the number of live entries is not too high.

Load Factor

In open addressing, every live entry must occupy a table slot, so the load factor must stay below 1. In practice, tables resize well before they are full. As the table fills, probe sequences become longer and operations slow down sharply.

Linear probing is especially sensitive to high load factors because clusters grow. Quadratic probing and double hashing also slow down as fewer empty slots remain.

Time and Space Complexity

Operation Expected Cost Worst Case Reason
Lookup \(O(1)\) \(O(n)\) Expected short probe sequence; worst case may scan many slots.
Put \(O(1)\) amortized expected \(O(n)\) May probe many slots or trigger a resize.
Remove \(O(1)\) \(O(n)\) Must find the key before placing a tombstone.
Space \(O(m)\) \(O(m)\) Entries and slot states are stored in the table array.

Open addressing often uses less extra memory than separate chaining because it avoids a linked node for every entry. Its performance depends more sharply on empty slots remaining available.

Probe Strategy Comparison

Strategy Formula Main Strength Main Risk
Linear probing \((h(k)+i) \bmod m\) Simple and cache-friendly Primary clustering
Quadratic probing \((h(k)+c_1i+c_2i^2) \bmod m\) Reduces primary clustering Secondary clustering and parameter sensitivity
Double hashing \((h_1(k)+i h_2(k)) \bmod m\) Better-separated probe sequences Needs a correct second hash and step size

Common Pitfalls

  • Clearing deleted slots: Replacing a deleted entry with an empty slot can break future searches.
  • Letting the table get full: Open addressing needs empty slots to terminate unsuccessful searches and support insertion.
  • Using a bad step size in double hashing: If the step does not work with the table size, the probe sequence may miss many slots.
  • Forgetting to reuse tombstones: Tombstones should be available for future insertion, or the table will appear fuller than necessary.
  • Ignoring clustering: Long probe runs can make operations much slower than expected.
  • Changing the probe rule: Lookup, insertion, and deletion must use the same sequence for a key.

Summary

Open addressing stores all entries directly in the table array and resolves collisions by probing alternate slots. Linear probing checks consecutive slots and is simple but can form primary clusters. Quadratic probing jumps by quadratic offsets, reducing primary clustering but requiring careful parameters. Double hashing uses a second hash function for the step size and usually reduces clustering further. All open-addressed tables need a controlled load factor, consistent probe sequences, and careful tombstone handling for deletion.

Reading Comprehension Questions

  1. Definition: Where are entries stored in an open-addressed hash table?
  2. Probing: Why must lookup use the same probe sequence as insertion?
  3. Linear probing: What is primary clustering?
  4. Quadratic probing: What problem does it reduce, and what clustering problem can remain?
  5. Double hashing: Why must the second hash function avoid a zero step size?
  6. Deletion: Why are tombstones needed?
  1. Answer: Entries are stored directly in the table array.
  2. Answer: The key may have been placed somewhere later in that sequence after earlier collisions.
  3. Answer: It is the growth of long consecutive occupied runs that attract more colliding keys and slow down probing.
  4. Answer: It reduces primary clustering, but keys with the same home slot still follow the same probe sequence, causing secondary clustering.
  5. Answer: A zero step would probe the same slot repeatedly and never search the rest of the table.
  6. Answer: A tombstone lets lookup continue past a deleted slot so it can still find keys placed farther along the probe sequence.
