Heaps

Overview

A heap is a tree-based abstract data type that maintains a partial order between elements. In a max-heap, each node is at least as large as its children; in a min-heap, each node is at most as large as its children. We call this ordering rule the heap property.

Heaps support efficient insert, peek, and extractMax or extractMin operations, which makes them useful for priority queues, selection problems, and in-place sorting with Heapsort. This page focuses on binary max-heaps; adapting the ideas to min-heaps mostly means reversing the comparisons.

Other heap families, such as binomial heaps and Fibonacci heaps, make different tradeoffs for operations like merging heaps or decreasing a key. Binary heaps remain the most common introductory implementation because they are simple, compact, and fast in practice.

Abstract Data Type and Operations

Abstractly, a heap behaves like a priority queue: it stores a collection of values and gives fast access to the highest-priority item. In a max-heap, the largest value has highest priority.

  • peekMax: Return the maximum value without removing it.
  • insert: Add a new value while preserving the heap property.
  • extractMax: Remove and return the maximum value, then repair the heap.
  • buildMaxHeap: Rearrange an arbitrary array into a valid max-heap.
  • heapify: Repair a local heap-property violation when the subtrees below are already heaps.

Conceptual Model

A binary heap is a complete binary tree: every level is full except possibly the last, and the last level is filled from left to right. That completeness is what makes the array implementation work so well, because there are no gaps in the tree.

With zero-based indexing, the relatives of the node at index \(i\) are:

  • left child: \(2i + 1\)
  • right child: \(2i + 2\)
  • parent: \(\lfloor(i - 1)/2\rfloor\)

Because all elements can be packed into one array, heaps avoid the pointer overhead of node-based tree implementations and benefit from good cache locality. In practice, we often maintain both the array and a separate heapSize variable so the array can have extra capacity while only the first heapSize entries are considered part of the heap.

Core Operations

The key heap operations all rely on the same invariant: every parent must be at least as large as its children. Some operations move a value downward to restore that invariant; others move a value upward.

heapify(A, i, heapSize)

heapify restores the max-heap property at index \(i\), assuming the left and right subtrees are already valid heaps. It compares the node with its children, swaps with the larger child if needed, and repeats one level lower.

  1. Compute the indices of the left and right children.
  2. Find the largest among the parent and the existing children.
  3. If one of the children is larger, swap and continue from that child.

buildMaxHeap(A)

To build a heap from an arbitrary array, start at the last internal node and call heapify while moving backward toward the root. This works because each subtree is fixed before its parent is processed.

  1. Locate the last non-leaf node at index \(\lfloor n/2 \rfloor - 1\).
  2. Call heapify on each internal node from right to left.
  3. Finish at the root, which yields a valid max-heap.

insert(A, value, heapSize)

Insertion places the new value at the end of the heap and then bubbles it up until the heap property is restored. Each swap moves the new value one level closer to the root.

  1. Store the new value at index heapSize and increment heapSize.
  2. Compare the new node with its parent.
  3. While the new node is larger than its parent, swap upward.

extractMax(A, heapSize)

The maximum value is always at the root, so extraction removes A[0]. To preserve the complete-tree shape, the last heap element moves to the root, the heap shrinks by one, and then heapify repairs the tree from the top down.

  1. Save the root value to return later.
  2. Move the last heap element into the root position and decrement heapSize.
  3. Call heapify(A, 0, heapSize) to restore the heap property.

Implementation Options

Implementation 1: Array-Based Binary Heap

The standard implementation stores the heap in an array and relies on the index formulas from the conceptual model. The main invariant is that the represented tree is complete and every parent is at least as large as its children.

How the Operations Work

heapify

Compare the element at index i with its children that lie within heapSize. If a child is larger, swap with the larger child and continue recursively or iteratively from there. 

public void heapify(int[] A, int i, int heapSize) {
    int largest = i;
    int left    = 2 * i + 1;
    int right   = 2 * i + 2;

    if (left < heapSize && A[left] > A[largest]) {
        largest = left;
    }
    if (right < heapSize && A[right] > A[largest]) {
        largest = right;
    }

    if (largest != i) {
        swap(A, i, largest);
        heapify(A, largest, heapSize);
    }
}

def heapify(A, i, heapSize):
    largest = i
    left = 2 * i + 1
    right = 2 * i + 2

    if left < heapSize and A[left] > A[largest]:
        largest = left
    if right < heapSize and A[right] > A[largest]:
        largest = right

    if largest != i:
        A[i], A[largest] = A[largest], A[i]
        heapify(A, largest, heapSize)
buildMaxHeap

Start at the last internal node and call heapify on each internal node going backward to the root. Although this seems like it should take \(O(n \log n)\), the total running time is actually \(O(n)\) because most nodes are near the bottom of the tree. 

public void buildMaxHeap(int[] A) {
    int n = A.length;
    for (int i = n / 2 - 1; i >= 0; i--) {
        heapify(A, i, n);
    }
}
void buildMaxHeap(int A[], int n) {
    for (int i = n / 2 - 1; i >= 0; i--) {
        heapify(A, i, n);
    }
}
def build_max_heap(A):
    n = len(A)
    for i in range(n // 2 - 1, -1, -1):
        heapify(A, i, n)
insert and extractMax

Insertion bubbles a new value upward from the end of the array, while extraction moves the last heap element to the root and then pushes it downward with heapify.

public void insert(int[] A, int value, int heapSize) {
    A[heapSize] = value;
    heapSize++;
    int i = heapSize - 1;

    while (i > 0 && A[(i - 1) / 2] < A[i]) {
        swap(A, i, (i - 1) / 2);
        i = (i - 1) / 2;
    }
}

    heapSize += 1
    i = heapSize - 1

    while i > 0:
        parent = (i - 1) // 2
        if A[parent] < A[i]:
            A[i], A[parent] = A[parent], A[i]
            i = parent
        else:
            break
Extract Max
public int extractMax(int[] A, int heapSize) {
    int max = A[0];
    A[0] = A[heapSize - 1];
    heapSize--;
    heapify(A, 0, heapSize);
    return max;
}
int extractMax(int A[], int heapSize) {
    int max = A[0];
    A[0] = A[heapSize - 1];
    heapSize--;
    heapify(A, 0, heapSize);
    return max;
}
def extract_max(A, heapSize):
    max_val = A[0]
    A[0] = A[heapSize - 1]
    heapSize -= 1
    heapify(A, 0, heapSize)
    return max_val

Notes

  • peekMax is just return A[0] after checking that the heap is nonempty.
  • If the backing array has spare capacity, heapSize can be smaller than A.length.
  • Every child-index calculation must be checked against heapSize, not just the physical array length.

Library Implementations

In practice, most languages already provide priority-queue libraries built on binary heaps or closely related structures. Those libraries are the right default choice in production code, but implementing a heap yourself is still valuable for understanding the invariant and the array representation.

  • Java: PriorityQueue is a min-heap by default, so a max-heap needs a reversed comparator.
  • C++: std::priority_queue is a max-priority queue by default.
  • Python: heapq provides a min-heap; a max-heap is usually simulated by negating priorities.

These libraries expose the priority-queue ADT rather than the full internal heap representation, so they are convenient for applications but less useful when you want to study how heap maintenance works internally.

Operation Costs

The height of a binary heap with \(n\) elements is \(\lfloor \log n \rfloor\), which drives the cost of the update operations. The array representation also uses only constant extra space beyond the storage for the items.

Operation Array-Based Binary Heap Library Priority Queue Notes
peekMax \(O(1)\) \(O(1)\) The maximum is always stored at the root.
insert \(O(\log n)\) \(O(\log n)\) The new element may bubble from a leaf to the root.
extractMax \(O(\log n)\) \(O(\log n)\) After removing the root, the replacement may move down one path.
heapify \(O(\log n)\) Internal The repair travels downward through at most the heap height.
buildMaxHeap \(O(n)\) Often available implicitly Bottom-up heap construction is linear time overall.

Applications

Heaps are used whenever a program repeatedly needs fast access to the current largest or smallest item.

  • Priority queues for task scheduling, simulation, and event processing.
  • Graph algorithms such as Dijkstra's algorithm and Prim's algorithm, usually with min-heaps.
  • Selection problems such as keeping the top \(k\) items seen so far.
  • In-place sorting with Heapsort.

Comparisons and Tradeoffs

Heaps are excellent when you care about repeated access to the extreme element, but they are not good at supporting fast arbitrary searches or preserving sorted order throughout the structure.

  • Compared with a sorted array, heaps make insertion much faster but make full-order traversal less direct.
  • Compared with a binary search tree, heaps give faster access to the maximum or minimum but do not support ordered search efficiently.
  • Compared with Fibonacci or binomial heaps, binary heaps are simpler and usually better in practice unless merge or decrease-key dominates.

Variations and Extensions

Binary heaps are only one member of a broader family of heap-like structures.

  • Min-heaps: Reverse the comparisons so that the minimum element is stored at the root.
  • Binomial heaps: Support efficient merging of heaps by organizing elements as a forest of trees.
  • Fibonacci heaps: Improve some theoretical bounds, especially for repeated decrease-key operations.
  • d-ary heaps: Give each node more than two children, trading a shorter height for more child comparisons per step.

Common Pitfalls

Heap mistakes are usually small indexing or invariant errors that quietly corrupt the structure.

  • Forgetting that the last valid heap index is heapSize - 1, not A.length - 1.
  • Using the wrong parent or child formulas for zero-based indexing.
  • Assuming that a heap is fully sorted; only the parent-child relationship is guaranteed.
  • Neglecting the empty-heap case before calling peekMax or extractMax.

Summary

A binary heap is a compact array-based implementation of the priority-queue ADT. Its power comes from the heap property and the complete-tree shape: they allow fast insertion, fast removal of the maximum or minimum, and linear-time heap construction. For most applications that need repeated extreme-value access, a binary heap is a simple and effective default choice.

Reading Comprehension Questions

  1. Shape vs. Order: Why does the completeness of the binary tree make an array implementation possible?
  2. Local Repair: Why is it enough for heapify to assume that the subtrees below index i are already heaps?
  3. Not Fully Sorted: What ordering information does a max-heap guarantee, and what does it not guarantee?
  1. Answer: A complete binary tree has no internal gaps, so its nodes can be listed level by level in a contiguous array while preserving fixed parent-child index formulas.
  2. Answer: The only possible violation is at the root of that local subtree. Once the larger child is promoted, the same kind of single-path violation may remain one level lower, which is exactly what the recursive call fixes.
  3. Answer: A max-heap guarantees that every parent is at least as large as its children, so the root is the maximum. It does not guarantee that siblings or nodes in different branches are in sorted order.

In-Class Activities

  1. Index Mapping Practice: Give students an array representation of a heap and ask them to label the parent, left child, and right child of selected indices.
  2. Manual Heapify: Start with one local heap-property violation and have students trace each comparison and swap that heapify performs.
  3. Build a Heap by Hand: Hand out an unsorted array and have students apply bottom-up buildMaxHeap to see why the process works.
