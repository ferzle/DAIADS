Queues

Overview

A queue is a first-in, first-out data type. The earliest inserted item is the first item removed. This rule is usually abbreviated FIFO.

Queues are the natural structure for waiting lines, buffers, scheduling, and breadth-first search. They preserve arrival order.

Like a stack, a queue is an abstract data type. The FIFO rule is the essential idea; the implementation might be a circular array, a linked list, or a specialized library deque. The queue interface deliberately hides random access to the middle of the sequence.

Abstract Data Type and Operations

A queue exposes two ends: values enter at the back and leave from the front.

  • enqueue(value): Add value to the back.
  • dequeue(): Remove and return the front value.
  • front(): Return the front value without removing it.
  • isEmpty(): Report whether the queue has no elements.
  • size(): Return the number of stored elements.

Conceptual Model

A queue behaves like a line. New arrivals join the back, and service happens at the front. 

front -> [14] [8] [31] <- back

Calling dequeue() removes 14. Calling enqueue(20) adds 20 after 31.

The front and back move in the same direction over time. A naive array implementation might physically shift every value left after each dequeue, but efficient queues avoid that shifting by moving an index or by relinking nodes.

Worked Example

Start with an empty queue and perform enqueue(4), enqueue(7), enqueue(9), dequeue(), enqueue(2), front().

Operation Queue Afterward Return Value
enqueue(4) [4] none
enqueue(7) [4, 7] none
enqueue(9) [4, 7, 9] none
dequeue() [7, 9] 4
enqueue(2) [7, 9, 2] none
front() [7, 9, 2] 7

The left side is the front and the right side is the back. The first value enqueued, 4, is the first value dequeued. This is the opposite of stack behavior.

Core Operations

Enqueue

enqueue stores a new value at the back of the queue. In a linked implementation this usually means updating the tail. In an array implementation it means writing at the back index.

If the queue is bounded, enqueue must check whether there is room. If the queue is resizable, enqueue may trigger a resize, similar to a dynamic array. The abstract operation is still simple, but the implementation must define what happens when capacity is exhausted.

Dequeue

dequeue removes the front value. A correct implementation handles underflow when the queue is empty.

In a circular array, dequeuing does not move every element. It clears or ignores the old front slot, advances the front index, and decreases the size. This is what keeps the operation constant time.

Circular Array Movement

A circular array uses modular arithmetic so the logical front and back can wrap around the physical end of the array. If the capacity is m, advancing an index uses (index + 1) mod m.

Many circular queues store frontIndex and size. Then the next enqueue position is (frontIndex + size) mod capacity. Storing size also avoids the ambiguity where frontIndex == backIndex could mean either empty or full.

Implementation Options

Implementation 1: Circular Array Queue

Store a backing array, a frontIndex, a size, and a capacity. The back position for the next enqueue is (frontIndex + size) mod capacity.

Operation Sketch

enqueue(value):
    if size == capacity:
        report overflow or resize
    backIndex = (frontIndex + size) mod capacity
    A[backIndex] = value
    size = size + 1

dequeue():
    if size == 0:
        report underflow
    value = A[frontIndex]
    A[frontIndex] = empty
    frontIndex = (frontIndex + 1) mod capacity
    size = size - 1
    return value

front():
    if size == 0:
        report underflow
    return A[frontIndex]

Implementation 2: Linked Queue

A linked queue stores references to both the head and tail nodes. The head is the front of the queue; the tail is the back. Enqueue changes the tail, and dequeue changes the head.

Operation Sketch

enqueue(value):
    node = Node(value)
    if tail == null:
        head = node
        tail = node
    else:
        tail.next = node
        tail = node
    size = size + 1

dequeue():
    if head == null:
        report underflow
    value = head.value
    head = head.next
    if head == null:
        tail = null
    size = size - 1
    return value

Notes

  • Both ends are updated in constant time.
  • When the last element is dequeued, both head and tail must become null.
  • Each element requires a node and at least one reference.

Library Implementations

  • Java: ArrayDeque<E> is a common queue implementation.
  • C++: std::queue is a container adapter.
  • Python: collections.deque supports efficient queue operations.

Python lists can remove from the front, but doing so shifts the remaining values and is not efficient for a true queue workload. A deque or circular buffer is the better match.

The point of a queue implementation is to avoid \(O(n)\) shifting on each dequeue. If an implementation shifts all remaining values after every removal, a long sequence of queue operations can become unnecessarily slow.

Applications

  • Breadth-first search: BFS uses a queue to process vertices in the order they are discovered. This is what makes the traversal expand layer by layer and compute shortest paths in unweighted graphs.
  • Level-order tree traversal: A queue stores the next nodes to visit from left to right across each level. Children are enqueued after their parent is processed, so the traversal naturally moves downward one level at a time.
  • Task scheduling: A FIFO queue is appropriate when tasks should be handled in arrival order. Print queues, simple job queues, and first-come-first-served simulations all use this model.
  • Producer-consumer buffers: One part of a program can enqueue work while another dequeues it. Bounded queues are especially useful when the producer must slow down if the consumer falls behind.
  • Service-line simulations: Queues model customers, packets, or requests waiting for service. Simulations can estimate waiting time, throughput, and the effect of changing the number of servers.
  • Message passing: Programs often use queues to decouple senders from receivers. The sender can add a message and continue, while the receiver processes messages in order later.
  • Rate-limited processing: When a system can process only a limited number of requests per time unit, a queue preserves fairness among waiting requests. Older requests are handled before newer ones unless priorities are introduced.
  • Streaming windows: A queue can store the active items in a sliding window. As new items enter, old items leave from the front, keeping the window's contents current.

Comparisons and Tradeoffs

  • Queue vs. stack: A queue preserves arrival order, while a stack reverses it. This difference explains why BFS uses a queue and DFS often uses a stack.
  • Circular array vs. shifting array: A circular array moves indices instead of moving all elements after each dequeue. This is the key implementation choice that keeps dequeue constant time.
  • Circular array vs. linked queue: A circular array queue is compact and cache-friendly but must handle capacity and wrap-around. A linked queue grows one node at a time but pays node-allocation and reference overhead.
  • Queue vs. priority queue: A FIFO queue removes the oldest item. A priority queue removes the item with highest priority, so it is a different ADT even though both support an insert-and-remove workflow.
  • Queue vs. deque: A deque generalizes queues by allowing insertion and removal at both ends. If an algorithm only needs FIFO behavior, the smaller queue interface communicates intent more clearly.
  • Bounded vs. unbounded queue: A bounded queue must define what happens when it fills: reject, block, overwrite, or resize. An unbounded queue is simpler conceptually but can still exhaust memory in real systems.

Variations and Extensions

  • Deque: A double-ended queue supports insertion and removal at both the front and back. Stacks and queues can both be viewed as restricted forms of a deque.
  • Priority queue: A priority queue removes by priority instead of arrival order. It is usually implemented with a heap and is used in algorithms such as Dijkstra's and Prim's.
  • Bounded queue: A bounded queue has fixed capacity. It is common in buffers, embedded systems, and producer-consumer settings where unlimited growth is not acceptable.
  • Blocking queue: A blocking queue waits when a consumer tries to remove from an empty queue or a producer tries to add to a full one. This is common in threaded programs.
  • Circular queue: A circular queue is an array-based implementation pattern where indices wrap around with modular arithmetic. It avoids the shifting cost of a naive array queue.
  • Monotonic queue: A monotonic queue stores candidates in increasing or decreasing order. It is useful for sliding-window minimum and maximum problems.

Common Pitfalls

  • Ignoring underflow: dequeue and front need defined behavior on an empty queue. A clean implementation reports an error, returns a documented special value, or uses an exception.
  • Shifting after dequeue: Moving every element left after each dequeue turns a normally constant-time operation into \(O(n)\). Efficient array queues move the front index instead.
  • Off-by-one wrap-around errors: Circular queues are simple in concept but easy to get wrong at the physical end of the array. Every index advance should consistently use modulo capacity.
  • Forgetting the empty linked-queue case: When the last node is dequeued, both head and tail must show that the queue is empty. Leaving tail pointing to the removed node creates stale state.
  • Confusing back with next insertion: The current back element and the next enqueue position are different when the queue is nonempty. Mixing them up can overwrite the last real item.
  • Ambiguous full and empty states: Some circular queue designs have frontIndex == backIndex in both empty and full states. A size field, one unused slot, or an explicit full flag resolves the ambiguity.
  • Using FIFO when priority matters: A queue is fair by arrival order, not by importance. If urgent items should go first, the correct structure is usually a priority queue or a set of multiple queues.

Summary

A queue stores items in first-in, first-out order. Efficient implementations keep direct access to both the front and the back, either with a circular array or with head and tail references in a linked list.
