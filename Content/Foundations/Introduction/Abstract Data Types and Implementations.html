Abstract Data Types and Implementations

Overview

An abstract data type, or ADT, describes what a collection of data is allowed to do without saying exactly how it is represented in memory. An implementation chooses the representation and writes the code that makes the operations work.

This separation is one of the central ideas in data structures. A stack can be described by the operations push, pop, and peek. It can be implemented with an array, a dynamic array, or linked nodes. Client code should be able to use the stack behavior without depending on which representation was chosen.

The ADT Contract

An ADT is a contract between the code that uses a structure and the code that implements it. The contract usually includes the values the ADT can store, the operations it provides, what each operation returns, and what happens in boundary cases.

Part of the contract Question it answers
Stored values What kind of items can be placed in the structure?
Operations What actions are allowed?
Preconditions What must be true before an operation is called?
Postconditions What must be true after the operation finishes?
Error behavior What happens for cases such as removing from an empty structure?
Performance expectations Which operations should be fast enough for the intended use?

Example: Stack ADT

A stack is a last-in, first-out collection. The most recently pushed item is the first item removed by pop. The ADT description focuses on that behavior.

Operation Behavior Boundary case
push(x) Add x so it becomes the top item. May fail or resize if capacity is fixed.
pop() Remove and return the current top item. Must define behavior for an empty stack.
peek() Return the current top item without removing it. Must define behavior for an empty stack.
isEmpty() Return whether the stack currently has no items. No special case needed.

None of these operation descriptions say whether the stack uses an array or linked nodes. That is intentional. The ADT says what clients are allowed to rely on; the implementation decides how to make it happen.

Implementing an ADT

To implement an ADT, you choose a representation and write algorithms for the operations. The implementation must preserve a representation invariant, which is the rule that makes the internal state valid.

For an array-based stack, the representation might be:

  • An array named items that stores the stack values.
  • An integer named size that records how many stack values are currently present.
  • An invariant saying that the stack contents occupy items[0] through items[size - 1], and the top item is at items[size - 1].

With that representation, push stores the new item at items[size] and increments size. pop reads items[size - 1], decreases size, and returns the saved value. The code may also clear the unused array slot or resize the array, depending on the language and design goals.

Multiple Implementations

One ADT can have multiple correct implementations. They provide the same visible behavior but make different tradeoffs internally.

Stack implementation Internal representation Tradeoff
Fixed array Values stored in contiguous array positions. Simple and fast, but capacity is limited unless resizing is added.
Dynamic array Array grows when it becomes full. Usually fast, but occasional resizing copies many items.
Linked nodes Each item is stored in a node that points to the next node. No array resizing, but each item needs extra link storage.

If client code uses only the ADT operations, the implementation can often be changed later without rewriting the client. This is one reason programmers care about interfaces, encapsulation, and information hiding.

The Abstraction Boundary

The abstraction boundary is the line between what clients are allowed to know and what the implementation is allowed to change. For a stack, clients should know that pop removes the most recently pushed item. They should not need to know whether that item is stored at the end of an array or in the first linked node.

This boundary protects both sides. Client code becomes easier to read because it uses meaningful operations instead of manipulating raw storage. Implementation code becomes easier to improve because internal details are not scattered throughout the program.

Important Distinction

An ADT is not the same thing as a class in a programming language. A class is one way to implement an ADT. The ADT is the behavioral idea; the class is one concrete programming construct that can express it.

Where Cost Fits

ADT descriptions sometimes include performance expectations, but those expectations are only achieved by particular implementations. For example, a priority queue ADT might promise insert and removeMin. An unsorted array implementation makes insertion easy but removal requires a scan. A heap implementation makes both operations efficient enough for many algorithms.

This is why data structure pages often discuss the ADT first, then implementation options, then time and space complexity. The behavior tells us what must happen. The representation determines how expensive it is.

Reading Comprehension Questions

  1. Definition: What does an ADT describe?
  2. Implementation: What additional decisions are needed to implement an ADT?
  3. Stack behavior: After push(A), push(B), and push(C), what should pop() return?
  4. Boundary: Why should client code avoid depending on a stack's internal array or linked-node representation?
  1. Answer: It describes the values and operations of a structure, including visible behavior, without committing to a memory representation.
  2. Answer: The implementer must choose a representation, maintain invariants, write operation algorithms, define error handling, and meet expected costs when possible.
  3. Answer: It should return C, because a stack removes the most recently pushed item first.
  4. Answer: Depending only on the ADT operations allows the implementation to change without breaking client code.

In-Class Activities

  1. Specify an ADT: Write an ADT contract for a playlist. Include at least five operations and define one boundary case for each.
  2. Representation choices: Propose two implementations for a queue. For each one, describe the internal state and the invariant.
  3. Client versus implementer: Mark each statement as something a stack client should know or something only the implementer should know: "pop removes the top item," "the array doubles when full," "size is the number of stored items," "peek does not remove an item."

Homework Problems

  1. Design an ADT: Define an ADT for a simple undo history in a text editor. Specify operations, preconditions, postconditions, and empty-history behavior.
  2. Compare implementations: Compare an array-based stack and a linked-node stack. Explain the cost of push, pop, memory use, and what happens when the structure grows.
  3. Find the boundary: Look at a class you have written before. Identify which parts are part of the public ADT contract and which parts are implementation details.
