What Is a Data Structure?

Overview

A data structure is a way of organizing data so that a program can use it effectively. The organization matters because every program needs to do more than store values. It must also find them, update them, compare them, remove them, and move through them in some useful order.

A grocery list, a dictionary, a browser history, a folder tree, and a map of roads all organize information differently because they support different tasks. The same is true in programs. Choosing a data structure is choosing which operations should be convenient and which tradeoffs are acceptable.

State and Operations

A data structure has state and operations. The state is the information currently stored. The operations are the actions allowed on that information.

Example: Browser History

The state might be a sequence of pages you have visited. The operations might include visit(page), back(), and forward(). The important question is not only where the pages are stored, but how those operations should behave.

Good data structure descriptions are precise about both parts. They say what data is stored, what relationships are maintained, and how each operation changes the structure.

Why Organization Matters

Imagine a collection of student records. If the records are stored in the order they were added, it is easy to print them in insertion order. If they are stored sorted by last name, it is easier to search alphabetically. If they are stored in a hash table keyed by student ID, lookup by ID can be very fast, but printing in sorted order requires extra work.

Goal Helpful organization Tradeoff to notice
Access the item at position i Array or dynamic array Insertions in the middle may require shifting items.
Add and remove from one end Stack Only the top item is directly available.
Process items in arrival order Queue Items in the middle are not the focus.
Find by key quickly Hash table Ordering is not preserved automatically.
Keep items sorted while updating Search tree The structure must maintain ordering rules after updates.

Invariants

Most data structures have an invariant: a property that must remain true after every operation. Invariants are what make the structure reliable.

  • In a stack, the next item removed is the most recently added item that has not already been removed.
  • In a queue, the next item removed is the least recently added item that has not already been removed.
  • In a binary search tree, values in the left subtree are smaller than the node's value, and values in the right subtree are larger, according to the chosen ordering rule.
  • In a min-heap, every parent has priority no larger than the priorities of its children.

When an operation breaks the invariant, later operations may give incorrect answers even if the data is still somewhere in memory. Much of data structure implementation is about changing the state while restoring the invariant.

Implementation Options

The same conceptual structure can often be implemented in several ways. A list can be stored in an array, a dynamic array, or linked nodes. A priority queue can be implemented with an unsorted array, a sorted array, or a heap. These choices change the cost of operations.

A useful way to compare implementations is to ask which operation should be fast most often. If a program performs millions of searches and very few insertions, a different structure may be appropriate than if it performs constant updates and only occasional searches.

Question Why it matters
How often do we insert or delete? Updates may require shifting, relinking, rotating, or rehashing.
Do we need fast search? Search requirements often drive the choice between arrays, trees, and hash tables.
Do we need ordering? Some structures preserve order naturally; others require extra work.
How much memory is acceptable? Pointers, unused array capacity, and auxiliary tables all have space costs.

Algorithms and Data Structures Together

Algorithms and data structures are usually studied together because each one shapes the other. An algorithm describes the steps. A data structure determines which steps are cheap, expensive, simple, or awkward.

For example, Dijkstra's shortest path algorithm repeatedly needs to remove the unsettled vertex with the smallest tentative distance. If that set of vertices is kept in a simple unsorted list, finding the minimum takes a scan. If it is kept in a priority queue, that operation is designed to be efficient. The high-level algorithm is the same, but the data structure changes the cost.

Reading Comprehension Questions

  1. Definition: What makes a data structure more than just a collection of values?
  2. Invariant: Why are invariants important when implementing operations?
  3. Tradeoff: Why might one program use a hash table while another uses a search tree?
  4. Connection: How can a data structure affect the running time of an algorithm?
  1. Answer: It organizes values and defines operations on them, including how those operations behave.
  2. Answer: They are the rules that keep the structure meaningful. If an operation breaks the invariant, later operations may return wrong results.
  3. Answer: A hash table emphasizes fast lookup by key without preserving order. A search tree supports ordered traversal and ordered queries, often with different costs.
  4. Answer: The structure determines the cost of basic actions such as access, insertion, deletion, minimum selection, or search.

In-Class Activities

  1. Choose a structure: For a music playlist, identify three operations users expect. Decide whether an array-based list, linked list, stack, queue, tree, or hash table best supports those operations.
  2. Find the invariant: State the invariant for a stack, a sorted array, and a binary search tree. For each one, describe an operation that could accidentally break it.
  3. Operation profile: Given a collection that is searched 1000 times for every insertion, discuss what information you would want before choosing an implementation.

Homework Problems

  1. Compare organizations: Describe two ways to store a classroom roster. For each representation, explain the cost or difficulty of adding a student, removing a student, finding a student by ID, and printing students alphabetically.
  2. Design tests: Pick a data structure you have implemented before. List tests that would check its normal cases, empty-structure cases, duplicate-value cases, and boundary cases.
  3. Algorithm dependency: Find an algorithm from another page on this site and identify at least one data structure assumption it makes.
