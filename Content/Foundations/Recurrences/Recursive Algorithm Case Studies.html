Recursive Algorithm Case Studies

Overview

This page brings the recurrence tools together. For each recursive algorithm, we identify the input size, write a recurrence, choose a solving method, and interpret the result in algorithmic terms.

The important habit is to connect the code shape to the recurrence shape. The recurrence is not separate from the algorithm; it is a compact description of the calls the algorithm makes and the work each call does.

Analysis Workflow

  1. Define \(T(n)\): State what input size \(n\) measures and which case is being analyzed.
  2. Read the base case: Identify the input sizes that stop recursing.
  3. Read the recursive calls: Count how many there are and how large their inputs are.
  4. Count local work: Include splitting, combining, comparisons, loops, and helper calls.
  5. Solve or bound the recurrence: Use iteration, recursion trees, substitution, or the Master Theorem.
  6. Interpret the answer: Explain what the growth rate says about the algorithm.

Case Study: Binary Search

Recursive binary search keeps one half of the current sorted range and discards the other. 

int binarySearch(int[] a, int left, int right, int target) {
    if (right < left) {
        return -1;
    }

    int middle = (left + right) / 2;
    if (a[middle] == target) {
        return middle;
    } else if (target < a[middle]) {
        return binarySearch(a, left, middle - 1, target);
    } else {
        return binarySearch(a, middle + 1, right, target);
    }
}

Let \(T(n)\) be the worst-case time to search a range of length \(n\). Each call does constant local work and makes at most one recursive call on a range of about half the size:

\[ T(n)=T(n/2)+1. \]

Iteration gives \(T(n)=1+\log_2 n\) for powers of two, so the worst-case running time is \(\Theta(\log n)\). The reason is structural: the recursion is a single path whose length is the number of times \(n\) can be halved.

Case Study: Merge Sort

Merge sort recursively sorts both halves of the array, then merges the two sorted halves. 

mergeSort(array):
    if length(array) <= 1:
        return array
    left, right = split array into halves
    sortedLeft = mergeSort(left)
    sortedRight = mergeSort(right)
    return merge(sortedLeft, sortedRight)

Let \(T(n)\) be the worst-case time for an array of length \(n\). There are two recursive calls on size \(n/2\), and merging costs \(\Theta(n)\).

\[ T(n)=2T(n/2)+\Theta(n). \]

The Master Theorem or a recursion tree gives \[ T(n)=\Theta(n\log n). \] Merge sort is not logarithmic even though the input is repeatedly halved, because every level of the recursion tree processes all \(n\) elements during merging.

Case Study: Towers of Hanoi

The Towers of Hanoi algorithm moves \(n-1\) disks out of the way, moves the largest disk, then moves the \(n-1\) disks onto the largest disk. 

hanoi(n, source, destination, spare):
    if n == 1:
        move one disk from source to destination
    else:
        hanoi(n - 1, source, spare, destination)
        move one disk from source to destination
        hanoi(n - 1, spare, destination, source)

Let \(H(n)\) be the number of disk moves. The base case is \(H(1)=1\). For \(n>1\), the algorithm solves two subproblems of size \(n-1\) and makes one move in between.

\[ H(n)=2H(n-1)+1. \]

Expanding or proving by substitution gives

\[ H(n)=2^n-1. \]

This is exponential. Adding one more disk roughly doubles the number of required moves.

Case Study: Naive Fibonacci

The simple recursive Fibonacci function is a classic example of repeated subproblems. 

int fib(int n) {
    if (n <= 1) {
        return n;
    }
    return fib(n - 1) + fib(n - 2);
}

Let \(A(n)\) be the number of additions performed. The base cases perform no additions: \(A(0)=A(1)=0\). For \(n>1\), the function computes two recursive calls and then adds their results.

\[ A(n)=A(n-1)+A(n-2)+1. \]

This recurrence grows exponentially. The problem is not that recursion is automatically slow. The problem is that the same Fibonacci values are recomputed many times. Memoization or dynamic programming changes the recurrence shape by storing answers instead of rebuilding the same call tree.

Case Study: Stooge Sort

Stooge sort is intentionally inefficient, but it is useful for recurrence practice. It sorts overlapping two-thirds portions of the array three times. 

stoogeSort(A, left, right):
    if right <= left:
        return
    if A[right] < A[left]:
        swap A[left] and A[right]
    if right - left > 1:
        third = (right - left + 1) / 3
        stoogeSort(A, left, right - third)
        stoogeSort(A, left + third, right)
        stoogeSort(A, left, right - third)

Let \(T(n)\) be the time on a range of length \(n\). Outside the recursive calls, the algorithm does constant work. It makes three recursive calls on ranges of about \(2n/3\).

\[ T(n)=3T(2n/3)+1. \]

To use the Master Theorem, rewrite \(2n/3\) as \(n/(3/2)\). Thus \(a=3\), \(b=3/2\), and \(d=0\). The theorem gives

\[ T(n)=\Theta\left(n^{\log_{3/2}3}\right), \]

which is approximately \(n^{2.71}\). This is much worse than merge sort's \(\Theta(n\log n)\).

Choosing a Solving Method

Recurrence shape Good first method Reason
\(T(n)=T(n-1)+f(n)\) Iteration Expands into a sum over \(n\) decreasing inputs.
\(T(n)=T(n/2)+f(n)\) Iteration or recursion tree There is one path with logarithmic depth.
\(T(n)=aT(n/b)+\Theta(n^d)\) Master Theorem The theorem directly summarizes the level-cost comparison.
Suspected exact formula Substitution Induction verifies that the formula satisfies the recurrence.
Messy or nonstandard recurrence Tree for insight, substitution for proof The tree suggests a bound; substitution checks it rigorously.

Interpreting the Result

A recurrence solution should lead back to an algorithmic explanation.

  • \(\Theta(\log n)\): The algorithm follows one shrinking path, as in binary search.
  • \(\Theta(n)\): The algorithm does constant work per element or has total linear work across all calls.
  • \(\Theta(n\log n)\): The algorithm has logarithmically many levels with linear work per level, as in merge sort.
  • Exponential: The recursive call tree grows too quickly, often because the algorithm recomputes overlapping subproblems or branches on \(n-1\)-size inputs.

Reading Comprehension Questions

  1. Why is binary search \(\Theta(\log n)\) instead of \(\Theta(n)\)?
  2. Why does merge sort have linear work at each recursion level?
  3. What makes naive recursive Fibonacci inefficient?
  4. For stooge sort, why is \(b=3/2\) rather than \(2/3\)?
  1. Answer: It follows one path that halves the remaining input each time, so the number of calls is logarithmic.
  2. Answer: At each level, all subarrays together contain \(n\) elements, and merging processes those elements linearly.
  3. Answer: It recomputes the same smaller Fibonacci values many times instead of saving them.
  4. Answer: The Master Theorem uses subproblem size \(n/b\). Since \(n/b=2n/3\), \(b=3/2\).

In-Class Activities

  1. Trace calls: Draw the call tree for fib(5) and mark repeated subproblems.
  2. Annotate merge sort: Label each line of merge sort as base-case work, recursive work, or combine work.
  3. Method selection: Given five recurrences, choose a solving method before solving any of them.

Homework Problems

  1. Recursive array sum: A function returns the sum of an array by adding the first element to the sum of the remaining \(n-1\) elements. Write and solve a recurrence for its running time.
  2. Five-way split: An algorithm creates five arrays of size \(n/3\), taking \(\Theta(n)\) time total, then recursively processes each one. Write and solve a recurrence.
  3. Repeated work: Explain how memoization changes the practical call structure of naive recursive Fibonacci.
  4. Algorithm comparison: Compare the growth rates of merge sort and stooge sort and explain which algorithm scales better.
