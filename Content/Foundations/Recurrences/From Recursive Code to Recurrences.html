From Recursive Code to Recurrences

Overview

A recurrence is a way to describe work that is defined in terms of smaller work. Recursive code naturally leads to recurrences because one call often does some local work and then asks smaller calls to finish the rest.

The goal of this page is not to solve recurrences yet. The goal is to translate recursive code into a clear mathematical description of its running time. The solving methods come next.

What a Recurrence Describes

In algorithm analysis, we usually define \(T(n)\) to mean the amount of work done by the algorithm on an input of size \(n\). A recurrence for \(T(n)\) has two parts.

  • Base case: The work for an input small enough that no recursive call is made.
  • Recursive case: The work done by smaller recursive calls plus the non-recursive work done by the current call.

The recursive terms come from calls to the same function. The non-recursive term comes from everything else: comparisons, arithmetic, copying, splitting input, combining answers, and other helper work.

Example: Factorial

Consider this recursive factorial function. Instead of counting every statement, suppose we count the number of multiplications. 

int factorial(int n) {
    if (n <= 0) {
        return 1;
    }
    return n * factorial(n - 1);
}

Let \(M(n)\) be the number of multiplications performed by factorial(n). The base case performs no multiplication, so \(M(0)=0\). For \(n>0\), the current call performs one multiplication after the recursive call returns.

\[ M(n) = \begin{cases} 0 & \text{if } n=0,\\ M(n-1)+1 & \text{if } n>0. \end{cases} \]

The term \(M(n-1)\) comes from the recursive call. The \(+1\) comes from the multiplication by n in the current call.

Example: Binary Search

Recursive binary search has a different shape. Each non-base call does constant local work and makes at most one recursive call on about half of the remaining array. 

int binarySearch(int[] a, int left, int right, int target) {
    if (right < left) {
        return -1;
    }

    int middle = (left + right) / 2;
    if (a[middle] == target) {
        return middle;
    } else if (target < a[middle]) {
        return binarySearch(a, left, middle - 1, target);
    } else {
        return binarySearch(a, middle + 1, right, target);
    }
}

Let \(T(n)\) be the worst-case running time on a sorted array segment of length \(n\). The base case is an empty segment or a segment of size 1, depending on how detailed we want to be. Either way, the base case is constant.

\[ T(n) = T(n/2) + c. \]

The constant \(c\) represents the comparisons, midpoint calculation, and return logic in one call. A more careful recurrence might use \(T(\lfloor n/2 \rfloor)\), but floors and ceilings usually do not change the asymptotic result.

Example: Merge Sort

Merge sort makes two recursive calls, one for each half of the array, and then merges the two sorted halves. The merge step touches each element in the current subarray a constant number of times, so the non-recursive work is linear. 

mergeSort(array):
    if array has length 0 or 1:
        return
    split array into two halves
    mergeSort(left half)
    mergeSort(right half)
    merge the sorted halves

Let \(T(n)\) be the running time on an array of length \(n\). The two recursive calls each cost \(T(n/2)\), and the merge costs \(cn\) for some constant \(c\).

\[ T(n) = 2T(n/2) + cn. \]

Notice how the recurrence records the structure of the algorithm: two half-size calls plus linear combine work.

Translation Process

Use this checklist when translating recursive code into a recurrence.

  1. Choose the input-size variable: Decide what \(n\) measures: array length, range length, number of disks, tree height, or something else.
  2. Define the function: State what \(T(n)\) counts and which case is being analyzed.
  3. Identify base cases: Find inputs that return without making recursive calls and give them constant cost unless exact counts are needed.
  4. Count recursive calls: Record how many calls are made and what size input each call receives.
  5. Count local work: Add the work outside the recursive calls.
  6. State assumptions: Mention worst case, average case, balanced input, powers of two, or ignored floors and ceilings when relevant.

Common Patterns

Code pattern Typical recurrence Example
One call on \(n-1\) \(T(n)=T(n-1)+f(n)\) Factorial, linear recursive sum
One call on \(n/2\) \(T(n)=T(n/2)+f(n)\) Binary search
Two calls on \(n/2\) \(T(n)=2T(n/2)+f(n)\) Merge sort
Two calls on \(n-1\) \(T(n)=2T(n-1)+f(n)\) Towers of Hanoi
Calls on \(n-1\) and \(n-2\) \(T(n)=T(n-1)+T(n-2)+f(n)\) Naive recursive Fibonacci

Common Mistakes

  • Counting the recursive call as one operation: A recursive call may do a large amount of work. Represent it with a term such as \(T(n-1)\) or \(T(n/2)\).
  • Forgetting local work: Splitting, merging, copying, and scanning can dominate the running time.
  • Using the wrong input size: In binary search, the array object may be the same, but the active range gets smaller.
  • Confusing best case and worst case: Binary search may stop immediately if the target is at the middle, but the worst case follows one half until the range is empty.
  • Over-focusing on exact constants: Constants are useful for exact counts, but asymptotic analysis usually keeps only the growth pattern.

Reading Comprehension Questions

  1. In a running-time recurrence, what does \(T(n)\) usually represent?
  2. Where does the \(+1\) or \(+c\) term usually come from?
  3. Why does recursive binary search lead to \(T(n)=T(n/2)+c\) in the worst case?
  4. Why does merge sort lead to two \(T(n/2)\) terms instead of one?
  1. Answer: It represents the work done by the algorithm on an input of size \(n\), usually for a stated case such as worst case.
  2. Answer: It comes from the work done by the current call outside of its recursive calls.
  3. Answer: Each worst-case call does constant local work and continues on one subarray of about half the previous size.
  4. Answer: Merge sort recursively sorts both halves of the array before merging them.

In-Class Activities

  1. Annotate code: Mark the base case, recursive calls, and local work in a recursive sum function.
  2. Compare shapes: Write recurrences for binary search, merge sort, and naive Fibonacci, then explain why their shapes differ.
  3. Case check: For binary search, write separate sentences describing the best case and worst case.

Homework Problems

  1. Recursive maximum: A function finds the maximum of an array by comparing the first element to the maximum of the remaining \(n-1\) elements. Write a recurrence for the number of comparisons.
  2. Array split: An algorithm makes three recursive calls on arrays of size \(n/2\), then scans the current array once. Write a recurrence for its running time.
  3. Base cases: Explain why a recurrence should include base cases even when they do not affect the final asymptotic bound.
