Recursion and the Call Stack

Overview

A recursive algorithm solves a problem by calling itself on a smaller or simpler version of the same problem. The idea is not just "a function calls itself." The useful part is that each call has a more manageable job, and the calls eventually reach a case that can be answered directly.

Recursive code is common in algorithms because many problems already have recursive structure: search one half of a sorted array, process one node and then its subtrees, sort each half of an array, or choose whether to include the next item in a candidate solution. Understanding recursion also prepares you for recurrence relations, divide and conquer, backtracking, trees, and graph traversal.

The Three Requirements

A recursive algorithm needs three pieces. Missing any one of them usually leads to incorrect output or nontermination.

  • Base case: One or more cases that are solved without another recursive call.
  • Recursive case: A rule that reduces the current problem to one or more smaller instances of the same problem.
  • Progress: Each recursive call must move toward a base case.

The base case is sometimes called the stopping case. Progress often means that a numeric argument gets smaller, but the deeper idea is that some measure of the remaining work decreases every time.

Worked Example: Factorial

The factorial function has a natural recursive definition. The value of 0! is 1. For any positive n, the value of n! is n * (n - 1)!. 

int factorial(int n) {
    if (n <= 0) {
        return 1;
    } else {
        return n * factorial(n - 1);
    }
}

In this code, n <= 0 is the base case. When n > 0, the function calls factorial(n - 1), so it makes progress toward the base case.

Call What it needs before it can return
factorial(4) Needs factorial(3), then returns 4 * factorial(3).
factorial(3) Needs factorial(2), then returns 3 * factorial(2).
factorial(2) Needs factorial(1), then returns 2 * factorial(1).
factorial(1) Needs factorial(0), then returns 1 * factorial(0).
factorial(0) Base case. Returns 1 immediately.

After the base case returns, the waiting calls finish in reverse order: factorial(1), then factorial(2), then factorial(3), then factorial(4).

The Call Stack

Each function call gets its own stack frame, which stores information such as parameter values, local variables, and where execution should continue after a called function returns. Recursive calls are not special to the computer. They are ordinary function calls, but several calls to the same function can be active at the same time.

During factorial(4), the call stack grows until it reaches the base case. Then the stack unwinds as calls return.

Moment Top of stack What is happening
1 factorial(4) The original call starts and waits for factorial(3).
2 factorial(3) A new frame is added. It waits for factorial(2).
3 factorial(2) A new frame is added. It waits for factorial(1).
4 factorial(1) A new frame is added. It waits for factorial(0).
5 factorial(0) The base case returns 1. Frames now begin to finish.

The maximum number of active frames is the recursion depth. For the factorial code above, the depth is proportional to n, so the algorithm uses O(n) extra stack space. This is different from an iterative loop version, which can compute the same value using O(1) extra space.

Recursion Versus Iteration

Many recursive algorithms can also be written iteratively. For factorial, a loop is shorter in memory use and avoids function-call overhead. 

int factorialIterative(int n) {
    int result = 1;
    for (int i = 2; i <= n; i++) {
        result = result * i;
    }
    return result;
}

Recursion is not automatically better or worse than iteration. It is a design tool. It is especially useful when the problem naturally splits into smaller versions of itself, as in tree traversal, divide-and-conquer algorithms, and backtracking. It is risky when the base case is wrong, progress is unclear, or repeated recursive calls redo the same work many times.

Common Mistakes

  • No base case: The function always calls itself again.
  • Base case too narrow: The code handles n == 0 but not negative values that can occur.
  • No progress: The recursive call uses the same arguments, or arguments that can move away from the base case.
  • Wrong return combination: The recursive call is correct, but the current call combines the smaller answer incorrectly.
  • Unexpected stack growth: The algorithm works for small inputs but exceeds the call stack for large ones.

Reading Comprehension Questions

  1. What makes an algorithm recursive?
  2. What are the three requirements every terminating recursive algorithm should satisfy?
  3. In factorial(4), which call reaches the base case?
  4. Why does recursive factorial use O(n) extra stack space?
  1. Answer: It calls itself on a smaller or simpler instance of the same problem.
  2. Answer: It needs a base case, a recursive case, and progress toward a base case.
  3. Answer: factorial(0) reaches the base case and returns 1 immediately.
  4. Answer: Before the base case returns, there can be one active stack frame for each value from n down to 0, so the maximum depth grows linearly with n.

In-Class Activities

  1. Identify the pieces: For a recursive method that prints numbers from n down to 1, identify the base case, recursive case, and progress measure.
  2. Stack frames by hand: Draw the stack frames created by factorial(3). Include the value of n in each frame.
  3. Fix the bug: Explain what goes wrong if a recursive sum method stops only when n == 0 but the caller is allowed to pass n = -1.

Homework Problems

  1. Recursive product: Write pseudocode for product(n), which returns 1 * 2 * ... * n for n >= 1. Identify the base case and recursive case.
  2. Stack depth: Suppose mystery(n) calls only mystery(n - 2) until n <= 0. What is the maximum recursion depth as a function of n?
  3. Compare versions: Write both recursive and iterative versions of a method that computes the sum of the first n positive integers. Compare their time and extra space use.
