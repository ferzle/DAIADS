Designing Recursive Solutions

Overview

Recursive design starts by giving one function call a clear meaning. Once the meaning of a call is precise, the recursive call can be treated as a helper that correctly solves a smaller instance. The current call then only has to reduce the problem and combine the smaller answer.

This is the same idea that appears in induction proofs: prove a small case directly, then show how solving a smaller case lets you solve the next case. In code, the base case gives the direct answer, and the recursive case performs the reduction.

Design Process

A reliable recursive design usually follows these steps.

  1. State the meaning of the function: Be precise about what one call should compute or do.
  2. Choose the base case: Identify the smallest input that can be answered immediately.
  3. Choose the smaller problem: Decide what recursive call solves a smaller instance of the same task.
  4. Combine the answer: Use the smaller answer to finish the current call.
  5. Check progress: Verify that every recursive path eventually reaches a base case.

The key question is: If the recursive call already worked, what would I do with its answer? That question keeps the current call focused on one layer of work.

Example: Sum of a Prefix

Suppose sumFirst(n) should return the sum 1 + 2 + ... + n for nonnegative n.

  • Meaning: sumFirst(n) returns the sum of the first n positive integers.
  • Base case: sumFirst(0) should return 0.
  • Smaller problem: sumFirst(n - 1) returns 1 + 2 + ... + (n - 1).
  • Combine: Add n to that smaller sum.
int sumFirst(int n) {
    if (n <= 0) {
        return 0;
    }
    return n + sumFirst(n - 1);
}

The recursive call is not a guess. It has the same meaning as the original call, just with a smaller argument.

Example: Binary Search

Some recursive algorithms reduce the input by changing a range instead of changing a single number. Binary search looks for a value inside a sorted array segment. 

int binarySearch(int[] a, int left, int right, int target) {
    if (right < left) {
        return -1;
    }

    int middle = (left + right) / 2;
    if (a[middle] == target) {
        return middle;
    } else if (target < a[middle]) {
        return binarySearch(a, left, middle - 1, target);
    } else {
        return binarySearch(a, middle + 1, right, target);
    }
}

The meaning of the helper is: search the sorted subarray from left through right. The base case is an empty range. Each recursive call searches a smaller range because it removes the middle element and one side of the array.

Helper Functions

Public methods often have a simple interface, while the recursive work needs extra parameters. A wrapper method can protect callers from those details. 

int search(int[] a, int target) {
    return binarySearch(a, 0, a.length - 1, target);
}

The caller asks to search the whole array. The helper keeps track of which part of the array is still under consideration. This pattern also appears in recursive array processing, tree traversal, backtracking, and divide-and-conquer algorithms.

Example: Palindrome Test

A string is a palindrome if it reads the same forward and backward. Recursively, a string is a palindrome if its first and last characters match and the inside substring is a palindrome. 

boolean isPalindrome(String s, int left, int right) {
    if (left >= right) {
        return true;
    }
    if (s.charAt(left) != s.charAt(right)) {
        return false;
    }
    return isPalindrome(s, left + 1, right - 1);
}

Here the progress measure is the number of unchecked characters between left and right. Each call moves both ends inward, so the unchecked region gets smaller.

Why the Design Works

Recursive correctness arguments often follow the same shape as the code.

  • Base case: Show that the direct answer is correct for the smallest input.
  • Inductive assumption: Assume the recursive call correctly solves the smaller instance.
  • Current case: Explain why the current call combines that smaller answer correctly.

For sumFirst, the base case is correct because the sum of the first zero positive integers is 0. If sumFirst(n - 1) correctly returns 1 + 2 + ... + (n - 1), then adding n gives the correct sum for n.

Efficiency Checks

A recursive algorithm can be correct and still be a poor implementation. Before committing to a recursive design, check the shape of the calls.

Call shape Typical sign What to check
Single chain One recursive call per non-base case Depth and stack space, often O(n) or O(log n).
Balanced split Two or more calls on smaller pieces Combine cost and recurrence, common in divide and conquer.
Repeated subproblems The same arguments appear many times Whether memoization or dynamic programming is needed.
Large search tree Many choices at each level Whether pruning or backtracking constraints reduce the work.

When to Use Recursion

Recursion is often a good fit when the problem naturally contains smaller versions of itself. It is less attractive when a simple loop expresses the idea just as clearly and uses less memory.

  • Good fits: Trees, divide and conquer, backtracking, recursive definitions, and nested structures.
  • Use caution: Very deep inputs, repeated subproblems, and problems where the recursive version is only a disguised loop.
  • Prefer iteration when: The loop version is just as clear and avoids unnecessary stack growth.

Reading Comprehension Questions

  1. Why is it useful to state the meaning of a recursive function before writing code?
  2. In sumFirst(n), what smaller problem does the recursive call solve?
  3. Why does recursive binary search need left and right parameters?
  4. What is one warning sign that a recursive algorithm may be inefficient?
  1. Answer: It clarifies what the recursive call is allowed to assume and what the current call must finish.
  2. Answer: It solves the sum of the first n - 1 positive integers.
  3. Answer: The helper must remember which subarray is still being searched.
  4. Answer: Repeated calls with the same arguments, very large recursion depth, or a large branching search tree can all indicate efficiency problems.

In-Class Activities

  1. Meaning first: For a recursive method maxPrefix(a, n), write a precise sentence describing what the method should return.
  2. Design together: Design a recursive method that counts how many times a target value appears in an array. Identify the base case, smaller problem, and combine step.
  3. Helper parameters: Explain why a recursive palindrome method is easier to write with left and right indices than by creating a new substring at each call.

Homework Problems

  1. Recursive maximum: Write pseudocode for a method that returns the maximum value in a[0..n-1]. State the meaning of the method and prove the recursive case informally.
  2. String reversal: Design a recursive method that returns the reverse of a string. Identify its base case and progress measure.
