Tracing Recursive Code

Overview

Tracing recursive code means following the actual calls the program makes. The most important rule is that each call has its own parameters and local variables. A call that has not reached a return statement waits while its recursive call runs.

This page focuses on tracing, not design. If you can trace recursion accurately, it becomes much easier to debug recursive code and to understand why a base case or recursive case is wrong.

A Reliable Tracing Method

When recursion feels confusing, slow down and keep a written record of the calls.

  1. Write the call: Record the function name and argument values.
  2. Check the base case: If the call reaches a base case, record the return value or output.
  3. Record the recursive call: If the call recurses, write the new call underneath the current one.
  4. Do not merge calls: Treat each call as a separate copy of the function with its own variables.
  5. Unwind in reverse order: Once a base case returns, finish the waiting calls from most recent to oldest.

This mirrors the call stack: calls are added as recursion goes deeper, then removed as return values come back.

Example: Countdown

This recursive method prints the current number before making the recursive call. 

void countdown(int n) {
    if (n > 0) {
        print(n);
        countdown(n - 1);
    }
}

Tracing countdown(4) shows why the output is 4 3 2 1.

Call Immediate output Next call
countdown(4) 4 countdown(3)
countdown(3) 3 countdown(2)
countdown(2) 2 countdown(1)
countdown(1) 1 countdown(0)
countdown(0) No output Base case

The print happens before the recursive call, so the values appear while the stack is growing.

Call Before Work

A small change in line order reverses the output. 

void countup(int n) {
    if (n > 0) {
        countup(n - 1);
        print(n);
    }
}

In countup(4), the recursive calls happen first. The print statements execute only while calls are returning.

Call returns from Then prints Output so far
countup(0) Nothing
countup(1) 1 1
countup(2) 2 1 2
countup(3) 3 1 2 3
countup(4) 4 1 2 3 4

A good tracing habit is to mark whether the work happens before the recursive call, after the recursive call, or both.

Tracing Return Values

Recursive methods often return values instead of printing. In that case, record what each waiting call still needs. 

int sumFirst(int n) {
    if (n <= 0) {
        return 0;
    }
    return n + sumFirst(n - 1);
}
Call Waits for Return value
sumFirst(4) 4 + sumFirst(3) 4 + 6 = 10
sumFirst(3) 3 + sumFirst(2) 3 + 3 = 6
sumFirst(2) 2 + sumFirst(1) 2 + 1 = 3
sumFirst(1) 1 + sumFirst(0) 1 + 0 = 1
sumFirst(0) Base case 0

The return values are computed from the bottom of the table back up to the top.

Branching Recursion

Some recursive functions make more than one recursive call. The naive Fibonacci method is a classic example. 

int fib(int n) {
    if (n <= 1) {
        return n;
    }
    return fib(n - 1) + fib(n - 2);
}

Tracing fib(5) creates a tree of calls rather than a single chain: 

fib(5)
  fib(4)
    fib(3)
      fib(2)
        fib(1)
        fib(0)
      fib(1)
    fib(2)
      fib(1)
      fib(0)
  fib(3)
    fib(2)
      fib(1)
      fib(0)
    fib(1)

This trace shows repeated work. For example, fib(3) and fib(2) are computed more than once. That does not make the code incorrect, but it makes the naive recursive version much slower than it first appears.

Debugging Checklist

  • Check the first call: Are the initial arguments correct?
  • Check the smallest inputs: What happens on empty input, one item, or zero?
  • Check progress: Does every recursive path move toward a base case?
  • Check call order: Does work happen before the recursive call, after it, or both?
  • Check repeated work: Are identical subproblems being recomputed?

Reading Comprehension Questions

  1. Why should each recursive call be treated as having its own copy of the parameters?
  2. Why does countdown(4) print 4 before 3?
  3. Why does countup(4) print 1 before 4?
  4. What does the call tree for fib(5) reveal about the naive recursive Fibonacci method?
  1. Answer: Each call has its own stack frame, so changing or returning from one call does not overwrite the parameters in the waiting calls.
  2. Answer: The method prints before making the recursive call.
  3. Answer: The method makes the recursive call first, so the print statements execute while the calls are returning.
  4. Answer: It reveals repeated subproblems, which explains why the naive method is inefficient.

In-Class Activities

  1. Trace output: Trace countdown(5) and countup(5). Explain why the output order differs.
  2. Trace returns: Complete a trace table for sumFirst(5), including each waiting expression and return value.
  3. Find repeated work: Draw the call tree for fib(6). Circle every repeated call to fib(3).

Homework Problems

  1. Trace mystery output: Given a recursive method that prints before and after its recursive call, predict the output for mystery(3) and justify it with a call trace.
  2. Trace a string method: Write a recursive method that prints a string backward. Trace it on "cat".
  3. Compare call shapes: Give one example of a recursive method whose calls form a single chain and one whose calls form a branching tree. Explain the difference in possible running time behavior.
