Cost Models and Input Size

Overview

Algorithm analysis asks a disciplined version of a simple question: how efficient is this algorithm as the input grows? To answer that, we need a cost model: a decision about what we will count and what we will treat as one unit of work.

The goal is not to predict the exact number of nanoseconds a program will take. Exact time depends on the machine, language, compiler, operating system, memory layout, and current load on the computer. Instead, we count enough work to understand the growth rate.

Why Not Just Time It?

Timing a program can be useful after an implementation exists, but timing alone is a poor way to compare algorithms.

  • Wall-clock time measures elapsed real time, including delays from other processes and system activity.
  • CPU time measures processor time spent on the program, but it still depends on the implementation and machine.
  • Operation counts let us analyze an algorithm before choosing a language, compiler, or hardware platform.

If one program takes 3 minutes on a fast computer and another takes 1 hour on an old computer, that does not prove the first algorithm is better. We need a machine-independent way to reason about the amount of work being done.

A Cost Model

A cost model says which operations we will count. For introductory analysis, we usually treat simple operations as constant-time work:

  • assigning a value to a variable,
  • comparing two primitive values,
  • performing basic arithmetic on primitive values,
  • accessing one array element by index,
  • returning a value from a function.

This is an abstraction. A comparison and an assignment may not take exactly the same number of machine cycles. For growth-rate analysis, that level of detail usually does not matter. What matters is how the number of counted operations changes when the input size doubles, triples, or grows very large.

Abstract Operations

Often we do not count every simple instruction. Instead, we choose an abstract operation that happens as often as the dominant work in the algorithm. For sequential search, the natural abstract operation is comparing the current array element with the target. 

int sequentialSearch(int[] a, int target) {
    for (int i = 0; i < a.length; i++) {
        if (a[i] == target) {
            return i;
        }
    }
    return -1;
}

Counting only a[i] == target ignores loop increments, bounds checks, and returns, but it still captures the growth rate. If the comparison executes once, the algorithm is fast. If it executes \(n\) times, the algorithm's work grows in proportion to \(n\).

Input Size

The variable \(n\) should describe the size of the input, not the value of some temporary loop variable. Choosing \(n\) clearly is part of the analysis.

Input Common size measure Notes
Array or list \(n\), the number of elements Extra parameters such as a target value usually do not change the size measure.
\(n \times m\) matrix \(nm\), or both \(n\) and \(m\) Some analyses need two independent size variables.
Graph \(|V|\), \(|E|\), or both Graph algorithms often depend on vertices and edges separately.
String \(n\), the number of characters Pattern-matching problems may use text length \(n\) and pattern length \(m\).

When an algorithm takes an array of \(n\) values plus a constant number of other primitive parameters, we usually call the input size \(n\), not \(n+1\) or \(n+2\). Constants will not affect the growth rate.

Time and Space

This section mostly analyzes time: how many operations an algorithm performs. Space also matters. A sorting algorithm might be fast but require a large auxiliary array. A recursive algorithm might use one stack frame per recursive call. A hashing algorithm might spend extra memory to make lookups faster.

When space use is important, we usually count extra space: memory beyond the input itself. For example, scanning an array with a few variables uses constant extra space, while making a full copy of the array uses linear extra space.

What Comes Next

The rest of this section builds the analysis toolkit in stages:

  1. Decide whether we are analyzing the best, worst, average, or expected case.
  2. Count enough work to get a function of the input size.
  3. Use asymptotic notation to simplify that function and focus on growth rate.
  4. Compare common growth rates so the result is meaningful.

Reading Comprehension Questions

  1. Why is wall-clock time not a reliable way to compare algorithms run on different machines?
  2. What is a cost model?
  3. For an array algorithm, why do we usually call the input size \(n\) instead of \(n+2\) when there are two extra primitive parameters?
  4. What is an abstract operation?
  1. Answer: The time depends on hardware, system load, implementation details, and many factors other than the algorithm itself.
  2. Answer: It is a rule for what operations we count as units of work during analysis.
  3. Answer: A constant number of extra parameters does not affect how the input grows, so the growth rate is governed by the array length.
  4. Answer: It is a representative operation chosen because counting it captures the dominant work of the algorithm.

In-Class Activities

  1. Choose \(n\): For an algorithm that takes two arrays of lengths \(n\) and \(m\), decide whether the input size should be \(n+m\), \(nm\), or two variables. Justify your choice for two different tasks.
  2. Pick an operation: For sequential search, maximum value, and counting zeros in an array, choose an abstract operation to count.
  3. Time versus operations: Describe a situation where a slower wall-clock run might still come from a better algorithm.

Homework Problems

  1. Input-size choices: Give a natural input-size measure for each: an \(n \times n\) image, a graph stored as an edge list, and a dictionary of \(n\) words plus a query word.
  2. Cost model: Write a cost model for a simple array-scanning algorithm. State which operations you count and which details you ignore.
  3. Space use: Compare the extra space used by an algorithm that scans an array in place with one that first copies the array.
