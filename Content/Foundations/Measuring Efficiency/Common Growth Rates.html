Common Growth Rates

Overview

Once a work count has been simplified with asymptotic notation, the result is useful only if you understand what that growth rate means. This page summarizes the most common growth rates in algorithms and data structures.

The order matters. A linear algorithm usually scales much better than a quadratic one, and a polynomial-time algorithm usually scales much better than an exponential one.

Growth Rate Table

Class Typical notation Common source Effect of doubling \(n\)
Constant \(\Theta(1)\) Fixed amount of work About the same
Logarithmic \(\Theta(\log n)\) Repeatedly cutting the input by a constant factor Adds about one step for base-2 logs
Linear \(\Theta(n)\) Scanning all items once About twice as much work
Linearithmic \(\Theta(n \log n)\) Many divide-and-conquer sorting algorithms A little more than twice as much work
Quadratic \(\Theta(n^2)\) All pairs, many simple nested-loop algorithms About four times as much work
Polynomial \(\Theta(n^k)\) Nested loops with fixed depth, many tractable algorithms Multiplies by about \(2^k\)
Exponential \(\Theta(c^n)\), \(c > 1\) Trying all subsets, assignments, or paths Multiplies by \(c^n\)'s doubling effect; grows very quickly

Standard Ordering

A useful ordering to remember is:

\[ 1 \ll \log n \ll n \ll n\log n \ll n^2 \ll n^3 \ll 2^n \ll n! \ll n^n. \]

The symbol \(\ll\) here means "grows much slower than." You do not need to memorize every possible function, but you should understand the broad pattern: logarithms grow slowly, polynomials grow faster as the exponent increases, and exponentials eventually beat every polynomial.

Constant, Logarithmic, and Linear

Constant-time operations do not grow with input size. Array indexing, stack push, and reading a stored size field are often treated as constant under the usual cost model.

Logarithmic growth often appears when each step removes a constant fraction of the remaining input. Binary search is the standard example: each comparison discards about half of the remaining array.

Linear growth appears when the algorithm must inspect each item a constant number of times. Sequential search, finding a maximum, and counting values in an array are typical examples.

\(n \log n\) and Quadratic

The growth rate \(n\log n\) often appears in divide-and-conquer algorithms that split the input into levels and do linear work across each level. Merge sort is the classic example.

Quadratic growth often appears when an algorithm considers all pairs of \(n\) items or uses nested loops where both loops depend on \(n\). Some simple sorting algorithms, such as selection sort and insertion sort in the worst case, have quadratic running time.

Polynomial and Exponential

A polynomial running time has the form \(\Theta(n^k)\) for some constant \(k\). Linear, quadratic, and cubic algorithms are all polynomial. In many theoretical settings, polynomial time is treated as a rough boundary for "efficient," though constants and the exponent still matter in practice.

Exponential algorithms have running time like \(\Theta(2^n)\), \(\Theta(3^n)\), or more generally \(\Theta(c^n)\) for a constant \(c>1\). These often come from exploring every subset, every binary string, or every combination of choices. They become infeasible quickly.

Log and Exponential Bases

Logarithms with different constant bases have the same asymptotic growth rate:

\[ \log_2 n = \Theta(\log_{10} n). \]

This is why algorithm analysis often writes \(\log n\) without specifying the base. Changing the base only changes the value by a constant factor.

Exponentials are different. \(2^n\) and \(3^n\) do not have the same growth rate. The base of an exponential matters.

Practice Warnings

  • Asymptotic better does not mean always faster: A more complex algorithm may have overhead that loses on small inputs.
  • Same class does not mean same runtime: \(4n^2\) and \(10000n^2\) are both \(\Theta(n^2)\), but the constants still matter in real implementations.
  • Faster-growing complexity means a slower algorithm: A running time of \(n^2\) grows faster than \(n\), so the quadratic algorithm scales worse.
  • Use the right variable: If the input is an \(n \times m\) matrix, a bound may need both \(n\) and \(m\).

Reading Comprehension Questions

  1. Why do logarithmic algorithms scale well?
  2. Why does a quadratic algorithm usually become much worse than a linear algorithm as \(n\) grows?
  3. Why can we usually omit the base of a logarithm in asymptotic notation?
  4. Why do exponential bases matter?
  1. Answer: They usually reduce the remaining input by a constant factor each step, so the number of steps grows very slowly.
  2. Answer: Doubling the input roughly doubles linear work but quadruples quadratic work.
  3. Answer: Changing a logarithm's base changes its value only by a constant factor.
  4. Answer: Exponentials with different bases differ by more than a constant factor as \(n\) grows.

In-Class Activities

  1. Rank functions: Put \(n^2\), \(\log n\), \(2^n\), \(n\log n\), \(1\), and \(n\) in increasing order of growth.
  2. Classify algorithms: Match sequential search, binary search, merge sort, and all-subsets generation to their common growth classes.
  3. Crossover discussion: Discuss why an \(O(n^2)\) algorithm might beat an \(O(n\log n)\) algorithm for small \(n\).

Homework Problems

  1. Growth ranking: Rank \(100n\), \(n^2\), \(n\log n\), \(2^n\), and \(\log^2 n\) from slowest to fastest growth.
  2. Class examples: Give one algorithm or operation from this course that is commonly constant, logarithmic, linear, quadratic, and exponential.
  3. Base comparison: Explain why \(\log_2 n\) and \(\log_{10} n\) are asymptotically equivalent but \(2^n\) and \(10^n\) are not.
