Best, Worst, Average, and Expected Cases

Overview

The same algorithm can take different amounts of work on different inputs of the same size. Before counting operations, we must say which case we are analyzing: best, worst, average, or expected.

The phrase "of size \(n\)" matters. Best case does not mean "choose the smallest possible input." It means: among all inputs of a fixed size \(n\), which one makes the algorithm do the least work?

Always Fix the Input Size

Suppose an algorithm searches an array. If we allow ourselves to choose an array of length 1 for the best case, almost every search algorithm looks constant-time. That is not useful.

Instead, we fix the array length at \(n\), then compare different arrays and target values of that size. This lets us ask meaningful questions about how the work grows as \(n\) grows.

Running Example: Sequential Search

Sequential search scans from left to right until it finds the target or reaches the end. 

int sequentialSearch(int[] a, int target) {
    for (int i = 0; i < a.length; i++) {
        if (a[i] == target) {
            return i;
        }
    }
    return -1;
}

We will count comparisons between a[i] and target. For an array of length \(n\), the count depends on where the target appears, or whether it appears at all.

Best Case

The best case is the least work the algorithm can do on any input of size \(n\).

For sequential search, the best case occurs when the target is the first element. The comparison executes once, no matter how long the array is. So the best-case work is constant.

This does not mean the input has size 1. The array can have one million elements; the best case is still that the target appears at index 0.

Worst Case

The worst case is the most work the algorithm can do on any input of size \(n\).

For sequential search, the worst case occurs when the target is not present, or when it is the last element. The comparison executes \(n\) times.

Worst-case analysis is common because it gives a guarantee: the algorithm will not do more work than this bound for inputs of that size.

Average Case

The average case asks how much work the algorithm does on average over a specified distribution of inputs. The distribution is part of the analysis.

For sequential search, suppose the target is equally likely to be in any one of the \(n\) positions and is guaranteed to be present. If it is in position \(k\), counting from 1, then the algorithm performs \(k\) comparisons. The average number of comparisons is

\[ \frac{1}{n}\sum_{k=1}^{n} k = \frac{1}{n}\cdot\frac{n(n+1)}{2} = \frac{n+1}{2}. \]

This grows linearly with \(n\). The average count is about half the worst-case count, but "half of \(n\)" still grows like \(n\).

Expected Running Time

The term expected running time is often used when randomness is involved or when we are describing typical behavior less formally. It is closely related to average-case analysis, but the assumptions must still be stated.

For example, randomized quicksort has an expected running time because the algorithm makes random choices. Hash tables often have expected constant-time operations under assumptions about hashing and load factor.

Case Summary

Case Question Sequential search example
Best What input of size \(n\) makes the algorithm do the least work? Target at the first position.
Worst What input of size \(n\) makes the algorithm do the most work? Target absent or at the last position.
Average What is the mean work over a stated distribution of inputs? Depends on assumptions about where the target is and whether it is present.
Expected What work should we expect under randomness or a typical-case model? Useful when algorithm choices or input assumptions are probabilistic.

Which Case Should We Use?

Worst-case analysis is often the default because it is a guarantee. Best-case analysis is useful for understanding possible behavior, but by itself it can be misleading. Average-case or expected analysis can be very informative, but only when the assumptions are realistic and clearly stated.

  • Use worst case when users need a guarantee or when bad inputs are important.
  • Use average case when a meaningful input distribution is known.
  • Use expected time for randomized algorithms and probabilistic data structures.
  • Use best case mainly to understand the full range of behavior.

Common Mistakes

  • Using the smallest input as the best case: The input size must stay fixed at \(n\).
  • Ignoring failed searches: For search algorithms, an absent target is often a worst-case input.
  • Saying "average" without a distribution: Average over what set of inputs, with what probabilities?
  • Assuming average and worst are always different: Sometimes they have the same growth rate, even if the exact counts differ.

Reading Comprehension Questions

  1. Why do we fix the input size \(n\) before discussing best or worst case?
  2. What is the best case for sequential search on an array of length \(n\)?
  3. What is the worst case for sequential search?
  4. Why must an average-case analysis state its assumptions?
  1. Answer: Otherwise every algorithm could look artificially good by choosing a tiny input.
  2. Answer: The target is at the first position, so only one comparison is needed.
  3. Answer: The target is absent or at the last position, so \(n\) comparisons are needed.
  4. Answer: The average depends on which inputs are considered possible and how likely they are.

In-Class Activities

  1. Case identification: For finding the maximum in an unsorted array, identify the best and worst cases. Are they different?
  2. Search scenarios: For sequential search, list three different assumptions about where the target might be. How would those assumptions affect average-case analysis?
  3. Insertion sort preview: Describe an input order that should be good for insertion sort and one that should be bad. Do not calculate the full complexity yet.

Homework Problems

  1. Sequential search cases: For an array of length \(n\), count comparisons in the best case, worst case, and when the target is at position \(k\).
  2. Average present search: Reproduce the average-case calculation for sequential search assuming the target is guaranteed to be present and equally likely to appear in each position.
  3. Distribution matters: Suppose a target has a 50 percent chance of being absent and, if present, is equally likely to appear in any position. Set up the expected comparison count.
